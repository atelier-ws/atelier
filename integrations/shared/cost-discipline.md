- **Fewest turns that fully solve it.** Every tool call re-reads the whole accumulated context, so each extra round-trip is re-billed against everything before it. Favor fewer, higher-value calls over many small ones.
- **Read a file once, whole.** When you'll work with more than a small region of a file, read it in full in a single call (`read` with `expand=true`) instead of paging it range-by-range across turns — repeated reads of the same file are the largest avoidable cost.
- **Pull the smallest result that answers the question.** Scope searches narrowly and request small token budgets — a broad `grep`/`search`/`explore` dump is carried in context for the rest of the run. Narrow the path or read one specific file instead of pulling a large map.
- **Localize, then act.** One grounding pass is enough for a scoped change. Once you know the fix, make it — don't keep searching to re-confirm what you already found.
- **Trust the environment.** Assume the toolchain and dependencies are already installed and working; don't reinstall, rebuild, or reconfigure unless a command actually fails because something is genuinely missing. Determine the run/test command once and reuse it — don't rediscover the interpreter each step.
- **Stay in the repository.** Don't fetch web pages or external docs unless the task explicitly requires outside information — the answer is almost always already in the code, and fetched pages are large and costly to carry.
- **Fix source, not tests.** Resolve the issue by changing source files; don't add or rewrite tests to exercise your change unless writing tests is the explicit task — authoring tests burns turns and is usually redundant with the project's own checks.
- **Verify narrowly, once.** Confirm the change with the smallest check that proves it — the single failing test or function — and run it once; don't re-run broad suites repeatedly.
- **Work inline; delegate rarely.** Each sub-agent re-pays the full system-and-tools overhead. Spawn one only when the task genuinely needs independent, parallel exploration — never for steps you can do directly.
