# Phase 3: Plan 02 - Owned Provider Execution Lanes

## Problem

Routing is only real when Atelier executes the selected provider/model path. Today, route recommendations can be recorded while actual execution still follows the host/default model. Owned subcalls need a concrete execution layer with provider/model provenance.

## Scope

- Add execution lanes for Atelier-owned subcalls selected explicitly or by auto route policy.
- Reuse existing internal LLM/provider clients where possible.
- Preserve host-native top-level chat while owned workflow steps can call routed providers.
- Record provider, model, transport, request id, duration, usage, and fallback path.

## Milestones

1. Add an execution interface consumed by the Phase 2 workflow runner.
2. Implement narrow provider lanes using existing OpenAI-compatible/LiteLLM/internal LLM infrastructure.
3. Add fallback behavior that re-routes on provider health failures without hiding the fallback in artifacts.
4. Wire execution receipts into ledger/reporting surfaces.

## Dependencies

- 03-01 provider catalog and route selection modes.
- Phase 2 workflow runner.
- Existing provider/client infrastructure.

## Validation

- Unit tests with fake providers proving selected routes are executed.
- Integration-style tests with stubbed provider responses proving provenance is recorded.
- Benchmark harness can select provider/model explicitly or use `auto`, and artifacts state which mode was used.

## Open questions

- Minimal first provider set for the MVP.
- Whether LiteLLM proxy remains external or gains a first-class local launch helper.
