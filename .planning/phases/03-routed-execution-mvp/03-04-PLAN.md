# Phase 3: Plan 04 - Shadow-Safe Local Host Router Bridge

## Problem

WozCode's installed plugin shows a useful pattern: a local router can sit between Claude Code and the model API, expose preset paths, map models/providers, and rewrite request shape. Atelier should have the same class of capability eventually, but forcing top-level host routing before owned subcall routing is proven would be risky.

## Scope

- Add a local router bridge design for Claude-Code-compatible Anthropic traffic.
- Start in shadow mode: observe, classify, and recommend without changing the live host provider by default.
- Support explicit opt-in enforcement after subcall routing and benchmark parity are proven.
- Reuse the same provider catalog, explicit/auto route modes, cache-affinity state, and provenance ledger as owned subcalls.

## Milestones

1. Define local preset paths and config shape, modeled after the useful Woz concepts: provider bindings, model mappings, preset paths, request modifications, and pass-through auth.
2. Add a shadow capture path that can classify host requests and emit route/cache recommendations without altering execution.
3. Add an opt-in enforced path for controlled benchmark runs or developer experiments.
4. Add safety checks so host-router enforcement is never silently enabled.

## Dependencies

- 03-01 provider catalog and route selection modes.
- 03-02 owned provider execution lanes.
- 03-03 prompt-cache affinity and accounting.

## Validation

- Shadow-mode tests prove host traffic can be classified and logged without provider mutation.
- Opt-in tests prove enforced routing requires an explicit env/config flag.
- Benchmark artifacts clearly distinguish native host, shadow-routed, and actively routed runs.

## Open questions

- Exact first transport surface: FastAPI/ASGI app, CLI daemon, or MCP-managed sidecar.
- How much request rewriting to support in MVP versus leaving provider-specific rewrites for later.
