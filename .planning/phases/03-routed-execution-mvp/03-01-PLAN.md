# Phase 3: Plan 01 - Provider Catalog And Route Selection Modes

## Problem

Atelier has route recommendations, but not a real owned-execution route contract. The safest MVP starts with explicit provider/model selection as a first-class feature, then adds `auto` mode on top of the same provider catalog and provenance model.

## Scope

- Add or extend a provider/model catalog for owned execution.
- Add explicit provider/model route selection for owned subcalls and benchmarks.
- Add `auto` mode as a first-class route mode, using the same route decision contract.
- Include prompt-cache warmth and cache-eviction cost in auto scoring.

## Milestones

1. Define `RouteRequest`, `RouteDecision`, `RouteMode`, provider capabilities, and explicit selection reasons.
2. Validate explicit provider/model choices against catalog capabilities before execution.
3. Score auto candidates using task class, tool-use need, context size, quality tier, price, latency, provider health, prior failures, and prompt-cache warmth.
4. Add deterministic tests for explicit choices, auto choices, fallback on unhealthy providers, and warm-route stickiness.
5. Emit decision artifacts with enough detail to explain why a route was chosen.

## Dependencies

- Phase 2 owned workflow runner and default definitions.
- Existing `model_routing`, `cross_vendor_routing`, `prefix_cache`, and pricing modules.

## Validation

- Focused unit tests for candidate scoring and route explanations.
- Trace replay tests showing explicit choices are preserved and auto policy chooses the same route from the same inputs.
- No network calls required for policy tests.

## Open questions

- Exact provider catalog file format: Python registry first, config overlay later.
- Whether benchmark solver profiles need stricter quality floors than normal owned subcalls.
