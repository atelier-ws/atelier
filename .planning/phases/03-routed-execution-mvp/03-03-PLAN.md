# Phase 3: Plan 03 - Prompt-Cache Affinity And Accounting

## Problem

Eval's major cost advantage comes from not restarting context cold at every phase. Routing that ignores prompt-cache locality can look cheaper on paper while destroying provider-side cache hits. Atelier needs cache-aware route stickiness and actual cache accounting.

## Scope

- Track cache affinity across related workflow steps.
- Preserve stable prompt prefixes and provider/model/session paths where safe.
- Record cache read/write tokens, cache misses, miss reasons, and modeled eviction cost.
- Use prompt-cache warmth as a route-policy input.

## Milestones

1. Define cache-affinity state for workflow/session lineage.
2. Compute stable prefix hashes from prompt compiler/context blocks.
3. Feed cache warmth and eviction cost into route decisions.
4. Record actual provider usage fields when available, and label modeled estimates when not.
5. Add report output for route decision, cache hit/miss, and warm-route stickiness.

## Dependencies

- 03-01 provider catalog and route selection modes.
- 03-02 owned provider execution receipts.
- Existing `prefix_cache`, `prompt_compilation`, `model_routing/cache_cost.py`, and ledger/reporting code.

## Validation

- Tests prove related workflow steps stay on the warm route when quality and health allow it.
- Tests prove route changes record a cache-eviction reason.
- Usage parser tests distinguish actual cache-read/write evidence from modeled estimates.

## Open questions

- Provider-specific usage field normalization for Anthropic, OpenAI-compatible, Bedrock, and LiteLLM responses.
- Whether cache affinity should be per workflow run, per task lineage, or both.
