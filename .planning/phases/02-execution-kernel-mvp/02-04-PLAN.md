# Phase 2: Plan 04 - Owned Workflow DAG Runner

## Phase Goal

**As a** terminal coding agent user, **I want to** run an Atelier-owned workflow DAG with persistent/forkable step context, **so that** plan -> review -> execute loops reduce prompt churn and are controlled by Atelier rather than host-only hooks.

<objective>
Deliver the owned execution layer missing from the current advisory workflow implementation.

Purpose: satisfy EXEC-06, EXEC-07, EXEC-08, and the workflow-step portion of EXEC-11 by introducing a thin core-owned runner that reuses existing state, tracing, gateway, and report surfaces.

Output: typed workflow schema, step context, DAG execution, safe scheduler, MCP invocation seam, and focused regression coverage.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| workflow definition -> runner | Untrusted or stale workflow definitions drive tool/shell execution |
| step output -> later prompt/template | Prior outputs can influence later commands or agent prompts |
| parallel scheduler -> filesystem | Misclassified writes could race and corrupt state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04-1 | Tampering | workflow schema | mitigate | Validate step kinds, references, command/tool fields, output ids, and unsupported combinations before execution |
| T-02-04-2 | Elevation of Privilege | tool scheduler | mitigate | Parallelize only allowlisted safe tool kinds; serialize writes, shell mutations, and interactive decisions |
| T-02-04-3 | Information Disclosure | step output carry-forward | mitigate | Store compact outputs and reuse existing redaction/reporting paths for user-visible artifacts |
| T-02-SC | Tampering | package installs | mitigate | No package installs in this plan |
</threat_model>
