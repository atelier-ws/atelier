# Spec 05 — Benchmark Publication Pipeline

> Phase 1. One-command export to publishable JSON + markdown.

## Why

Honest, weekly benchmark publication is the marketing engine and the trust signal. To do it sustainably, generating a publishable post must be **one command**, not an afternoon of copy-paste from terminal output.

This spec creates the pipeline: run the existing benchmarks, render them into both data (JSON) and prose (markdown), and stamp a publishable artifact.

## What — user-visible

```bash
$ atelier benchmark publish --since 7d
Running benchmarks:
  ✓ routing savings ........ 50 sessions, $43.61 saved
  ✓ routing quality ........ 0.887 quality score
  ✓ compact savings ........ 18 events, 84K avg freed
  ✓ compact quality ........ 0.810 retention
  ✓ replay (3 sessions × 3 turns, real haiku) ........ 22.2% match

Rendering report:
  ✓ JSON  → reports/2026-W20/benchmark.json
  ✓ MD    → reports/2026-W20/benchmark.md
  ✓ Index → reports/index.json

Done. Preview:
  cat reports/2026-W20/benchmark.md
```

The markdown is **ready to publish** — title, lead, data tables, methodology footnote.

```bash
$ atelier benchmark publish --dry-run     # just render, don't write
$ atelier benchmark publish --output reports/  # custom dir
```

## Output structure

Each run creates a directory `reports/YYYY-Www/` containing:

```
reports/2026-W20/
├── benchmark.json           # all data, machine-readable
├── benchmark.md             # publishable post
├── routing_savings.json     # individual bench results
├── routing_quality.json
├── compact_savings.json
├── compact_quality.json
└── replay.json
```

A top-level `reports/index.json` is appended with metadata for each weekly publication.

## Markdown template structure

The post follows a fixed structure (so readers know what to expect):

```markdown
# Atelier benchmark report — week of YYYY-MM-DD

> Generated by `atelier benchmark publish`. Reproduction: see methodology.

## Headline numbers

| Metric | Value | Δ vs last week |
|--------|-------|----------------|
| Sessions analysed | 50 | +2 |
| Total routing savings | $43.61 | -$8.20 |
| Routing quality score | 0.887 | +0.003 |
| Compact retention score | 0.810 | -0.012 |
| Replay tool-match rate | 22.2% | n/a |

## Routing

Per-vendor downtier rates …

## Compaction

Pre / post error drift …

## Replay counterfactual

Where haiku diverged from sonnet …

## Methodology

How each benchmark was run. Reproducibility command.

## Raw data

[benchmark.json](./benchmark.json)
```

Numbers come from the existing bench JSON outputs. Δ (delta) is computed by comparing against the previous week's `reports/YYYY-Www-1/benchmark.json`.

## Render rules

- Numbers always formatted with thousands separators
- Costs always USD with two decimals, prefixed with `$`
- Percentages with one decimal place
- Δ values colour-coded by direction in terminal (red worse / green better) but plain in markdown
- No marketing language — facts only. The template enforces this by having no marketing slots.

## Methodology block (always included)

The published markdown must always include:
- Number of sessions analysed and source (real Claude exports)
- Pricing assumptions (per-model input/output rates)
- Exclusions (synthetic sessions, sub-agent context flushes)
- Reproduction command (`atelier benchmark publish --since 7d --corpus <path>`)

This is the **trust line** that prevents "they cherry-picked the data" criticism.

## Comparison with prior week

If `reports/YYYY-W<N-1>/benchmark.json` exists, populate Δ columns. If not, show `n/a`.

## Out of scope

- **Automatic posting to dev.to / HN.** Manual paste from the rendered markdown is fine for v1.
- **Charts / graphs.** Markdown tables only. Web dashboard (spec 10) adds visuals.
- **Embargo / scheduling.** Publish-when-ready.
- **Multi-language.** English only for v1.
- **Email distribution.** Future.

## Open questions for the executor

1. Are the benchmarks deterministic enough that re-running on same data produces same numbers? **Default: assume yes; if floating-point drift is significant, round to 4 decimals before comparison.**
2. Should `publish` also include a Claude/Codex/Gemini cross-vendor benchmark? That requires spec 09 (cross-vendor routing). **Default: no for v1 — Atelier-internal benchmarks only.**
3. Should methodology footnote include the exact model versions used? **Default: yes; pull from `pricing.py`.**
