## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

[assistant] OkHttp turns every HTTP request into a walk down a **stack of interceptors**, each of which gets to observe, modify, short-circuit, or retry the request/response. Here's the flow from the code above.

## 1. Entry point: `RealCall.execute()` / `enqueue()`

When you call `call.execute()` (sync) or `call.enqueue()` (async), the request eventually lands in `getResponseWithInterceptorChain()` (`RealCall.kt:208`) — synchronous calls go straight there; async calls route through the `Dispatcher` first, then an `AsyncCall` invokes the same method on a background thread.

## 2. Building the interceptor stack

`getResponseWithInterceptorChain()` assembles the full list **in order** (`RealCall.kt:210-219`):

```kotlin
interceptors += client.interceptors        // your application interceptors
interceptors += RetryAndFollowUpInterceptor()  // retries, redirects, auth follow-ups
interceptors += BridgeInterceptor()        // user headers → HTTP headers (gzip, cookies, etc.)
interceptors += CacheInterceptor()         // serve/store from the HTTP cache
interceptors += ConnectInterceptor         // open a connection / find an exchange
if (!forWebSocket) interceptors += client.networkInterceptors  // your network interceptors
interceptors += CallServerInterceptor      // actually write request & read response
```

So the order is: **application interceptors → built-in stack → network interceptors → the terminal `CallServerInterceptor`.**

## 3. Kicking off the chain

It then constructs a `RealInterceptorChain` starting at `index = 0` and calls `chain.proceed(originalRequest)` (`RealCall.kt:221-232`).

## 4. The recursive `proceed()` mechanism

This is the heart of it — `RealInterceptorChain.proceed()` (`RealInterceptorChain.kt:312`):

```kotlin
override fun proceed(request: Request): Response {
  check(index < interceptors.size)
  ...
  val next = copy(index = index + 1, request = request)  // a chain pointing at the NEXT interceptor
  val interceptor = interceptors[index]                  // the current one
  val response = interceptor.intercept(next)             // hand it the "next" chain
  ...
  return response
}
```

The key idea: each chain instance knows only its own `index`. When `proceed()` is called it:
1. Creates a **copy** of itself advanced to `index + 1` (`next`),
2. Pulls the interceptor at the current `index`,
3. Calls `interceptor.intercept(next)`.

Each interceptor, in its own `intercept(chain)`, does its pre-work, then calls `chain.proceed(request)` to hand off to the next one — which repeats the process. This recursion drives the request *down* the stack until it reaches `CallServerInterceptor` (the last element), which performs the real network I/O and returns a `Response` instead of calling `proceed()` again. That response then unwinds back *up* the stack, and each interceptor can post-process it before returning.

`UppercaseRequestInterceptor` (`UppercaseRequestInterceptor.kt:29`) is a minimal illustration of the contract:

```kotlin
override fun intercept(chain: Chain): Response =
    chain.proceed(uppercaseRequest(chain.request()))
```

It transforms the request, then calls `proceed()` with the new request.

## 5. Guarantees enforced by the chain

`proceed()` also enforces invariants (`RealInterceptorChain.kt:315-340`):
- `calls++` tracks how many times each chain link was used.
- For **network interceptors** (where `exchange != null`), it asserts the interceptor didn't change the host/port and called `proceed()` **exactly once** — so retries and rewrites are only allowed above the connection layer.

## Summary of the request's journey

```
RealCall.execute()
  └─ getResponseWithInterceptorChain()   builds the ordered interceptor list
       └─ RealInterceptorChain(index=0).proceed(request)
            → App interceptors        (observe/rewrite; may retry)
            → RetryAndFollowUpInterceptor
            → BridgeInterceptor       (adds gzip/cookies/Content-Length…)
            → CacheInterceptor        (may short-circuit with a cached response)
            → ConnectInterceptor      (acquires connection + Exchange)
            → Network interceptors    (see the real wire request; exactly one proceed())
            → CallServerInterceptor   (writes request, reads response — terminal)
       ← Response unwinds back up, each interceptor post-processing it
```

The elegance is that the same `proceed()` recursion handles both directions: the request flows down as interceptors call `proceed()`, and the response flows back up as each `intercept()` returns.

The two extension points you plug into are **application interceptors** (`client.interceptors`, added first — see the full request/final response, run even on cache hits) and **network interceptors** (`client.networkInterceptors`, added just before `CallServerInterceptor` — see the actual bytes on the wire and the real connection).
