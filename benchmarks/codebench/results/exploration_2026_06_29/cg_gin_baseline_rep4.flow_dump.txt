## The core idea: one flat handler slice + an index

Gin doesn't use nested wrapper functions for middleware. Instead, **middleware and route handlers are stored together in a single flat slice** (`HandlersChain`, which is just `[]HandlerFunc` — `gin.go:57`), and the `Context` walks that slice with a cursor.

### 1. Building the chain (at registration time)

When you register a route, gin merges the group's middleware with the route's own handlers via `combineHandlers` (`routergroup.go:241`):

```go
mergedHandlers := make(HandlersChain, finalSize)
copy(mergedHandlers, group.Handlers)                  // middleware first
copy(mergedHandlers[len(group.Handlers):], handlers)  // route handlers last
```

So a route ends up with one pre-computed slice like `[Logger, Recovery, Auth, finalHandler]`. This merged chain is stored in the radix routing tree (`addRoute` → `tree.go`). There's no per-request allocation of the chain — it's built once.

### 2. Dispatch (per request)

`ServeHTTP` (`gin.go:662`) grabs a pooled `Context`, resets it, and calls `handleHTTPRequest` (`gin.go:690`). That function:
1. Picks the routing tree for the HTTP method (`engine.trees`).
2. Looks up the path via `root.getValue(...)` (radix-tree match).
3. On a hit, copies the matched chain onto the context and kicks it off:

```go
c.handlers = value.handlers
c.fullPath = value.fullPath
c.Next()
```

(Misses fall through to trailing-slash redirects, 405 `allNoMethod`, or 404 `allNoRoute`, which are themselves combined chains.)

### 3. Walking the chain — `Context.Next()`

This is the heart of it (`context.go:188`):

```go
func (c *Context) Next() {
	c.index++
	for c.index < safeInt8(len(c.handlers)) {
		if c.handlers[c.index] != nil {
			c.handlers[c.index](c)
		}
		c.index++
	}
}
```

The context holds an `index` cursor (reset to `-1` for a fresh request). `handleHTTPRequest` calls `Next()` once; `index` becomes `0` and the loop runs each handler in order.

The clever part is **re-entrancy**: a middleware can itself call `c.Next()` in its body. That inner call advances the cursor and runs *the rest of the chain* to completion before returning, so any code in the middleware **after** `c.Next()` runs on the way back out. That's what gives you the classic "wrap" behavior without wrappers:

```go
func Logger(c *gin.Context) {
	start := time.Now()
	c.Next()                       // run downstream handlers
	log.Print(time.Since(start))   // runs after they return
}
```

If a middleware doesn't call `Next()`, the outer loop in the *original* `Next()` call just continues to the next handler anyway — so calling `Next()` is only required when you need the post-processing behavior.

### 4. Short-circuiting — `Abort()`

`Abort()` (`context.go:207`) sets `c.index = abortIndex` (a large sentinel value, 63). That pushes the cursor past the end of any real chain, so the `for` loop condition in `Next()` fails and **no further handlers run**. Note it does *not* stop the currently executing handler — you typically `return` after calling it (or use helpers like `AbortWithStatusJSON`). `IsAborted()` (`context.go:199`) just checks `index >= abortIndex`.

### Why this design

- **Index, not call stack of wrappers** → cheap, and the chain is a contiguous slice that's friendly to CPU cache.
- **Context pooling** (`sync.Pool` in `ServeHTTP`) → almost zero per-request allocation.
- **Chain pre-merged into the route tree** → middleware resolution is just a slice copy, not a composition step per request.

One subtle constraint worth noting: because `index` is an `int8` and `abortIndex`/`safeInt8` cap things at 63, a single route's combined middleware + handler count must stay under that limit — `combineHandlers` asserts `finalSize < abortIndex` (`routergroup.go:243`).
