## The big picture

Tokio has two schedulers behind one API: the **multi-thread** scheduler (the default `#[tokio::main]` runtime) and the **current-thread** scheduler. Both share the same task machinery (`task::Notified`, the `Harness`, the injection queue) but differ in how many worker loops run and where the run queue lives. Everything centers on one idea: a **task is a heap allocation that can be cheaply scheduled by pushing a handle onto a queue**, and workers loop popping handles and polling them.

## The multi-thread scheduler

### Worker threads and the run loop

Each worker thread runs `run()` (`multi_thread/worker.rs:508`), which acquires an exclusive **`Core`** (`worker.core.take()` — line 528; if another thread already holds it, this thread just returns). The `Core` is the "capability token" that grants the right to schedule: it owns the local run queue, the LIFO slot, stats, and the park handle. It's stored in a thread-local `Context` so that code polling inside a task can reach back into the scheduler.

The heart is `Context::run` (`worker.rs:561`), a loop that runs until shutdown:

```rust
while !core.is_shutdown {
    core.tick();
    core = self.maintenance(core);
    if let Some(task) = core.next_task(&self.worker) {   // 1. local/global work
        core = self.run_task(task, core)?;
        continue;
    }
    if let Some(task) = core.steal_work(&self.worker) {  // 2. steal from peers
        core = self.run_task(task, core)?;
    } else {
        core = self.park(core);                          // 3. sleep
    }
}
```

### Where a worker looks for work (in priority order)

**1. `next_task` (`worker.rs:1062`)** — the local sources:
- The **LIFO slot** first, then the local run queue (`next_local_task`, line 1132). The LIFO slot is a single-task fast path (explained below).
- Periodically — every `global_queue_interval` ticks — it checks the **global injection queue** *first* instead, so global work can't be starved by a hot local queue. When it does pull from the inject queue while idle, it grabs a *batch* (`inject().len() / num_workers + 1`, capped at half the local queue capacity, lines 1109–1126): returns one task directly and pushes the rest onto its local queue.

**2. `steal_work` (`worker.rs:1141`)** — work-stealing. If local + global are empty, the worker transitions to the "searching" state and tries to steal. It starts at a **random** peer and scans round-robin, calling `steal_into` to move roughly half of a victim's queue into its own. As a fallback it checks the global queue again. Only a limited number of workers search at once (`transition_to_searching`) so they don't thundering-herd.

**3. `park` (`current_thread/mod.rs:380`-style, and multi-thread equivalent)** — if nothing is found, the worker parks on the I/O/timer driver. Before actually sleeping it re-checks for pending work; the last searching worker double-checks all queues (`transition_to_parked`, line 1204) to close the race where work appears just as everyone goes to sleep.

### How a task actually runs

`run_task` (`worker.rs:630`) is where a `Notified` handle becomes a poll:

1. `assert_owner` confirms the task belongs to this scheduler.
2. `transition_from_searching` — the worker leaves the searching state and, if it was the last searcher, **notifies another parked worker** so stealing can continue in parallel with this poll.
3. It puts the `Core` back into the thread-local context (so nested scheduling works), then calls `coop::budget(|| task.run())` — `task.run()` drives the actual `Future::poll` through the `Harness`.
4. **The LIFO optimization** (lines 689–717): after polling, it checks the LIFO slot again and keeps polling those tasks in a tight loop *without going back through the queue*, up to `MAX_LIFO_POLLS_PER_TICK` and while coop budget remains. This is the message-passing fast path: when task A does `tx.send()` and wakes task B, B lands in the LIFO slot and runs immediately on the same worker with warm cache, instead of round-tripping through the queue.

Note the poll can lose the core mid-loop (`self.core.borrow_mut().take()` returns `None` at line 696) — that happens when the task called `block_in_place`, handing the core to another thread. The loop then returns `Err(())`, unwinding back to `run()`.

## How a task gets scheduled (the wake path)

When a future is spawned or a waker fires, the scheduler's `schedule_task` (`worker.rs:1327`) decides where the `Notified` handle goes:

- **If called from a worker thread of this same runtime that still holds a core** → `schedule_local` (line 1353): a non-yield wake goes to the **LIFO slot** (bumping any prior occupant to the run queue); a yield (`yield_now`) or LIFO-disabled goes to the **back of the local run queue** via `push_back_or_overflow`. If the local queue is full, overflow moves half its contents to the global inject queue (`queue.rs:188`).
- **Otherwise** (scheduling from a non-worker thread, another runtime, or a resource driver) → `push_remote_task` pushes onto the **global injection queue** under a lock (`worker.rs:1397`), then `notify_parked_remote` unparks a sleeping worker to come pick it up.

This dual path is why waking a task from inside the runtime is nearly lock-free (local queue / LIFO slot), while waking from outside pays a lock on the shared inject queue.

## The task harness — what `task.run()` does

`task.run()` ends up in `Harness::poll` (`task/harness.rs:153`). The harness wraps the future in a state machine (`State`/`Snapshot`) that tracks ref-counts and the RUNNING/NOTIFIED/COMPLETE bits. `poll_inner` builds a `Waker` pointing back at this task (so waking = re-scheduling via `Schedule::schedule`), polls the future, and dispatches on the result:

- **`Complete`** → store the output, wake the `JoinHandle`'s waker, drop the future.
- **`Notified`** → the task woke *itself* during its own poll; it's handed to `yield_now` to be re-scheduled fairly rather than looping hot.
- **`Dealloc` / `Done`** → ref-count bookkeeping.

The `Waker` is the entire bridge between "the outside world is ready" (an I/O readiness event, a timer firing, a channel send) and "put this task back on a run queue" — waking simply calls back into `schedule_task`.

## The current-thread scheduler (contrast)

`current_thread/mod.rs` is the same shape collapsed to one thread: the `Core` holds a simple `VecDeque<Notified>` run queue (line 63), `next_task` alternates local vs. global by tick (line 324), and `run_task` (line 371) polls under a coop budget just like the multi-thread one — but there's no stealing, no LIFO slot, and no worker-to-worker notification. When idle it parks the single thread directly on the driver (`park`, line 380).

## Cross-cutting mechanisms

- **Cooperative scheduling (`coop::budget`)** wraps every poll. Each task gets a budget of "readiness" operations; tokio's own resources decrement it and return `Pending` once exhausted, forcing long-running tasks to yield so they can't starve peers — this is what makes a single busy task not hog a worker.
- **`block_in_place`** lets a task give up its core (handing it to another thread to keep the runtime live) while it blocks — that's the `None` branch in `run_task`.
- **`spawn_blocking`** is separate entirely: a dedicated thread pool (`blocking/pool.rs:503`) with its own condvar-driven BUSY/IDLE loop, for work that would otherwise block a worker.

**In one sentence:** spawning/waking pushes a task handle onto a queue (LIFO slot → local queue → global inject, in decreasing locality); each worker loops popping the most-local task, polls it through the harness under a coop budget, prefers just-woken tasks via the LIFO slot, steals from peers when idle, and parks on the I/O driver when there's nothing left to do.
