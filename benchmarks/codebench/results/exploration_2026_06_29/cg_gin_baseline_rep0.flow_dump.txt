[assistant] Here's how gin routes a request through its middleware chain, traced through the actual code in this repo.

## The core idea: middleware and handlers are one flat slice

Gin doesn't distinguish "middleware" from "handlers" at execution time. Both are just `HandlerFunc` values stored together in a `HandlersChain` (a `[]HandlerFunc`), where **the last element is the actual route handler** and everything before it is middleware (`gin.go:56-65`).

## 1. Build time: chains are assembled when routes are registered

When you register a route, gin pre-computes the *entire* chain for that route by concatenating the group's accumulated middleware with the route's handlers:

- `RouterGroup.Use()` appends middleware to `group.Handlers` (`routergroup.go:65-68`).
- When you call `GET`/`POST`/etc., it funnels into `handle()`, which calls `combineHandlers()` (`routergroup.go:86-91`).
- `combineHandlers()` allocates one slice = `group.Handlers` (middleware) **followed by** the route handlers (`routergroup.go:241-248`).
- That combined chain is stored in the route's radix-tree node via `engine.addRoute()` → `root.addRoute()` (`gin.go:364-386`).

So by the time the server is running, each route node already holds a fully-resolved `[middleware..., finalHandler]` slice. Groups nest by combining at each level. Global `engine.Use()` middleware also gets baked into the `allNoRoute`/`allNoMethod` chains for 404/405 (`gin.go:340-362`).

## 2. Request time: match the route, install the chain

Every request enters `Engine.ServeHTTP` (`gin.go:662-675`):

1. A `Context` is pulled from a `sync.Pool` (reused to avoid allocations) and reset.
2. `handleHTTPRequest(c)` does the actual routing (`gin.go:690-760`):
   - Picks the radix tree for the HTTP method (`engine.trees`, one tree per method).
   - `root.getValue(rPath, ...)` walks the tree to find the matching node.
   - On a match, it copies the node's pre-built chain onto the context: `c.handlers = value.handlers`, sets `c.fullPath`, then calls **`c.Next()`** (`gin.go:719-724`).
   - If nothing matches, it sets `c.handlers` to `allNoMethod` (405) or `allNoRoute` (404) and runs those chains via `serveError` (`gin.go:738-759`).

## 3. Execution: the `Next()` index walk

The chain is driven by an index cursor on the Context (`context.go:188-196`):

```go
func (c *Context) Next() {
	c.index++
	for c.index < safeInt8(len(c.handlers)) {
		if c.handlers[c.index] != nil {
			c.handlers[c.index](c)
		}
		c.index++
	}
}
```

`c.index` starts at `-1` (set in `reset()`, `context.go:107`). The first `Next()` call advances to `0` and loops through the chain. The key behavior: **when a middleware itself calls `c.Next()`, it re-enters this loop and runs all subsequent handlers before returning** — that's what gives middleware its "around" semantics. Code before `c.Next()` runs on the way in; code after it runs on the way out, unwinding back through the stack. A middleware that *doesn't* call `Next()` still lets the outer loop continue to the next handler automatically.

## 4. Short-circuiting: `Abort`

`Abort()` sets `c.index = abortIndex` (`math.MaxInt8 >> 1 = 63`) (`context.go:57, 207-209`). Because the `Next()` loop condition is `c.index < len(c.handlers)`, jumping the index past the chain length stops any pending handlers from running. Note Abort does *not* stop the currently-executing handler — only pending ones. `IsAborted()` just checks `c.index >= abortIndex`. This `abortIndex` value also caps how many handlers a chain may hold (`combineHandlers` asserts `finalSize < abortIndex`, `routergroup.go:243`).

## Summary flow

```
Use()/Group() → combineHandlers() bakes [mw…, handler] into each tree node (build time)
        │
ServeHTTP → handleHTTPRequest → tree.getValue() matches route
        │
   c.handlers = node chain;  c.Next()
        │
   Next() walks index 0→N, each middleware optionally calls Next()
   to wrap downstream handlers; Abort() jumps index past the end
```

The elegant part is that there's no separate middleware dispatcher — it's one pre-allocated slice per route plus a single integer cursor, which is why gin's middleware layer is so cheap at request time.
