## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

## The multi-canvas architecture

Excalidraw doesn't use one `<canvas>` — it layers several, each with its own render path and cache-invalidation strategy. Two matter most:

- **`StaticCanvas`** (`components/canvases/StaticCanvas.tsx`) — the shapes themselves. Expensive to draw, changes rarely.
- **`InteractiveCanvas`** (`components/canvases/InteractiveCanvas.tsx`) — selection boxes, handles, remote collaborator cursors, snap lines. Changes constantly.

Splitting them means moving your mouse over a selection doesn't force every shape to redraw. There's also a `NewElementCanvas` for the element currently being drawn.

## The single source of truth: `Scene`

`Scene` (`packages/element/src/Scene.ts`) owns the element arrays and derived maps (`elementsMap`, `nonDeletedElements`, etc.). The `App` component instantiates it once in its constructor (`App.tsx:827`) along with a `Renderer` and the React `Store`/`History`.

Every mutation funnels through `Scene`, and the key invalidation primitive is `triggerUpdate()` (`Scene.ts:303`):

```ts
triggerUpdate() {
  this.sceneNonce = randomInteger();   // <-- bumps the nonce
  for (const callback of this.callbacks) callback();
}
```

The bumped `sceneNonce` is what lets downstream memoization know "something changed," and the callbacks (registered via `onUpdate`) are wired to `App.triggerRender`.

## The update flow (element → screen)

1. **Mutation.** Code calls `app.mutateElement(el, updates)` (`App.tsx:4644`), which delegates to `scene.mutateElement` (`Scene.ts:411`). That applies the update, and only if the element is in the scene, its `version` actually changed, and `informMutation` is set, does it call `triggerUpdate()`. Bulk changes go through `replaceAllElements` (`Scene.ts:271`), which rebuilds the maps and also calls `triggerUpdate()`.

2. **Scene → React.** `triggerUpdate` fires the `onUpdate` callback, wired to `triggerRender` (`App.tsx:4655`), which calls `this.setState({})` — a no-op state change purely to force a React re-render. (`triggerRender(true)` forces a scene update even without a state change.)

3. **Compute what's renderable.** In `App.render()` (`App.tsx:2091`), `this.renderer.getRenderableElements(...)` runs. The `Renderer` (`scene/Renderer.ts`):
   - Builds a `canvasNonce` from the scene nonce (`Renderer.ts:210`).
   - `_getRenderableElements` is **`memoize`d** — if the nonce and viewport params are unchanged, it returns the cached result, so no recomputation happens.
   - It filters out the currently-edited text element, computes `visibleElements` via `isElementInViewport` (viewport culling — off-screen shapes aren't drawn), and reorders elements when dragging into a frame.

4. **React reconciliation gate.** `StaticCanvas` is wrapped in `React.memo` with a custom `areEqual` (`StaticCanvas.tsx:108`). It bails out of re-rendering unless `canvasNonce`, `scale`, `elementsMap`/`visibleElements` identity, or a shortlist of relevant appState props changed. This is the second layer of "don't redraw if nothing visible changed."

5. **Draw to canvas.** When `StaticCanvas` does render, its `useEffect` calls `renderStaticScene(...)` (`renderer/staticScene.ts:491`), passing `isRenderThrottlingEnabled()`. That dispatches to either `renderStaticSceneThrottled` (a `throttleRAF`-wrapped version, capping to animation frames) or `_renderStaticScene` directly. `_renderStaticScene` clears the canvas, applies frame clipping, and loops over `visibleElements` calling `renderElement` (`packages/element/src/renderElement.ts:780`) for each shape and any bound text.

6. **Interactive layer, separately.** `InteractiveCanvas`'s effect builds collaborator/selection render config and drives `renderInteractiveScene` through an `AnimationController` RAF loop (`InteractiveCanvas.tsx:173`) — independent of the static shapes.

## The mental model

```
mutateElement / replaceAllElements
        │
   Scene.triggerUpdate()  ── bumps sceneNonce, fires callbacks
        │
   App.triggerRender()  ── setState({})
        │
   App.render() ── Renderer.getRenderableElements()  [memoized on nonce + viewport cull]
        │
   <StaticCanvas> React.memo(areEqual)  [skip if nonce/props unchanged]
        │
   renderStaticScene() ── throttleRAF ── loop renderElement() → <canvas>
```

The recurring theme is **layered invalidation**: a nonce bump, a memoized renderable-elements computation, viewport culling, a `React.memo` gate, and RAF throttling — so a change to one element does the minimum work needed to get pixels on screen, and the separate interactive canvas keeps cursor/selection updates from touching the shape rendering at all.
