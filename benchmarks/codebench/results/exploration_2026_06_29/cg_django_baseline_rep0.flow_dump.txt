## The big picture: three objects, two phases

There are three distinct objects, and understanding the division of labor between them is the key:

1. **`QuerySet`** (`django/db/models/query.py`) — the public, *lazy* API you interact with.
2. **`Query`** (`django/db/models/sql/query.py`) — an in-memory, database-agnostic *representation* of the SQL (a tree of Python objects: `WhereNode`, joins, aliases).
3. **`SQLCompiler`** (`django/db/models/sql/compiler.py`) — turns a `Query` into an actual SQL string + params, and runs it.

And two phases: **building** (lazy, no DB contact) and **execution** (triggered by iteration).

---

## Phase 1: Building (lazy)

When you write `Model.objects.filter(...).exclude(...)`, no SQL runs. Each call clones and mutates the abstract `Query`:

- `QuerySet.filter()` / `exclude()` (`query.py:1648`, `:1656`) both funnel into `_filter_or_exclude()` (`:1664`).
- That first calls `_chain()` → `_clone()` (`:2200`, `:2218`), which clones the `QuerySet` *and* its `Query` via `Query.chain()` (`sql/query.py:433`). **This is why QuerySets are immutable/chainable** — every filter returns a fresh object.
- Then `_filter_or_exclude_inplace()` (`:1675`) calls `self._query.add_q(Q(...))`.

Inside the `Query`:
- `add_q()` (`sql/query.py:1663`) → `_add_q()` (`:1692`) recursively walks the `Q` object tree.
- For each leaf, `build_filter()` (`:1488`) resolves the field lookup (e.g. `name__startswith`), creates `Lookup`/join objects, and produces a `WhereNode`.
- These get combined into `self.where` (a `WhereNode`, from `sql/where.py`), with joins tracked in `self.alias_map` and `self.alias_refcount`.

At the end of this phase you have a `Query` holding the SELECT columns, a `WhereNode` tree, join/alias map, ordering, etc. — but still **zero database access**.

---

## What triggers execution

Evaluation happens lazily on first need — iteration (`for x in qs`), `list(qs)`, `bool(qs)`, slicing, etc. All routes go through:

- `QuerySet.__iter__()` (`:420`) → `_fetch_all()` (`:2238`)
- `_fetch_all()` populates `self._result_cache = list(self._iterable_class(self))`. The default iterable class is **`ModelIterable`** (`:89`). The cache is why iterating a QuerySet twice only hits the DB once.

---

## Phase 2: Compilation + execution

`ModelIterable.__iter__()` (`:92`) drives the rest:

**1. Get a compiler** — `query.get_compiler(using=db)` (`sql/query.py:370`) grabs the DB connection and instantiates a backend-specific `SQLCompiler`. This is where the database-agnostic `Query` meets a concrete database.

**2. Generate SQL** — `compiler.execute_sql()` (`compiler.py:1595`) calls `as_sql()` (`:757`), the heart of compilation:
   - `pre_sql_setup()` (`:83`) → `setup_query()` → `get_select()` resolves the SELECT columns, ordering, and grouping; it also splits conditions into WHERE vs. HAVING.
   - `compile(self.where)` (`:574`) dispatches to each node's `as_sql()`, turning the `WhereNode` tree into a `"WHERE ..."` string with parameter placeholders.
   - `get_from_clause()` (`:1141`) walks `alias_map` to emit `FROM table JOIN ... ON ...`.
   - Everything is assembled into `SELECT ... FROM ... WHERE ... GROUP BY ... HAVING ... ORDER BY ... LIMIT ...`, returned as a **`(sql_string, params)` tuple**.

   Note the SQL string uses placeholders and params are kept separate — that's how Django gets parameterized queries (SQL injection protection) for free.

**3. Run it** — `execute_sql()` calls `cursor.execute(sql, params)` (`compiler.py:~1625`) against the raw DB-API cursor and fetches rows (`fetchmany(chunk_size)` for chunked iteration).

**4. Materialize objects** — `compiler.results_iter()` (`:1565`) applies backend type converters and yields raw row tuples. Back in `ModelIterable`, each tuple is turned into a model instance via `model_cls.from_db(...)`, related/annotated fields are populated, and the instance is yielded — then cached in `_result_cache`.

---

## End-to-end

```
.filter()/.exclude()        →  clone QuerySet + Query, add_q() → build_filter() → WhereNode   [lazy]
─────────────────────────────────────────────────────────────────────────────────────────────
iterate / list() / bool()   →  _fetch_all() → ModelIterable.__iter__()                        [triggers]
   get_compiler(using=db)   →  SQLCompiler(query, connection)
   execute_sql() → as_sql() →  pre_sql_setup → compile(where) + get_from_clause → (sql, params)
   cursor.execute(sql,params)→  DB returns rows
   results_iter() + from_db →  model instances → cached in _result_cache
```

The clean separation is the design payoff: `QuerySet` is the lazy/chainable API, `Query` is the portable abstract representation, and `SQLCompiler` is the only piece that knows about a specific database dialect — which is how Django supports multiple backends from one query-building API.
