## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

[assistant] Excalidraw uses a **two-layer canvas + React-driven render pipeline** with a single source of truth (`Scene`) and imperative, throttled canvas painting. Here's the full cycle.

## The data model: `Scene`

`Scene` (`packages/element/src/Scene.ts`) is the source of truth. It holds `elements` plus derived caches (`elementsMap`, `nonDeletedElements`, `frames`, etc.). The `App` instance owns one Scene, a `Renderer`, and two `<canvas>` elements (`App.tsx:635-637, 827-831`).

## Updating an element

All mutations funnel through `Scene.mutateElement` (`Scene.ts:411`):

1. It calls the low-level `mutateElement` (from `@excalidraw/element`), which applies the `updates` and bumps the element's `version`/`versionNonce`.
2. If the element is actually in the scene, its version changed, and `informMutation` is set, it calls `triggerUpdate()`.

`triggerUpdate()` (`Scene.ts:303`) assigns a new random `sceneNonce` and fires every registered callback. Bulk changes go through `replaceAllElements` (`Scene.ts:271`), which rebuilds all the caches and then also calls `triggerUpdate()`.

The public `app.mutateElement` (`App.tsx:4644`) is just a thin wrapper over `scene.mutateElement`.

## From mutation to a React re-render

The key wiring is a callback: `App` registers `triggerRender` on `scene.onUpdate` (at `App.tsx:3147`). So the chain is:

```
mutateElement → Scene.triggerUpdate() → callback → App.triggerRender() → setState({}) → React re-render
```

`triggerRender` (`App.tsx:4655`) forces a React update. **React is used only to schedule and orchestrate**, not to paint pixels.

## `render()`: computing what's renderable

`App.render()` (`App.tsx:2091`) calls `this.renderer.getRenderableElements(...)` (`Renderer.ts:208`) with the current viewport (zoom, scroll, dimensions) and selection state. The `Renderer`:

- Computes a `canvasNonce` from the scene nonce (`Renderer.ts:210`).
- `_getRenderableElements` is **memoized** (`Renderer.ts:159`) on the nonce + viewport — so if nothing changed, it returns the same references and downstream canvases skip repainting.
- It builds the `elementsMap` (excluding the element being text-edited), filters to `visibleElements` actually in the viewport via `isElementInViewport` (`Renderer.ts:47`), and handles frame-reordering during drags.

`render()` then passes these props into three canvas components (`App.tsx:2331-2411`):
- **`<StaticCanvas>`** — committed elements (the main drawing).
- **`<NewElementCanvas>`** — the element currently being drawn, isolated for perf.
- **`<InteractiveCanvas>`** — selection boxes, handles, remote cursors, snap lines.

## Painting the pixels (imperative + throttled)

The canvas components paint in a `useEffect`, comparing nonces to decide whether to repaint:

- **StaticCanvas** calls `renderStaticScene` (`staticScene.ts:491`), which delegates to `renderStaticSceneThrottled` (throttled to animation-frame rate, `staticScene.ts:482`). `_renderStaticScene` (`staticScene.ts:229`) does the actual work: normalizes canvas dimensions, applies zoom, draws the grid, then iterates `visibleElements` calling `renderElement` per element (handling frame clipping, bound text, embeddables on top, and link icons). Rough.js (`rc`) provides the hand-drawn look.
- **InteractiveCanvas** calls `renderInteractiveScene` (`interactiveScene.ts:2082`) similarly for the overlay layer.

## Why this design

- **Separation of layers** means a selection drag or cursor move only repaints the cheap interactive canvas, leaving the static drawing untouched.
- **Nonce-based memoization** at two levels (`Renderer` memoize + per-canvas nonce checks) prevents redundant work — a re-render that changed nothing paints nothing.
- **rAF throttling** caps painting at the display's frame rate regardless of how many mutations fire.

**In one line:** `mutateElement` bumps a version and fires a scene callback → `App.triggerRender` does a React `setState` → `render()` asks the memoized `Renderer` for visible elements → the static/interactive canvas components imperatively repaint via rAF-throttled `renderStaticScene`/`renderInteractiveScene`.
