## The big picture

Alamofire is organized around three actors that pass work between them on a set of **serial dispatch queues**:

- **`Session`** ŌĆö the factory and coordinator. Builds requests, drives the lifecycle, owns the `URLSession`.
- **`Request`** (and subclasses `DataRequest`, `UploadRequest`, `DownloadRequest`, `DataStreamRequest`) ŌĆö a state machine wrapping one logical request, plus all the `URLRequest`s and `URLSessionTask`s it spawns over retries.
- **`SessionDelegate`** ŌĆö receives raw `URLSession` callbacks and routes them back to the right `Request` via a `RequestTaskMap`.

Everything that mutates shared state runs on `Session.rootQueue` (a serial queue); request building runs on `requestQueue` and response work on `serializationQueue`, both of which *target* `rootQueue`. State itself is guarded with a `Protected<T>` lock. This is why callbacks fire in a deterministic order and there are no data races.

## 1. Build

**Entry point** ŌĆö `Session.request(...)` in `Source/Core/Session.swift` has several overloads (lines ~318ŌĆō407). They differ only in how parameters are described:

- Dictionary parameters + `ParameterEncoding` ŌåÆ wrapped in a `RequestConvertible`
- `Encodable` parameters + `ParameterEncoder` ŌåÆ wrapped in a `RequestEncodableConvertible`

Both wrappers conform to **`URLRequestConvertible`** (`Source/Core/URLConvertible+URLRequestConvertible.swift`), whose single job is `func asURLRequest() throws -> URLRequest`. All overloads converge to construct a `DataRequest` and hand it to the session.

**Two ways to encode parameters** ŌĆö note these are *parallel* APIs:
- **`ParameterEncoding`** (`ParameterEncoding.swift`) works on untyped `[String: Any]` dictionaries ŌåÆ `URLEncoding`, `JSONEncoding`.
- **`ParameterEncoder`** (`ParameterEncoder.swift`) works on strongly-typed `Encodable` values via Swift's `Codable` ŌåÆ `JSONParameterEncoder`, `URLEncodedFormParameterEncoder`.

The actual `URLRequest` isn't built when you call `request()`. It's built lazily inside `Session.performSetupOperations(...)` (Session.swift ~1246), which calls `convertible.asURLRequest()`, then `initialRequest.validate()` (sanity-checks the URL/method).

**Adaptation** ŌĆö if a `RequestInterceptor`/`RequestAdapter` is attached, `performSetupOperations` calls `adapter.adapt(initialRequest, using: state) { result in ... }` *asynchronously* before the task is created. This is where things like auth-token injection happen. `RequestInterceptor` is just `RequestAdapter & RequestRetrier` (`Source/Features/RequestInterceptor.swift`), and the `Interceptor` class chains multiple adapters sequentially.

The `Request` subclass chosen determines the eventual task type: `DataRequest` ŌåÆ `dataTask`, `UploadRequest` (a `DataRequest` subclass) ŌåÆ `uploadTask`, `DownloadRequest` ŌåÆ `downloadTask`, `DataStreamRequest` ŌåÆ streamed `dataTask`.

## 2. Send

Calling **`request.resume()`** (`Source/Core/Request.swift` ~768) flips the state `.initialized ŌåÆ .resumed`. If no task exists yet, it calls `delegate.readyToPerform(request:)`, which lands in **`Session.perform(...)`** (Session.swift ~1166):

1. On `rootQueue`, inserts the request into `activeRequests`.
2. On `requestQueue`, dispatches to `performDataRequest` / `performUploadRequest` / etc. by type.
3. That runs `performSetupOperations` (build + adapt, above), then back on `rootQueue` calls **`didCreateURLRequest(_:for:)`** (~1291):
   - `task = request.task(for: urlRequest, using: session)` ŌĆö e.g. `session.dataTask(with:)`
   - records the mapping in `RequestTaskMap` (a bidirectional `Request Ōåö URLSessionTask` map, `RequestTaskMap.swift`)
   - calls `request.didCreateTask(task)`

**`Request.didCreateTask`** (~388) inspects the request's current state and resumes the underlying `URLSessionTask` *only if* the request is `.resumed` ŌĆö so a request suspended/cancelled before its task existed behaves correctly. Now the OS executes the network call.

**Callbacks come back** through `SessionDelegate` (`SessionDelegate.swift`), e.g. `urlSession(_:task:didCompleteWithError:)`. It looks up the originating `Request` via the task map (`stateProvider?.request(for: task)`) and calls `request.didCompleteTask(task, with:)`.

**Retry** ŌĆö after completion, `Request.retryOrFinish(error:)` (~543) asks `delegate.retryResult(for:dueTo:)`, which consults the `RequestRetrier`. On `.retry` / `.retryWithDelay(t)`, `Session.retryRequest` calls `request.prepareForRetry()` and re-enters `perform(request, forRetry: true)` ŌĆö looping back to step 1 with a fresh `URLRequest`/task. On `.doNotRetry`, it calls `finish()`.

Throughout, an **`EventMonitor`** (`Source/Features/EventMonitor.swift`, dispatched via `CompositeEventMonitor`) observes every transition ŌĆö `didCreateTask`, `didResume`, `isRetrying`, `didFinish` ŌĆö without participating in control flow. This is what powers logging/metrics.

## 3. Validate

Validation and serialization are deliberately **ordered: validate first, serialize second.**

**`.validate()`** (`Source/Features/Validation.swift`) doesn't run immediately ŌĆö it *appends a closure* to the request's `validators` array (`Request.validators`, a `Protected<[() -> Void]>`). The variants:
- `validate(statusCode:)` ŌĆö checks the code is in an acceptable sequence, else `AFError.responseValidationFailed(.unacceptableStatusCode)`.
- `validate(contentType:)` ŌĆö MIME-type match with wildcard support.
- `validate()` ŌĆö the default: status `200..<299` and content type derived from the `Accept` header.

A `ValidationResult` is just `Result<Void, Error & Sendable>`.

**When validators run** ŌĆö inside `Request.didCompleteTask` (~515), right when the task finishes and *before* any retry/serialization:
```swift
let validators = validators.read(\.self)
validators.forEach { $0() }   // each sets self.error on failure
...
retryOrFinish(error: self.error)
```
So a failed validation sets `self.error`, which can then trigger a retry, and otherwise gets carried into serialization.

**Serialization** ŌĆö `.response`, `.responseDecodable(of:)`, etc. append a *response serializer* (`Source/Features/ResponseSerialization.swift`). After `finish()`, `processNextResponseSerializer()` runs them one at a time on `serializationQueue`. Each serializer's `serialize(request:response:data:error:)` receives `self.error` and **short-circuits if it's non-nil**:
```swift
guard error == nil else { throw error! }
```
That's the mechanism by which a validation failure propagates to your completion handler instead of a decoded value. Built-in serializers include `DataResponseSerializer`, `StringResponseSerializer`, and `DecodableResponseSerializer` (uses a `DataDecoder`, `JSONDecoder` by default, with `DataPreprocessor` and empty-response handling).

**Delivery** ŌĆö the serializer's output is wrapped into a `Result<Success, AFError>`, packaged into a **`DataResponse`** (`Source/Core/Response.swift` ŌĆö carrying `request`, `response`, `data`, `metrics`, `serializationDuration`, `result`), and the completion handler is invoked on the caller's queue (default `.main`). `AFDataResponse<T>` is just `DataResponse<T, AFError>`.
