## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

## Overview

The flow spans four core types: `Session` (orchestration), a `URLRequestConvertible` (request building), `Request`/`DataRequest` (per-request lifecycle & state machine), and `SessionDelegate` (bridge from `URLSession` callbacks back into the `Request`).

## 1. Build

Everything a request needs to become a `URLRequest` is captured behind the `URLRequestConvertible` protocol. When you call `Session.request(...)` with URL/method/parameters, Alamofire wraps them in a `RequestConvertible` struct (`Source/Core/Session.swift:286`):

```swift
struct RequestConvertible: URLRequestConvertible {
    func asURLRequest() throws -> URLRequest {
        var request = try URLRequest(url: url, method: method, headers: headers)
        try requestModifier?(&request)          // your inout mutation hook
        return try encoding.encode(request, with: parameters)   // ParameterEncoding
    }
}
```

`Session.request(_:interceptor:...)` (`Session.swift:393`) constructs a `DataRequest` from that convertible and hands it the `rootQueue`, `serializationQueue`, `eventMonitor`, interceptor, and `self` as `RequestDelegate`. The actual `URLRequest` isn't built yet — the `DataRequest` is returned immediately (lazy), and `performEagerlyIfNecessary` kicks off setup on the `requestQueue`.

The real building happens in `Session.performSetupOperations` (`Session.swift:1246`):

```swift
initialRequest = try convertible.asURLRequest()   // build it
try initialRequest.validate()                     // request-side validation
...
rootQueue.async { request.didCreateInitialURLRequest(initialRequest) }
```

Note the two-stage validation-on-*build*: `URLRequest.validate()` (`Source/Extensions/URLRequest+Alamofire.swift:34`) rejects illegal requests up front — e.g. a GET with a body throws `.bodyDataInGETRequest`. If building throws, the request short-circuits to `didFailToCreateURLRequest`.

## 2. Adapt (interceptors)

Still in `performSetupOperations`, if an interceptor/adapter applies (`adapter(for:)` at `Session.swift:1315` merges session + per-request interceptors into one `Interceptor`), the built request is run through `adapter.adapt(...)`:

```swift
adapter.adapt(initialRequest, using: adapterState) { result in
    let adaptedRequest = try result.get()
    try adaptedRequest.validate()               // re-validate adapted request
    self.rootQueue.async { self.didCreateURLRequest(adaptedRequest, for: request) }
}
```

Adapters chain recursively (`RequestInterceptor.swift:290`), each transforming the `URLRequest` (e.g. `AuthenticationInterceptor` attaching a credential, `DeflateRequestCompressor` compressing the body). If no adapter, it goes straight to `didCreateURLRequest`.

## 3. Send

`Session.didCreateURLRequest` (`Session.swift:1291`) creates the actual `URLSessionTask` and wires it up:

```swift
request.didCreateURLRequest(urlRequest)
let task = request.task(for: urlRequest, using: session)   // subclass picks task type
mutableState.write { $0.requestTaskMap[request] = task }   // bidirectional map
request.didCreateTask(task)
```

`task(for:using:)` is overridden per request kind — `DataRequest` makes a `dataTask`, `UploadRequest` picks `uploadTask(with:from:/fromFile:/withStreamedRequest:)` (`UploadRequest.swift:110`), `DownloadRequest` a download task. The `RequestTaskMap` keeps a two-way task↔request mapping so delegate callbacks can find their `Request`.

The task is actually started via the `Request` state machine. `resume()` (`Request.swift:767`) transitions to `.resumed` and calls `task.resume()`. This is typically triggered when you add a response handler: `appendResponseSerializer` (`Request.swift:589`) resumes automatically if `shouldAutomaticallyResume`/`startImmediately` is true — that's why a request "sends" once you attach `.response { ... }`.

While in flight, `SessionDelegate` (the `URLSessionDelegate`) receives all the URLSession callbacks and routes them via `stateProvider?.request(for: task)`:
- `didReceive response` → `request.didReceiveResponse(...)` (`SessionDelegate.swift:249`)
- `didReceive data` → `request.didReceive(data:)` accumulates body data
- auth challenges → server-trust / credential evaluation
- redirects → `RedirectHandler`

## 4. Validate & complete

When the task finishes, `urlSession(_:task:didCompleteWithError:)` (`SessionDelegate.swift:222`) calls back into the request:

```swift
stateProvider?.didCompleteTask(task) {
    request?.didCompleteTask(task, with: error.map { $0.asAFError(...) })
}
```

`Request.didCompleteTask` (`Request.swift:515`) is where **response validation runs** — this is the dynamic-dispatch link the graph surfaced:

```swift
mutableState.write { $0.error = $0.error ?? error }
let validators = validators.read(\.self)
validators.forEach { $0() }          // ← run all validation closures
...
retryOrFinish(error: self.error)
```

Each validator was registered earlier by `DataRequest.validate(_:)` (`DataRequest.swift:144`). It captures a closure that only runs if there's no error yet and a response exists, computes a `ValidationResult`, and promotes a `.failure` into `self.error`:

```swift
let result = validation(request, response, data)
if case let .failure(error) = result {
    self.error = error.asAFError(or: .responseValidationFailed(...))
}
```

The built-in validators live in `Source/Features/Validation.swift`: `validate(statusCode:response:)` checks the code is in the acceptable set (default `200...299`, else `.unacceptableStatusCode`), and `validate(contentType:response:isEmpty:)` MIME-matches against the `Accept` header (else `.unacceptableContentType`/`.missingContentType`). Because validators just set `self.error`, an unacceptable status/content-type turns into a failure that response serializers surface.

## 5. Retry or finish

`retryOrFinish` (`Request.swift:543`) closes the loop. If there's an error and a delegate, it consults the retrier:

```swift
delegate.retryResult(for: self, dueTo: error) { retryResult in
    case .doNotRetry:           self.finish()
    case .retry, .retryWithDelay: delegate.retryRequest(self, withDelay: ...)
}
```

On retry, `prepareForRetry` → `reset()` (`Request.swift:665`) clears error/serializer state and the whole build→adapt→send pipeline reruns (`UploadRequest` even re-creates its uploadable). Otherwise `finish()` (`Request.swift:563`) flips `isFinishing` and drives `processNextResponseSerializer`, which runs your serializers on the `serializationQueue` and delivers the final `AFDataResponse` to completion handlers on the target queue.

## The pipeline in one line

`Session.request` → `DataRequest` (lazy) → `performSetupOperations`: `asURLRequest()` + `URLRequest.validate()` → adapter chain (`adapt`) → `didCreateURLRequest` creates `URLSessionTask` → `resume()` → `URLSession` runs it, `SessionDelegate` routes callbacks → `didCompleteTask` runs **response validators** → `retryOrFinish` → serializers deliver the response.

Two distinct "validate" stages worth distinguishing: **request validation** at build time (`URLRequest.validate()`, synchronous, throws) vs. **response validation** at completion time (`DataRequest.validate` closures in `didCompleteTask`, which set `self.error` and feed the retry decision).
