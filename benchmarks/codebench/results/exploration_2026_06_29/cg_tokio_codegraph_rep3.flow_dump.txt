## Overview

Tokio has **two schedulers** picked at runtime-build time:
- **`current_thread`** — single-threaded, runs everything on the `block_on` thread.
- **`multi_thread`** — a work-stealing thread pool (the default `#[tokio::main]`).

Both are built from the same building blocks: an **owned-tasks registry**, per-worker **run queues**, a **global injection queue**, a **LIFO slot**, a cooperative **budget**, and a **driver** (I/O + timers) that parks the thread when there's no work.

A "task" is a spawned future wrapped in a heap allocation (`Cell` → `Harness`/`RawTask`) that the scheduler holds by reference-count. Scheduling means moving a task's handle into a run queue; running it means calling `task.run()`, which polls the inner future once.

## The multi-threaded scheduler

### Worker structure
`create()` (`worker.rs:267`) builds `size` workers, each with its own `Core` holding a `tick` counter, a `lifo_slot`, a bounded `run_queue`, `is_searching`/`is_shutdown` flags, a `park` handle, and a `FastRand`. A shared `Handle` holds the `inject` (global) queue, the `idle` set, the `owned` task registry, and every worker's *stealable* `Remote`. Each worker runs `run(worker)` on a blocking-pool thread (`worker.rs:508`, launched at `Launch::launch`, `:501`).

### The core loop — `Context::run` (`worker.rs:561`)
Each worker loops until shutdown:

1. `core.tick()` — bump the tick counter.
2. `maintenance(core)` (`:780`) — every `event_interval` ticks, do a non-blocking `park_yield` so the I/O driver and timers get serviced even under constant CPU load.
3. **`next_task(&worker)`** (`:1062`) — get the next task to run.
4. If a task is found → **`run_task`** and `continue`.
5. If not → **`steal_work`** from other workers (`:1141`).
6. If stealing also fails → **`park`** the thread (`:810`) until the driver or another worker wakes it.

### Where the next task comes from — `next_task` (`worker.rs:1062`)
The worker checks queues in a rotation biased for fairness:
- **Every `global_queue_interval` ticks**, it checks the **global injection queue first** (`next_remote_task`), then local. This prevents the global queue from starving under a busy local queue. The interval is auto-tuned (`tune_global_queue_interval`) from measured poll times.
- **Otherwise**, it checks **local first** (`next_local_task` → `lifo_slot.take()` then `run_queue.pop()`, `:1132`), and only if empty pulls a **batch** from the injection queue — taking roughly `inject.len() / num_workers + 1` tasks (never all of them, so other workers get some), returning one and pushing the rest onto its local queue.

### The LIFO slot — latency optimization
When a task is scheduled from *within* a running worker task (`schedule_local`, `:1353`), and it's not a yield, the task goes into the single **`lifo_slot`** rather than the queue tail. The next `next_local_task` picks it up immediately. This optimizes message-passing (task A wakes task B → B runs right away, warm cache).

To prevent starvation, `run_task` (`:630`) drains the LIFO slot inline in a loop but caps it at `MAX_LIFO_POLLS_PER_TICK = 3` (`:264`), after which it disables the slot for that tick. It also bails out of the LIFO loop if the cooperative budget is exhausted, pushing the LIFO task back onto the queue.

### Work-stealing — `steal_work` (`worker.rs:1141`)
An idle worker transitions to **searching** (rate-limited: it won't steal if too many workers already are), picks a random victim, and calls `steal_into`, which moves *half* of the victim's run queue into its own. If all victims are empty it falls back to the global queue. This random start + half-steal keeps load balanced without a central bottleneck.

### Parking & wakeups
If there's genuinely no work, `transition_to_parked` (`:1204`) records the worker as idle and it blocks on the I/O/timer driver (`park`, `:810`). When a task is scheduled:
- from a worker thread → `schedule_local` may `notify_parked_local`, waking one idle worker;
- from off-runtime (or when no core is held) → `schedule_task` (`:1327`) pushes to the **global injection queue** and calls `notify_parked_remote`.

The "last searcher parking" re-checks all queues to close the race where work appears just as everyone goes idle.

## The current-thread scheduler

Simpler: `CoreGuard::block_on` (`current_thread/mod.rs:769`) runs one loop on the calling thread:
- Alternates between polling the top-level `block_on` future (when `reset_woken` fires) and draining scheduled tasks.
- For up to `event_interval` iterations: `core.tick()`, `next_task` (`:324`, same global-vs-local rotation, no stealing), then `run_task` (`:371`).
- When the queue empties, it `park`s on the driver (`:380`); if there's deferred work it does a non-blocking `park_yield`.

`schedule` here just pushes onto the single `VecDeque` run queue (or the inject queue if scheduled from another thread).

## Running a task — what `task.run()` does

Both schedulers ultimately call `task.run()`, which routes through the **task harness** (`harness.rs`):
- `Harness::poll` (`:153`) → `poll_inner` → **`poll_future`** (`:521`).
- `poll_future` calls the future's `poll` inside a `catch_unwind` panic guard (a `Drop` guard drops the future if it panics mid-poll). 
- `Poll::Pending` → return, task stays registered, will be re-scheduled when its `Waker` fires. `Poll::Ready(output)` → `store_output`, and the harness transitions to complete, notifying any `JoinHandle` waker.
- The whole poll runs inside **`coop::budget`** (`run_task`), tokio's cooperative scheduling: each task gets a fixed poll budget so resource operations (channels, I/O) return `Pending` once exhausted, forcing the task to yield and preventing one busy task from monopolizing a worker.

## The wake path
A future's `Waker`, when fired, calls the scheduler's `schedule` (via the `Schedule` trait impl on `Handle`). That re-inserts the task's handle into a run queue (LIFO slot, local queue, or inject queue depending on origin) and unparks a worker if needed — closing the loop back to `next_task`.

## Blocking work
`spawn_blocking` and `block_in_place` are separate: blocking tasks go to a dedicated blocking thread pool (`blocking/pool.rs`), whose `Inner::run` (`:503`) is a plain BUSY/IDLE condvar loop — no polling, it just runs each closure to completion. `block_in_place` (`worker.rs:359`) hands the worker's `Core` off to another thread (via `run(worker)`) so the pool keeps making progress while the current thread blocks.

**In one sentence:** each worker loops pulling tasks from its LIFO slot → local queue → global queue → stealing from peers, polls each once through the harness under a cooperative budget, and parks on the I/O/timer driver when idle — with wakers re-injecting tasks and idle workers being unparked to rebalance load.
