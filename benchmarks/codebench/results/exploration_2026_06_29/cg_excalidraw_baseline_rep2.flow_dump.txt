### 2. Multi-Canvas Architecture

Excalidraw uses a **three-layer canvas system** for performance separation:

**Canvas Components** (`packages/excalidraw/components/canvases/`):

| Canvas | File | Purpose | Update Trigger |
|--------|------|---------|-----------------|
| **StaticCanvas** | `StaticCanvas.tsx` | Renders static content (grid, all non-interactive elements) | useEffect + `canvasNonce` prop |
| **InteractiveCanvas** | `InteractiveCanvas.tsx` | Renders interactive UI overlays (selection, handles, snap lines) | AnimationController continuous loop (via `renderInteractiveSceneCallback`) |
| **NewElementCanvas** | `NewElementCanvas.tsx` | Renders the element currently being created (on top layer) | useEffect when `newElement` changes |

**Why this separation?**
- Static canvas is throttled and only re-renders when elements change (`canvasNonce` invalidation)
- Interactive canvas uses continuous animation loop for smooth interactions (handles, feedback)
- New element canvas is layered on top for immediate visual feedback while drawing

### 5. Rendering Scheduling & Trigger Mechanism

## The big picture

Excalidraw keeps elements in a central **`Scene`**, draws them across **three stacked `<canvas>` layers**, and propagates changes through a **mutation → nonce-bump → React re-render** loop. Two layers of caching (RoughJS shapes + per-element canvases) keep redraws cheap.

---

## 1. Where elements live: the `Scene`

`packages/element/src/Scene.ts` is the single source of truth. It holds:
- `elements` — all elements including deleted ones (soft-delete via `isDeleted`)
- `nonDeletedElements` + map variants — cached, pre-filtered for fast lookup
- `sceneNonce` — a random integer regenerated on every update, used purely as a cache-invalidation signal (not element versioning)

Accessors: `getNonDeletedElements()`, `getElementsMapIncludingDeleted()`, `getNonDeletedElementsMap()`, etc.

## 2. The three-canvas architecture

In `packages/excalidraw/components/canvases/`, rendering is split so each layer updates on its own schedule:

| Canvas | Renders | Update trigger |
|--------|---------|----------------|
| **StaticCanvas** | grid + all committed elements | `useEffect`, gated by a `canvasNonce` prop; throttled via `throttleRAF` |
| **InteractiveCanvas** | selection/transform handles, snap lines, binding highlights, remote cursors | continuous `requestAnimationFrame` loop (`AnimationController`) |
| **NewElementCanvas** | the element currently being drawn | `useEffect` when `newElement` changes |

Why split: static content is expensive but rarely changes, so it's throttled and cache-gated; interactive feedback needs to be buttery, so it runs on its own RAF loop without forcing the whole element set to redraw.

The renderer entry points are `renderStaticScene()` (`renderer/staticScene.ts`), `renderInteractiveScene()` (`renderer/interactiveScene.ts`), and `renderNewElementScene()` (`renderer/renderNewElementScene.ts`).

## 3. Drawing a single element + RoughJS

`packages/element/src/renderElement.ts` → `renderElement()` draws each element. RoughJS (`rough`) provides the hand-drawn look: shapes become RoughJS `Drawable`s, drawn via a `RoughCanvas` (`rc.draw(shape)`).

**Two caching tiers** keep this fast:
1. **`ShapeCache`** (`packages/element/src/shape.ts`) — a `WeakMap<element, {shape, theme}>` caching the generated RoughJS shape per element.
2. **Element-canvas cache** (`renderElement.ts`) — a `WeakMap<element, ...withCanvas>` caching a fully pre-rendered `HTMLCanvasElement` per element. `generateElementCanvas()` builds it; it's regenerated only when zoom, theme, angle, bound-text version, image crop, or frame opacity change. On the main canvas, the cached bitmap is just composited via `drawElementFromCanvas()`.

## 4. How an update flows through the system

The mutation path is two-tiered:

- **`mutateElement()`** (`packages/element/src/mutateElement.ts`) — low-level. Applies updates, and *if a value actually changed*, increments `element.version`, regenerates `element.versionNonce`, updates `element.updated`, and clears that element's shape cache. It does **not** trigger a re-render (collaboration-safe and silent).
- **`Scene.mutateElement()`** — wraps the above and, when the version changed and `informMutation: true`, calls `Scene.triggerUpdate()`.

`triggerUpdate()` regenerates `sceneNonce` and fires all registered callbacks. In `App.tsx`, the App subscribes via `this.scene.onUpdate(this.triggerRender)`; `triggerRender` calls `setState({})`, forcing a React re-render. On render, `Renderer.getRenderableElements()` (`scene/Renderer.ts`) computes a new `canvasNonce` from `sceneNonce`, which is passed to the canvas components — their memo comparison redraws only when the nonce changed.

So the chain is:
```
scene.mutateElement(el, updates, {informMutation:true})
  → version++/versionNonce regenerated + shape cache cleared
  → triggerUpdate(): new sceneNonce + callbacks
  → App.triggerRender(): setState({})
  → React re-render → new canvasNonce
  → StaticCanvas redraws (cached shapes/bitmaps reused where unchanged)
```

## 5. Versioning, ordering, and history (the supporting systems)

- **`version` + `versionNonce`**: the change fingerprint. `version` increments on every change; `versionNonce` is a random tiebreaker for deterministic conflict resolution in multiplayer reconciliation.
- **Fractional indices** (`packages/element/src/fractionalIndex.ts`): each element's `index` is a fractional-form string giving z-order. `syncMovedIndices()` / `syncInvalidIndices()` keep array order and index order consistent, enabling conflict-free reordering across clients.
- **Store / delta / history**: `Store` (`packages/element/src/store.ts`) snapshots scene state and emits increments; `StoreDelta`/`ElementsDelta` (`delta.ts`) categorize changes into added/removed/updated; `History` (`packages/excalidraw/history.ts`) records inverse deltas for undo/redo. `CaptureUpdateAction` (`IMMEDIATELY`/`EVENTUALLY`/`NEVER`) controls whether a change lands on the undo stack.
