[assistant] Here's how Django's ORM takes a `QuerySet` from Python code to database rows. The key architectural idea is **separation of building from executing**: a `QuerySet` is a lazy wrapper whose real state lives in a `Query` object (an AST-like representation), and execution only happens when the results are actually needed, via a `SQLCompiler`.

## 1. Building — `QuerySet` mutates a `Query`, lazily

A `QuerySet` (`django/db/models/query.py:326`) holds a `Query` object (`self._query`, exposed via the `query` property at line 347). Almost nothing about a `QuerySet` touches the database. Chained methods like `.filter()` are pure builders:

```
filter()                      query.py:1648
  └─ _filter_or_exclude()     query.py:1664  →  self._chain()  (clones the QuerySet + Query)
       └─ _filter_or_exclude_inplace()  →  self._query.add_q(Q(*args, **kwargs))
```

Two things matter here:
- **Every filter clones.** `_filter_or_exclude` calls `_chain()`, which does `query=self.query.chain()` (`query.py:2225`). This is why `QuerySet`s are immutable/chainable — `qs.filter(a).filter(b)` produces new objects and never mutates the original.
- **Deferred filters.** The `query` property (line 347) can carry a `_deferred_filter` that only gets applied the first time `.query` is actually accessed — a micro-optimization so filter-building work is skipped if the queryset is never evaluated.

The actual query construction happens inside the `Query` object in `django/db/models/sql/query.py`:

```
add_q()          query.py:1663   — entry point, resolves the Q tree
  └─ _add_q()    query.py:1692   — recurses over the Q node's children
       └─ build_filter()  query.py:1488  — the workhorse
```

`build_filter` (line 1488) is where a lookup string like `"author__name__icontains"` becomes structured SQL pieces:
- `solve_lookup_type` / `names_to_path` split the string on `__` and walk the model's `_meta` to resolve fields and relations.
- `setup_joins` (line 1577) creates the JOINs needed to traverse relations, registering table aliases in the query's `alias_map`.
- `build_lookup` turns the final `icontains`/`exact`/etc. into a `Lookup` object, wrapped in a `WhereNode` (line 1615) and added to the query's `where` tree.
- If an exclude crosses a to-many relation, `split_exclude` (line 2113) builds a correlated `NOT EXISTS` subquery.

At the end of all this, the `Query` holds the full logical plan: selected columns, joins/aliases, a `WhereNode` tree, ordering, grouping, etc. **Still no SQL and no database contact.**

## 2. Triggering execution — laziness ends at iteration

A `QuerySet` executes only when its results are consumed — iterating it, `list()`, `len()`, `bool()`, `get()`, indexing, etc. These all funnel through:

```python
def _fetch_all(self):                    # query.py:2238
    if self._result_cache is None:
        self._result_cache = list(self._iterable_class(self))
    ...
```

`_result_cache` is the memoization: the query runs **once**, and later iterations reuse the cached list. `_iterable_class` selects how rows are turned into Python — `ModelIterable` for normal model instances, `ValuesIterable`/`ValuesListIterable`/`FlatValuesListIterable` for `.values()`/`.values_list()`.

`.iterator()` (line 561) is the exception that bypasses `_result_cache` and streams via server-side cursors (`_iterator`, line 546).

## 3. Compiling — `Query` → `SQLCompiler` → SQL string

When the iterable runs, it asks the `Query` for a compiler:

```python
# e.g. FlatValuesListIterable.__iter__  (query.py:295)
compiler = queryset.query.get_compiler(queryset.db)
for row in compiler.results_iter(...):
    ...
```

`get_compiler` (`query.py:370`) picks a backend-specific compiler class through the connection — `connection.ops.compiler(self.compiler)(...)`. This is how PostgreSQL/MySQL/Oracle each supply their own `SQLCompiler` subclass while sharing the base logic.

The compiler's `as_sql()` (`compiler.py:757`) is the translation step — it walks the `Query`'s structure and emits the actual SQL text plus a parameters list:
- `pre_sql_setup` / `setup_query` resolve the SELECT columns, GROUP BY, ORDER BY.
- `compile(node)` (line 574) recursively calls each expression/lookup's own `as_sql()` — this is the `Transform`/`Lookup` runtime dispatch the graph flagged (50+ implementations). Each node knows how to render itself, so `as_sql` composes them into `SELECT ... FROM ... JOIN ... WHERE ...`.
- Parameters are always kept **separate** from the SQL string — Django uses parameterized queries (`%s` placeholders), which is its baseline SQL-injection defense.

`Query.__str__`/`sql_with_params` (`query.py:357`) route through the same `get_compiler().as_sql()` — that's what you see when you print `qs.query`.

## 4. Executing — running SQL and mapping rows back

```
results_iter()   compiler.py:1565
  └─ execute_sql(MULTI, ...)   compiler.py:1595
       ├─ sql, params = self.as_sql()
       ├─ cursor = self.connection.cursor()
       └─ cursor.execute(sql, params)
```

`execute_sql` (`compiler.py:1595`) is the single point that touches the DB-API cursor. Its behavior depends on `result_type`:
- `MULTI` — `fetchmany()` in chunks (wrapped in `cursor_iter`), for row sets.
- `SINGLE` — `fetchone()`, e.g. for `.get()` / `exists()`.
- `ROW_COUNT` — used by UPDATE/DELETE (`SQLUpdateCompiler.execute_sql` at `compiler.py:2114` returns affected rows).
- `CURSOR` / `NO_RESULTS`.

If the query is provably empty (e.g. `filter(pk__in=[])`), `as_sql` raises `EmptyResultSet` and `execute_sql` short-circuits to an empty iterator — **no SQL is ever sent**.

Then raw DB rows are turned back into Python:
- `results_iter` (line 1565) applies **converters** (`get_converters`/`apply_converters`, lines 1514/1536) — backend + field-level transforms that turn raw DB values into proper Python types (dates, decimals, UUIDs, etc.).
- For model instances, `ModelIterable` calls `model_cls.from_db(...)` per row (see `RelatedPopulator.populate`, `query.py:3014`), and `RelatedPopulator` fills in `select_related` objects from the same row so no extra queries fire.

## The mental model

```
QuerySet  ──(.filter/.exclude/… build)──▶  Query          "what to fetch" (logical plan, DB-agnostic)
   │                                          │
   │  iterate / list() / get()                │ get_compiler(using)
   ▼                                          ▼
_fetch_all ──▶ _iterable_class ──▶ SQLCompiler.as_sql()   "how to say it in this backend's SQL"
                                          │
                                          ▼
                            execute_sql → cursor.execute(sql, params)
                                          │
                                          ▼
                            rows → converters → from_db() → model instances
```

- **`QuerySet`** = user-facing, lazy, immutable/chainable, result-caching.
- **`Query`** = backend-independent structural representation (joins, where-tree, select).
- **`SQLCompiler`** = backend-specific translator + executor.
- Laziness boundary is `_fetch_all` → `_result_cache`; the DB is touched exactly once at `execute_sql`, and always with separated parameters.
