# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops

a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a

251	    /// another opportunity.

262	        ///

264	        /// queue as we are also inserting the `task` argument.
265	        const NUM_TASKS_TAKEN: UnsignedShort = (LOCAL_QUEUE_CAPACITY / 2) as UnsignedShort;

303	        // means that if a task is in the second half, then we know for sure that this task is not
304	        // a task we just got from the injection queue. This ensures that when we take a task out

306	        // least not until after we have polled it at least once).
307	        //
308	        // Note that if a concurrent worker tries to steal from us between these two operations and

310	        // notify it about these tasks becoming available for stealing again. Ordinarily this would
311	        // be a problem, but it isn't in this case because the worker will be notified about the
312	        // tasks we are adding to the injection queue instead, which ensures that the stealer wakes

355	        stats.incr_overflow_count();

[assistant] I now have the full picture across the multi-thread scheduler, the task harness, and the state machine. Here's how tokio schedules and runs async tasks.

## Overview

Tokio's default runtime is a **multi-threaded, work-stealing scheduler**. A fixed pool of worker threads each run a loop that pulls a task off a queue, polls its future once, and either re-queues it (if it yielded `Pending` and got re-notified) or completes it. The whole machine is coordinated by an atomic **task state machine** so a task is only ever polled by one thread at a time, and wakeups from anywhere (I/O, timers, channels) turn into re-scheduling.

The key files:
- `scheduler/multi_thread/worker.rs` — the worker run loop, queues, work-stealing, parking
- `scheduler/multi_thread/handle.rs` + `queue.rs` — scheduling / enqueueing tasks
- `task/harness.rs` + `task/state.rs` — polling a future and the atomic lifecycle transitions

## 1. Spawning a task

`spawn` binds the future into a heap-allocated task (`RawTask`, `raw.rs:211`) and produces a `Notified` handle plus a `JoinHandle`. `Handle::bind_new_task` (`handle.rs:81`) then hands the freshly-notified task straight to the scheduler:

```rust
let (handle, notified) = me.shared.owned.bind(future, me.clone(), id, spawned_at);
me.schedule_option_task_without_yield(notified);   // → schedule_task
```

## 2. Scheduling — the three queue levels

Where a task lands depends on **who is scheduling it** (`Handle::schedule_task`, `worker.rs:1327`):

```rust
with_current(|maybe_cx| {
    if let Some(cx) = maybe_cx {
        if self.ptr_eq(&cx.worker.handle) {              // scheduled from a worker of THIS runtime
            if let Some(core) = cx.core.borrow_mut().as_mut() {
                self.schedule_local(core, task, is_yield);
                return;
            }
        }
    }
    self.push_remote_task(task);      // scheduled from outside → global inject queue
    self.notify_parked_remote();
});
```

Three destinations, in priority order (`schedule_local`, `worker.rs:1353`):

1. **LIFO slot** — a single-task slot per worker. A newly-notified task (not a yield) is put here so the *most recently woken* task runs next. This keeps hot, communicating tasks (e.g. A wakes B wakes A) cache-warm. Any task already in the slot is bumped to the local queue.
2. **Local run queue** — a fixed-capacity (256) lock-free ring buffer, one per worker (`queue.rs`). Yields and LIFO-evictions go here via `push_back_or_overflow` (`queue.rs:188`). When it's full, **half its contents overflow into the global queue** so other workers can grab them.
3. **Global "inject" queue** — a shared MPMC queue used when a task is scheduled from a non-worker thread (or from another runtime). This is the entry point for work coming from I/O reactors, `spawn` from `main`, etc.

To prevent the LIFO fast-path from starving other tasks, it's capped: after `MAX_LIFO_POLLS_PER_TICK` consecutive LIFO runs the slot is disabled for that tick (`worker.rs:743`).

## 3. The worker run loop

Each worker thread runs `Context::run` (`worker.rs:561`):

```rust
while !core.is_shutdown {
    core.tick();
    core = self.maintenance(core);

    if let Some(task) = core.next_task(&self.worker) {   // local queue (+ periodic global)
        core = self.run_task(task, core)?;
        continue;
    }
    if let Some(task) = core.steal_work(&self.worker) {  // steal from peers / global
        core = self.run_task(task, core)?;
    } else {
        core = self.park(core);                          // nothing to do → sleep
    }
}
```

**`next_task`** normally pops from the local queue, but every `global_queue_interval` ticks it checks the global queue *first* so injected work and fairness are preserved (the current-thread scheduler does the same, `current_thread/mod.rs:324`).

**Work stealing** (`steal_work`, `worker.rs:1141`): when a worker's own queues are empty, it transitions to a "searching" state, picks a **random** peer worker as a starting point, and tries to steal *half* of that peer's local queue into its own. If all peers are empty it falls back to the global inject queue. Starting at a random index avoids all idle workers hammering worker 0.

**Parking** (`park`, `worker.rs:810`): if there's genuinely no work, the worker parks on the driver (blocking on epoll/timers). It's woken by `notify_parked_local`/`notify_parked_remote` (`worker.rs:1441`), which unpark an idle worker when new work is scheduled. The searching/parked transitions are carefully managed so that at least one worker stays awake to notice incoming work (`transition_to_parked`, `transition_from_searching`).

## 4. Running a task = polling the future once

`run_task` (`worker.rs:630`) sets the core into the thread-local context and calls `task.run()` inside a **cooperative budget** (`coop::budget`). After the poll it drains the LIFO slot in a loop as long as budget remains (`worker.rs:693`) — this is what makes ping-pong tasks fast.

`coop::budget` gives each task a quota of ~128 resource operations. Leaf futures (channels, I/O) decrement it and return `Pending` when it hits zero, forcing the task to yield even if data is ready. That's tokio's **cooperative scheduling** — it prevents a single hot task from starving the runtime.

`task.run()` dispatches through the task's vtable (`raw.rs:271`) into `Harness::poll` (`harness.rs:153`), which:

1. **Transitions the state to Running** (`state.rs:117`). This atomic op locks the RUNNING bit and clears the NOTIFIED bit, so any wakeup *during* the poll is detected. If the task was already cancelled/complete, it bails out here.
2. **Polls the future** via `poll_future` (`harness.rs:521`), wrapping it in `catch_unwind` (so a panicking task becomes a `JoinError` rather than crashing the worker) and a drop-guard.
3. Based on the result:
   - `Poll::Pending` → **transition back to Idle** (`transition_to_idle`, `state.rs:151`). If a wakeup arrived during the poll, the task is immediately re-notified and re-scheduled; otherwise it goes dormant, holding a waker.
   - `Poll::Ready(output)` → store the output, transition to Complete, and wake the `JoinHandle`.

## 5. How wakeups turn back into scheduling

When a future is `Pending`, it has stashed a `Waker` (backed by the task itself). Later, some event — an I/O readiness notification from the reactor, a timer firing, a channel send — calls `wake()`/`wake_by_ref()`. Those go through `transition_to_notified_by_val`/`_by_ref` (`state.rs:215`/`253`):

- If the task is **currently running**, it just sets the NOTIFIED bit; the running worker will re-schedule it when it finishes the poll (step 3 above). No duplicate scheduling.
- If the task is **idle**, it flips to NOTIFIED, creates a fresh `Notified` handle, and calls `schedule` again — dropping the task right back into the queue system from §2.

This atomic state machine is the core invariant: a task is polled by exactly one thread at a time, wakeups are never lost, and the task is never double-queued.

## Other runtime flavors

- **Current-thread runtime** (`current_thread/mod.rs`): same task machinery, but a single thread with one local `VecDeque` + inject queue, no stealing. `schedule` (`current_thread/mod.rs:671`) pushes locally if on the runtime thread, else to the inject queue and unparks the driver.
- **LocalSet / LocalRuntime** (`task/local.rs`): runs `!Send` tasks on one thread; `tick` (`local.rs:756`) polls up to `MAX_TASKS_PER_TICK` before yielding back to the outer runtime for fairness.
- **Blocking pool** (`blocking/pool.rs`): `spawn_blocking` work runs on a separate, dynamically-sized thread pool (`Inner::run`, `pool.rs:503`) — a plain mutex+condvar queue, not the async scheduler — so blocking calls never stall async workers.

**In one sentence:** worker threads loop pulling tasks from a per-worker LIFO slot → local ring buffer → stolen-from-peers → global inject queue, poll each future exactly once under a cooperative budget through an atomic state machine, and re-queue tasks whenever a waker fires — parking on the I/O driver when there's nothing left to do.
