## 1. Building — every operation clones and mutates a `Query`

A `QuerySet` is just a thin wrapper holding a `Query` object (`self._query = query or sql.Query(self.model)`, `query.py:333`). It's **lazy** — nothing touches the database while you chain.

When you call `.filter(**kwargs)` (`query.py:1648`):

```python
def filter(self, *args, **kwargs):
    return self._filter_or_exclude(False, args, kwargs)
```

This flows through `_clone()` (`query.py:2218`) → `self.query.chain()` (`sql/query.py:433`) → `clone()`. Every mutating op returns a **new** QuerySet whose `Query` is a lightweight copy — that's why filters chain without side-effects. The clone is deliberately shallow (`Query.clone`, `sql/query.py:388`): it copies dicts/refs, not the whole tree.

The actual filter is translated by `add_q` (`sql/query.py:1663`), which wraps the kwargs in a `Q` object, resolves the lookups (`author__name` → a join + a `WHERE` condition), performs **join promotion**, and adds the resulting clause to `self.where` (a `WhereNode` tree). At this point the query lives entirely as an in-memory tree: `alias_map` (joins), `where`, `annotations`, `select`, `group_by`, etc. — **no SQL yet, no DB touched.**

## 2. The lazy trigger — evaluation forces execution

Execution only happens when you actually consume the QuerySet — iterating it, `list()`, `len()`, indexing, `bool()`, etc. All of these funnel through `_fetch_all` (`query.py:2238`):

```python
def _fetch_all(self):
    if self._result_cache is None:
        self._result_cache = list(self._iterable_class(self))   # ← runs the query
    ...
```

The results are cached in `_result_cache`, so a QuerySet hits the DB **once** and reuses the rows. The `_iterable_class` picks how rows are shaped: `ModelIterable` (default → model instances), `ValuesIterable` (dicts), `ValuesListIterable` (tuples), etc. (`query.py:340`, and swapped to `ValuesIterable` by the `query` setter at line 357 when `.values()` is used).

## 3. Compiling — `Query` → SQL

`ModelIterable.__iter__` (`query.py:92`) kicks off the real work:

```python
compiler = queryset.query.get_compiler(using=db)     # sql/query.py:370
results = compiler.execute_sql(chunked_fetch=..., chunk_size=...)
```

`get_compiler` asks the **backend** for the right compiler class — this is the pluggable seam that lets each database emit its own dialect:

```python
def get_compiler(self, using=None, connection=None, ...):
    connection = connections[using]
    return connection.ops.compiler(self.compiler)(self, connection, using, elide_empty)
```

The heavy lifting is `SQLCompiler.as_sql()` (`compiler.py:757`), which walks the `Query` tree and assembles each clause — `get_select()`, `get_from_clause()`, the compiled `where`, `get_group_by()`, `get_order_by()`, limits — returning a `(sql_string, params)` tuple. `compile(node)` (`compiler.py:574`) recursively delegates to each expression's own `as_sql`, so every field, lookup, and function contributes its fragment plus parameters. Params are always kept **separate** from the SQL string → parameterized queries, no injection.

## 4. Executing — cursor and row fetching

`SQLCompiler.execute_sql` (`compiler.py:1595`) is where the database is finally hit:

```python
sql, params = self.as_sql()
cursor = self.connection.cursor()   # or chunked_cursor() for server-side
cursor.execute(sql, params)
```

`result_type` controls the fetch strategy — `MULTI` (fetchmany, wrapped in `cursor_iter` for chunked reads), `SINGLE` (fetchone, used by `has_results`/`.get()`), `ROW_COUNT` (used by `UPDATE`/`DELETE`), or `CURSOR`. If the query was provably empty (`EmptyResultSet`), it short-circuits without any DB call.

## 5. Rows → objects

Back in `ModelIterable.__iter__`, raw rows are streamed through `results_iter` (`compiler.py:1565`), which applies **DB converters** (`apply_converters`, e.g. turning DB values into `Decimal`/`datetime`/UUID) and handles composite fields. Then for each row:

```python
obj = model_cls.from_db(db, init_list, row[model_fields_start:model_fields_end], fetch_mode=...)
```

`Model.from_db` (`base.py:618`) instantiates the model. Then `related_populators` fill in `select_related` objects from the same row (no extra queries), and `annotation_col_map` copies annotation columns onto the instance as attributes. Each `obj` is `yield`ed.

## The full path, end to end

```
Book.objects.filter(author__name="X").order_by("title")
    │  .filter / .order_by  → _clone() → Query.chain()/add_q()   [build tree, no DB]
    ▼
list(qs)  →  _fetch_all()  →  ModelIterable.__iter__()
    │
    ├─ query.get_compiler(using=db)          → backend picks SQLCompiler subclass
    ├─ compiler.execute_sql()
    │     ├─ as_sql()  → walk Query tree, compile() each node → (sql, params)
    │     └─ cursor.execute(sql, params)      [DB hit]
    ├─ results_iter()  → apply_converters()   [normalize values]
    └─ model_cls.from_db(row)  per row        → Model instances
                                              → cached in _result_cache
```

**Key design points:** the `Query` IR is database-agnostic — the backend-specific SQL is injected only at `get_compiler` time; construction is fully lazy and copy-on-write via cloning; execution is triggered once and cached; and params stay separate from SQL throughout. Write paths (`UPDATE`/`INSERT`/`DELETE`) use the same shape with specialized compilers — `SQLUpdateCompiler.execute_sql` (`compiler.py:2114`) and `SQLInsertCompiler.execute_sql` (`compiler.py:1925`) — that return row counts or inserted PKs instead of building model instances.
