The core code is clear. Here's how OkHttp processes a request through its interceptor chain.

## 1. The chain is assembled as a flat, ordered list

When a call executes, `RealCall.getResponseWithInterceptorChain()` builds one `List<Interceptor>` in a fixed order (`RealCall.kt:208-219`):

```kotlin
val interceptors = mutableListOf<Interceptor>()
interceptors += client.interceptors        // your application interceptors
interceptors += RetryAndFollowUpInterceptor()
interceptors += BridgeInterceptor()
interceptors += CacheInterceptor()
interceptors += ConnectInterceptor
if (!forWebSocket) {
  interceptors += client.networkInterceptors // your network interceptors
}
interceptors += CallServerInterceptor
```

So the order is:

| Position | Interceptor | Role |
|---|---|---|
| 1 | **Application interceptors** (`client.interceptors`) | Your code. Run once per call, see the original request, don't see retries/redirects. |
| 2 | `RetryAndFollowUpInterceptor` | Recovers from failures and follows redirects/auth challenges — the retry loop. |
| 3 | `BridgeInterceptor` | App ↔ network bridge: adds headers (`Host`, `Content-Length`, `Cookie`, gzip), transparently decompresses gzip responses. |
| 4 | `CacheInterceptor` | Serves from / writes to the HTTP cache; may short-circuit the network entirely. |
| 5 | `ConnectInterceptor` | Opens a connection / creates the `Exchange`. |
| 6 | **Network interceptors** (`client.networkInterceptors`) | Your code, but now with a live connection (skipped for WebSockets). |
| 7 | `CallServerInterceptor` | Writes the request to and reads the response from the socket. The "terminal" interceptor. |

## 2. Progression happens via recursive `proceed()`

A `RealInterceptorChain` holds the list plus an `index` pointing at the *next* interceptor to run. The whole flow is driven by `proceed()` (`RealInterceptorChain.kt:312-343`):

```kotlin
override fun proceed(request: Request): Response {
  check(index < interceptors.size)
  calls++
  ...
  // Call the next interceptor in the chain.
  val next = copy(index = index + 1, request = request)
  val interceptor = interceptors[index]
  val response = interceptor.intercept(next) ?: throw NullPointerException(...)
  ...
  return response
}
```

The mechanism is a recursion, not a loop:

1. `RealCall` creates the chain at `index = 0` and calls `chain.proceed(originalRequest)` (`RealCall.kt:232`).
2. `proceed()` makes a **copy of the chain at `index + 1`** and hands it to `interceptors[index].intercept(next)`.
3. Each interceptor does its "before" work, calls `next.proceed(request)` to descend deeper, then does its "after" work on the returned `Response`.
4. This continues down to `CallServerInterceptor`, which actually talks to the server and returns a `Response` *without* calling `proceed()` — that's the bottom of the recursion.
5. The `Response` then unwinds back up the stack, each interceptor optionally transforming it on the way out.

So each interceptor wraps the rest of the chain: code *before* `proceed()` runs on the way down (request side), code *after* runs on the way up (response side).

## 3. Application vs. network interceptors — the key distinction

The difference is purely positional, but it matters:

- **Application interceptors** sit *above* `RetryAndFollowUpInterceptor`, so they're invoked **exactly once** per `Call` and observe the final response — they never see intermediate redirects or retries, and they see the request before OkHttp adds its automatic headers.
- **Network interceptors** sit *below* `ConnectInterceptor`, so a live `Exchange`/`Connection` exists. They can be called **multiple times** (once per redirect/retry) and observe the on-the-wire request/response including OkHttp's added headers.

The chain enforces this contract. A network-layer chain carries a non-null `exchange`, and `proceed()` adds extra checks for it (`RealInterceptorChain.kt:317-340`):

```kotlin
if (exchange != null) {
  check(exchange.finder.routePlanner.sameHostAndPort(request.url)) {
    "network interceptor ${interceptors[index - 1]} must retain the same host and port"
  }
  check(calls == 1) {
    "network interceptor ${interceptors[index - 1]} must call proceed() exactly once"
  }
}
```

This is why a network interceptor must call `proceed()` exactly once and can't rewrite the request to a different host — whereas an application interceptor is free to short-circuit (call `proceed()` zero times) or call it multiple times. The `with*Timeout`/`withDns`/etc. configuration methods likewise `check(exchange == null)`, so they can only be adjusted from an application interceptor.

The doc comment on the class (`RealInterceptorChain.kt:46-52`) sums it up: the chain "carries the entire interceptor chain: all application interceptors, the OkHttp core, all network interceptors, and finally the network caller."
