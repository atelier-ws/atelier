## 1. Build — turning your call into a `URLRequest`

The global entry point is `AF` (`Source/Alamofire.swift`), which is just `Session.default`. Calling `AF.request(...)` lands in one of `Session`'s overloads (`Source/Core/Session.swift`):

`Session` wraps your inputs in an internal `RequestConvertible` that conforms to `URLRequestConvertible`. The actual `URLRequest` isn't built yet — it's built **lazily**, later, on a background queue, when `asURLRequest()` is called:

```swift
    try requestModifier?(&request)              // your last-chance mutation hook
```

```
                      └─ task.resume()   ← network actually starts
```

Two hooks live in this send path via `RequestInterceptor` (`Source/Features/RequestInterceptor.swift`), which is just `RequestAdapter` + `RequestRetrier`:
- **adapt** runs *before* the task is created, so it can mutate the `URLRequest` (e.g. attach an auth token).
- **retry** runs *after* failure (see below). Session-level and request-level interceptors compose via `Interceptor(adapters: [session, request])`, session first.

`didCreateTask` only calls `task.resume()` if the request's state is `.resumed` — this is what makes suspend/cancel-before-start work correctly.

Once running, `URLSession` callbacks are routed by `SessionDelegate` (`Source/Core/SessionDelegate.swift`) back to the `Request`: `didReceive data:` → `DataRequest.didReceive(data:)` accumulates `data`; `didCompleteWithError:` → `Request.didCompleteTask(_:with:)`.

## 3. Validate — and how it fits the pipeline

Validation is **opt-in** via `.validate()` on `DataRequest` (`Source/Features/Validation.swift`). The closure type is:

```swift
typealias Validation = @Sendable (URLRequest?, HTTPURLResponse, Data?) -> ValidationResult
// ValidationResult = Result<Void, any Error & Sendable>
```

`.validate()` with no args is the common default — it checks status `200..<300` plus the `Accept` header content types:

```swift
public func validate() -> Self {
    return validate(statusCode: acceptableStatusCodes)        // 200..<300
        .validate(contentType: acceptableContentTypes())      // from Accept header, or */*
}
```

Validators don't run immediately — they're **stored** and run later. Each `validate(...)` appends a closure to a protected array (`Request.swift`):

```swift
let validators = Protected<[@Sendable () -> Void]>([])
```

The closure captures request/response/data, runs your validation, and on `.failure` sets `self.error` to an `AFError.responseValidationFailed(...)`.

**Timing is the key detail:** all validators fire synchronously in `didCompleteTask`, *after* the task completes but *before* response serializers run:

```swift
func didCompleteTask(_ task: URLSessionTask, with error: AFError?) {
    mutableState.write { $0.error = $0.error ?? error }
    validators.read(\.self).forEach { $0() }   // ← validation runs here
    eventMonitor?.request(self, didCompleteTask: task, with: error)
    retryOrFinish(error: self.error)           // ← may trigger retry, then serializers
}
```

So a validation failure becomes the request's `error` *before* `retryOrFinish`, which means:
1. It can trigger the `RequestRetrier` (e.g. retry a 401 after refreshing a token).
2. If not retried, it propagates into the `AFDataResponse.error` your `.responseDecodable { ... }` handler receives.

Failures are typed via `AFError.responseValidationFailed(reason:)` (`Source/Core/AFError.swift`) with reasons like `.unacceptableStatusCode(code:)`, `.unacceptableContentType(...)`, `.missingContentType(...)`, and `.customValidationFailed(error:)`. Convenience accessors include `isResponseValidationError`, `responseCode`, and `acceptableContentTypes`.

## End-to-end picture

```
AF.request(url, parameters:)         build deferred, DataRequest(.initialized)
   → perform → performSetupOperations
        asURLRequest()  [encode params]           ← BUILD
        RequestInterceptor.adapt()                ← mutate request
   → dataTask(with:) → task.resume()              ← SEND
   → SessionDelegate: didReceive data / didComplete
        validators.forEach { $0() }               ← VALIDATE (sets AFError)
        retryOrFinish → RequestRetrier? → finish
   → serializers on serializationQueue → completion on .main
```

The unifying ideas: building is lazy and deferred to a background queue; sending is funneled through serial queues with `adapt` as a pre-flight hook; validation is a stored-closure pass that runs between task completion and serialization, feeding both the retry decision and the final response error.
