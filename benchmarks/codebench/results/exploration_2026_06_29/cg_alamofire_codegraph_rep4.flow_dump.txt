## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

## Overview

The flow spans four core types: **`Session`** (orchestrator), **`Request`** / **`DataRequest`** (the request object + state machine), **`URLRequestConvertible`** (turns your inputs into a `URLRequest`), and the **`Validation`** feature. Everything is queue-hopping between a `rootQueue`, a `requestQueue`, and a `serializationQueue`, with mutable state guarded by a `Protected` lock.

## 1. Build

`AF.request(...)` lands on `Session.request(...)` (`Source/Core/Session.swift:318`). It wraps your inputs (URL, method, parameters, encoding, headers, `requestModifier`) into a lightweight `RequestConvertible` struct, then hands it to the internal `request(_:interceptor:...)` overload (`Session.swift:393`), which constructs the `DataRequest`:

```swift
let request = DataRequest(convertible: convertible,
                          underlyingQueue: rootQueue,
                          serializationQueue: serializationQueue, ...)
performEagerlyIfNecessary(request)
return request
```

Note the request is returned **before** any network work happens — `DataRequest` is a handle whose actual `URLRequest` is produced lazily. The recipe for producing it lives in `RequestConvertible.asURLRequest()` (`Session.swift:294`):

```swift
var request = try URLRequest(url: url, method: method, headers: headers)
try requestModifier?(&request)              // your inout mutation hook
return try encoding.encode(request, with: parameters)   // URLEncoding / JSONEncoding
```

`asURLRequest()` is a protocol requirement with ~8 implementations (`RequestConvertible`, `RequestEncodableConvertible` for `Encodable` params, `ParameterlessRequestConvertible`, plain `URLRequest`, etc.) — the build strategy is chosen by which API you called.

## 2. Send

When the request needs to run (either eagerly, or when the first response handler triggers `resume()` at `Request.swift:768`), `Request` calls back to its delegate — the `Session` — via `readyToPerform` (`Session.swift:1341`) → **`perform`** (`Session.swift:1166`). `perform` inserts into `activeRequests` and dispatches to the right `performXxxRequest` by type (`UploadRequest` before `DataRequest` due to subtyping). All of these funnel into **`performSetupOperations`** (`Session.swift:1246`), the heart of sending:

1. **Materialize** the `URLRequest`: `initialRequest = try convertible.asURLRequest()`, then `try initialRequest.validate()` (a *URLRequest* sanity check — e.g. GET with a body fails). Failure → `didFailToCreateURLRequest`.
2. Notify `request.didCreateInitialURLRequest`.
3. **Adapt** via interceptors. `adapter(for:)` (`Session.swift:1315`) merges the session-level and request-level interceptors into a single `Interceptor`. If none, skip to step 5. Otherwise `adapter.adapt(initialRequest, using: adapterState)` runs asynchronously — this is dynamic dispatch across ~16 `RequestInterceptor` implementations (`Adapter`, `Interceptor`, `AuthenticationInterceptor`, `DeflateRequestCompressor`, …).
4. On adapter success, the adapted request is **validated again** (`adaptedRequest.validate()`), then `didAdaptInitialRequest`. Failure → `didFailToAdaptURLRequest` → retry-or-finish.
5. Both paths converge on `didCreateURLRequest(_:for:)` (`Session.swift:1291`), which creates the actual OS task:

```swift
request.didCreateURLRequest(urlRequest)
let task = request.task(for: urlRequest, using: session)   // subclass-specific
mutableState.write { $0.requestTaskMap[request] = task }
request.didCreateTask(task)
```

`task(for:using:)` is overridden per subclass — `DataRequest` → `session.dataTask`, `DownloadRequest` → `session.downloadTask` (`DownloadRequest.swift:205`), etc. The `RequestTaskMap` bidirectionally links `Request ↔ URLSessionTask` so `SessionDelegate` callbacks can route back. `resume()` (`Request.swift:768`) transitions state to `.resumed` and calls `task.resume()` to actually put bytes on the wire.

## 3. Validate

Validation is **deferred, not immediate**. Calling `.validate()` (e.g. `Validation.swift:160`) doesn't check anything — it *appends a closure* to the request's `validators` array:

```swift
public func validate<S: Sequence>(statusCode: S) -> Self ... {
    validate { [unowned self] _, response, _ in
        self.validate(statusCode: acceptableStatusCodes, response: response)
    }
}
```

The stored validator (see `DownloadRequest.swift:313` / `DataStreamRequest.swift:203` for the concrete builders) captures the `request`/`response`/data, runs the check, and on `.failure` sets `self.error = AFError.responseValidationFailed(...)`. The underlying checks live in `Validation.swift`: `validate(statusCode:response:)` (`:81`, checks the acceptable set), and `validate(contentType:response:)` (`:105`, MIME-type matching with wildcard support).

These validators fire when the task completes. `SessionDelegate` calls `Request.didCompleteTask` (`Request.swift:515`):

```swift
mutableState.write { $0.error = $0.error ?? error }
let validators = validators.read(\.self)
validators.forEach { $0() }        // ← validation runs here, may set self.error
eventMonitor?.request(self, didCompleteTask: task, with: error)
retryOrFinish(error: self.error)
```

So the order is: network error (if any) is recorded first, then each validator runs and may overwrite `error` with a validation failure, then `retryOrFinish` (`Request.swift:543`) decides — if there's an error and a delegate, it asks the retriers via `retryResult`; on `.retry` it loops back through `retryRequest` → `perform(..., forRetry: true)`, otherwise `finish()` starts the response serializers.

## The whole loop, condensed

```
AF.request → Session.request → new DataRequest (returned immediately)
                                      │ resume()
        readyToPerform → perform → performSetupOperations
                                      │ convertible.asURLRequest() + URLRequest.validate()
                                      │ adapter.adapt()  (+ re-validate)
                                      │ didCreateURLRequest → request.task(for:) → task.resume()
        [network I/O via SessionDelegate]
        didCompleteTask → run validators (set error) → retryOrFinish → finish → response serializers
```

Two distinct validation stages worth remembering: **`URLRequest.validate()`** guards the *outgoing* request during setup, while the **`validators` closures** guard the *incoming* response at completion time.
