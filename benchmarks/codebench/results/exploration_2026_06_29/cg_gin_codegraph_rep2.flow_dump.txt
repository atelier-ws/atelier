# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few

[assistant] Gin routes a request through its middleware chain in two phases: **building** the chain at registration time, and **executing** it per request. Here's the full picture.

## 1. Building the chain (registration time)

Middleware and route handlers are stored together in a single flat slice — `HandlersChain` (a `[]HandlerFunc`).

- **`RouterGroup.Use()`** (`routergroup.go:65`) just appends middleware to the group's `Handlers` slice.
- When you register a route (e.g. `GET`), the group's `handle()` (`routergroup.go:86`) calls `combineHandlers()` to concatenate the group's middleware with the route's final handler(s), then stores that combined chain in the route tree via `engine.addRoute()`.

So by request time, each route in the tree already holds one merged slice: `[middleware..., finalHandler]`.

## 2. Executing the chain (request time)

**`Engine.ServeHTTP`** (`gin.go:662`) is the `http.Handler` entry point:
1. Grabs a pooled `*Context`, resets it, attaches the request.
2. Calls **`handleHTTPRequest`** (`gin.go:690`).

**`handleHTTPRequest`** does the routing:
1. Picks the tree for the HTTP method (`engine.trees`).
2. Looks up the path with `root.getValue(rPath, ...)` (radix-tree match).
3. On a hit, copies the matched chain onto the context: `c.handlers = value.handlers`, sets `c.fullPath`, then calls **`c.Next()`** — this kicks off execution.
4. On a miss, it points `c.handlers` at `engine.allNoRoute` (or `allNoMethod` for 405) and runs those instead.

## 3. The chain driver: `Context.Next()`

This is the heart of it (`context.go:188`):

```go
func (c *Context) Next() {
	c.index++
	for c.index < safeInt8(len(c.handlers)) {
		if c.handlers[c.index] != nil {
			c.handlers[c.index](c)
		}
		c.index++
	}
}
```

`c.index` is a cursor into the handler slice (starts at -1). Each call advances the index and invokes handlers in order. The key design: **a middleware that itself calls `c.Next()` re-enters this loop**, running the *rest* of the chain nested inside its own call. Control returns to the middleware after downstream handlers finish — this is what enables "before/after" logic (e.g. Logger timing a request, Recovery deferring a panic handler around `Next()`).

A middleware that does *not* call `c.Next()` still works: when it returns, the outer loop advances to the next handler anyway. So `Next()` is only needed when you want to wrap post-processing around downstream handlers.

## 4. Aborting

- **`Abort()`** (`context.go:207`) sets `c.index = abortIndex` (a large constant), so the `for` loop condition fails and no further handlers run. The *current* handler still finishes.
- Variants like `AbortWithStatusJSON` write a response and abort in one call.

## Summary of the flow

```
ServeHTTP → handleHTTPRequest → tree lookup → c.handlers = matched chain
         → c.Next() → [mw1 → (Next) → mw2 → (Next) → finalHandler]
```

The chain is a flat slice built by `combineHandlers` at registration; `Next()` walks it via an index cursor, and nesting happens naturally because middleware call `Next()` recursively. Abort short-circuits by jumping the cursor past the end.
