[assistant] Django's ORM turns a `QuerySet` into SQL through a layered pipeline. The key idea is that nothing touches the database until you actually need the results — *lazy evaluation* — and the work is split across three main objects: `QuerySet`, `Query`, and `SQLCompiler`.

## The three layers

**1. `QuerySet` (`django/db/models/query.py`)** — the high-level, user-facing API. When you write `Author.objects.filter(...)`, the manager hands you a `QuerySet`. It holds:
- a reference to a `Query` object (`self.query`)
- a result cache (`self._result_cache`, initially `None`)

Every method like `.filter()`, `.exclude()`, `.order_by()`, `.annotate()` is **chainable and non-mutating**: it clones the current QuerySet (`._chain()` → `._clone()`), mutates the *clone's* Query, and returns it. This is why QuerySets are reusable and why nothing executes mid-chain.

**2. `Query` (`django/db/models/sql/query.py`)** — the database-agnostic, structural representation of the SQL statement. It's a tree-ish data structure, not a string. It accumulates:
- `where` — a `WhereNode` tree of lookups/conditions
- `select`, `annotations`, `group_by`, `order_by`, `distinct`
- `alias_map` — table aliases and the JOINs between them

When you call `.filter(author__name="x")`, the QuerySet calls `query.add_q(Q(...))`. The Query resolves `author__name` by splitting on `__`, walking the model's `_meta` to follow relations (setting up JOINs via `setup_joins`/`names_to_path`), and identifying the final field + lookup (`exact`, `gt`, `icontains`, etc.). The result is added to the `WhereNode` tree. No SQL yet.

## When execution happens (triggering evaluation)

The Query stays inert until something *forces* it. Triggers include:
- iterating (`for obj in qs` → `__iter__`)
- `len(qs)`, `list(qs)`, `bool(qs)`, slicing with a step
- `repr()` (in a shell), `.count()`, `.exists()`, `.get()`, etc.

The first time, `QuerySet._fetch_all()` runs, calls the iterable class (e.g. `ModelIterable`), and populates `_result_cache`. Subsequent iterations reuse the cache — no second query.

## Compilation: Query → SQL string

`_fetch_all` ultimately calls `query.get_compiler(using=db)` to get a **`SQLCompiler`** (`django/db/models/sql/compiler.py`), backend-specific. The compiler's `as_sql()`:

1. `pre_sql_setup()` — resolves the SELECT columns, GROUP BY, ORDER BY.
2. `get_from_clause()` — builds FROM + JOINs by walking `alias_map`.
3. Compiles the `WhereNode` to a SQL fragment with `%s` placeholders, collecting parameters separately (this is how Django avoids SQL injection — values never get interpolated into the string).
4. Assembles the final `SELECT ... FROM ... WHERE ... ORDER BY ...` string plus a `params` tuple.

Each backend's operations/compiler can vary the dialect (LIMIT vs TOP, quoting, etc.).

## Execution against the database

`SQLCompiler.execute_sql()`:
1. Gets a connection for the chosen DB alias (`connections[using]`).
2. Obtains a cursor and calls `cursor.execute(sql, params)` — the DB driver (psycopg, mysqlclient, sqlite3) does the actual parameter binding.
3. Fetches rows (`cursor.fetchmany`/iterates) and returns raw tuples.

## Mapping rows back to objects

The iterable class (`ModelIterable`) takes the raw rows and reconstructs model instances — calling `model.from_db()`, assigning column values to fields, wiring up `select_related` objects, deferred fields, and annotations. These populated instances go into `_result_cache`.

## Summary of the flow

```
Author.objects.filter(...).order_by(...)
        │  (chainable, lazy — clones QuerySet, mutates Query)
        ▼
   QuerySet ──holds──► Query (WhereNode tree, aliases, no SQL)
        │  evaluation triggered (iterate / len / list / get ...)
        ▼
   _fetch_all → ModelIterable
        │
        ▼
   query.get_compiler(db) → SQLCompiler.as_sql()
        │  builds "SELECT ... WHERE col = %s", params=(...)
        ▼
   execute_sql() → cursor.execute(sql, params)
        │  rows returned
        ▼
   from_db() → model instances → cached in _result_cache
```

**The two takeaways:** (1) Query construction is pure data-structure manipulation, decoupled from SQL generation, which is why the same Query can compile to different dialects; and (2) SQL strings and parameters are kept separate end-to-end, which is what makes parameterized queries safe by default.
