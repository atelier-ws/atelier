## The three layers

Django splits the work across three objects, each at a different altitude:

| Object | File | Role |
|---|---|---|
| `QuerySet` | `django/db/models/query.py` | Public, lazy, chainable API + result caching |
| `Query` | `django/db/models/sql/query.py` | Backend-agnostic AST of the query (the "what") |
| `SQLCompiler` | `django/db/models/sql/compiler.py` | Turns the `Query` into a real SQL string for one specific backend and runs it (the "how") |

A `QuerySet` wraps a `Query` (`self.query`). Building a query mutates/clones the `Query`; executing it spins up a `SQLCompiler`.

## 1. Building — chaining is lazy and copy-on-write

Methods like `.filter()`, `.exclude()`, `.annotate()` never touch the database. Each returns a **new** `QuerySet` via `_clone()` (`query.py:2218`):

```python
def _clone(self):
    c = self.__class__(model=self.model, query=self.query.chain(), ...)
    ...
    return c
```

The important line is `query=self.query.chain()` — the underlying `Query` is copied, so the original is untouched (this is why `qs2 = qs1.filter(...)` doesn't mutate `qs1`).

Inside the cloned `Query`, `.filter(name="x")` is translated into a **WHERE tree**:
- `Query.add_q()` (`sql/query.py:1663`) normalizes the `Q` object and adds it to `self.where`.
- `Query.build_filter()` (`sql/query.py:1488`) resolves each lookup (`name__icontains`, FK spanning, etc.), creating any needed JOINs in the `alias_map` and producing a node in the where-tree (a `WhereNode`).

So the `Query` accumulates pure state: `where`, `select`, `annotations`, `group_by`, `order_by`, `alias_map`/joins, `low_mark`/`high_mark` (slicing), etc. No SQL exists yet — it's all an abstract representation.

## 2. Triggering execution

The `QuerySet` stays lazy until something *forces* it. Every such trigger funnels through `_fetch_all()` (`query.py:2238`):

```python
def _fetch_all(self):
    if self._result_cache is None:
        self._result_cache = list(self._iterable_class(self))
    ...
```

`_result_cache` is the key to laziness + caching: it's `None` until first evaluation, then holds the materialized list. These all call `_fetch_all()`:
- `__iter__` (`query.py:420`) — `for obj in qs`
- `__len__` (`query.py:416`) — `len(qs)`
- `__bool__` (`query.py:449`) — `if qs:`
- `__getitem__` (`query.py:468`) — indexing/slicing (slicing an unevaluated qs instead just sets `low_mark`/`high_mark` on the `Query` and returns a clone)

Because the result is cached, iterating the same `QuerySet` twice hits the DB once. `_iterable_class` defaults to `ModelIterable` (it changes to `ValuesIterable`, `FlatValuesListIterable`, etc., after `.values()`/`.values_list()`).

## 3. Compiling and running

`list(self._iterable_class(self))` invokes `ModelIterable.__iter__` (`query.py:89`), which is where the three layers connect:

```python
db = queryset.db
compiler = queryset.query.get_compiler(using=db)        # Query -> SQLCompiler
results = compiler.execute_sql(chunked_fetch=..., chunk_size=...)
```

**Backend selection** happens in `Query.get_compiler()` (`sql/query.py:370`):

```python
connection = connections[using]
return connection.ops.compiler(self.compiler)(self, connection, using, elide_empty)
```

This is the moment the backend-agnostic `Query` is bound to a concrete connection. The backend's `ops.compiler` lets e.g. PostgreSQL vs SQLite supply their own `SQLCompiler` subclass.

**SQL string generation** — `SQLCompiler.as_sql()` (`compiler.py:757`) assembles the statement from the `Query`'s state: `pre_sql_setup()` resolves select/order/group-by, then it compiles each clause — `get_from_clause()`, `self.compile(self.where)` → the `WHERE` text and params, distinct, having, limit/offset — and concatenates them into `(sql, params)`. Params are kept **separate** from the SQL string, which is how Django parameterizes queries (SQL-injection safe).

**Execution** — `SQLCompiler.execute_sql()` (`compiler.py:1595`):

```python
sql, params = self.as_sql()
...
cursor = self.connection.cursor()      # or chunked_cursor()
cursor.execute(sql, params)
...
result = cursor_iter(cursor, ...)      # for MULTI
return list(result)                    # unless chunked reads are supported
```

The `result_type` controls fetch behavior: `MULTI` uses `fetchmany()` wrapped in `cursor_iter`, `SINGLE` uses `fetchone()` (used by `.get()`/`[0]`), `ROW_COUNT` returns `cursor.rowcount`, `CURSOR` hands back the raw cursor. If the where-tree is provably empty it raises `EmptyResultSet` and short-circuits to `iter([])` — no DB round-trip at all.

## 4. Rows back to model instances

`execute_sql()` returns raw DB rows (tuples). Back in `ModelIterable.__iter__`, the compiler now has `compiler.select`, `compiler.klass_info`, and `annotation_col_map` populated (a side effect of `as_sql`'s setup). The iterable uses these to map column positions to model fields, build each model instance via its `from_db`/`__init__`, attach annotations, and populate select-related objects (`get_related_populators`). Those instances are what land in `_result_cache`.

## How lookups become JOINs (`build_filter`, `sql/query.py:1488`)

When you write `.filter(author__name="John")`, `build_filter` does the join resolution:
1. `solve_lookup_type()` splits `"author__name"` into the field path and the lookup (`exact`).
2. `setup_joins()` walks the relation path and **registers each needed JOIN in the `Query.alias_map`** (assigning table aliases). This is the only place joins get created.
3. `trim_joins()` drops joins that aren't actually needed (e.g. when the final field is the FK column itself).
4. `build_lookup()` builds the leaf `Lookup` object (`Col(...) = %s`), wrapped in a `WhereNode`.

`add_q()` (`sql/query.py:1663`) then attaches that node to `Query.where` with `AND`. Alias reuse is governed by `used_aliases`, which `Query.chain()` resets per filter chain — that's why `.filter(x).filter(y)` across a relation can produce *two* joins, while `.filter(x, y)` reuses one.

## The type-conversion step I skipped over

Between raw cursor tuples and model instances there's a converter pass. `SQLCompiler.results_iter()` (`compiler.py:1565`) takes the rows from `execute_sql()` and applies `get_converters()` — backend/field-specific functions that turn DB representations into Python objects (e.g. SQLite has no native datetime, so a string gets parsed into a `datetime`). Only *then* does `ModelIterable` call `model_cls.from_db()` to build instances. So the full row pipeline is:

```
cursor.fetchmany() → cursor_iter (chunks) → results_iter (type converters) → ModelIterable (from_db → model instance) → _result_cache
```
