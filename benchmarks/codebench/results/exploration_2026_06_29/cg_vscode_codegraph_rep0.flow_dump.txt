## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

## Overview

The extension host runs in a **separate process** (or a Web Worker in the browser). It never touches the workbench UI directly — all communication happens over an **asynchronous, message-passing RPC protocol**. There are three layers:

```
Extension API code  ($ methods on proxies)
        │
   RPCProtocol      ← the actor/proxy RPC layer (rpcProtocol.ts)
        │
IMessagePassingProtocol  ← a duplex byte-stream (send(VSBuffer) / onMessage)
        │
   transport         ← MessagePort (Electron), socket (remote), Worker (web)
```

## 1. The transport: `IMessagePassingProtocol`

At the bottom is a simple duplex channel — `send(buffer)` out, `onMessage` in — abstracted so the same RPC works over different transports:

- **Local desktop** (`localProcessExtensionHost.ts`): a Node child process connected via an Electron **MessagePort** (`acquirePort` / `MessagePortExtHostConnection`).
- **Remote** (`remoteExtensionHost.ts`): a **socket** to the remote server.
- **Web** (`extensionHostWorker.ts`): a **Web Worker** `postMessage` channel.

Startup handshake uses simple typed control messages (`extensionHostProtocol.ts`): `VSCODE_EXTHOST_IPC_READY` / `..._SOCKET`, and `MessageType.Initialized` / `Ready` / `Terminate` (single-byte `VSBuffer`s via `createMessageOfType`).

## 2. The RPC layer: `RPCProtocol`

`src/vs/workbench/services/extensions/common/rpcProtocol.ts` is the heart of it. Both sides construct one over the transport:

```typescript
this._rpcProtocol = new RPCProtocol(protocol, null, uriTransformer);  // extensionHostMain.ts:176
```

It implements a bidirectional **actor model**. Each side registers local objects and gets **proxies** to remote ones, keyed by `ProxyIdentifier` (numeric ids):

- **Main → ExtHost** proxies live in `ExtHostContext` (e.g. `ExtHostContext.ExtHostDocuments`).
- **ExtHost → Main** proxies live in `MainContext` (e.g. `MainContext.MainThreadExtensionService`).

`getProxy(identifier)` returns a JS `Proxy`. Any property beginning with `$` becomes a remote call (`extHost.protocol.ts` — all cross-process methods are named `$foo` by convention):

```typescript
const proxy = rpcProtocol.getProxy(MainContext.MainThreadExtensionService);
await proxy.$asBrowserUri(uri);   // → serialized, sent, awaited
```

## 3. Request/response mechanics

- **Outgoing** (`_remoteCall`, line 461): assigns an incrementing `req` id, serializes args into a `MessageBuffer` (`MessageIO.serializeRequest`), stores a `LazyPromise` in `_pendingRPCReplies`, and calls `this._protocol.send(msg)`.
- **Incoming** (`_receiveOneMessage`, line 282): reads the message type + `req`, then dispatches — `_receiveRequest` invokes the local actor's method (`_doInvokeHandler`) and sends back a reply; `_receiveReply` / `_receiveReplyErr` resolve/reject the pending promise by `req`.
- **Cancellation**: if the last arg is a `CancellationToken`, a `Cancel` message is sent and the receiver cancels its `CancellationTokenSource`.
- **Serialization**: multiple `MessageType`s optimize the payload — JSON args, "mixed" args (JSON + raw `VSBuffer`s), plain `VSBuffer` replies, etc. URIs are transformed via the `IURITransformer` (important for remote scenarios where paths differ across machines).
- **Responsiveness monitoring**: every request is **acknowledged** (`MessageType.Acknowledged`). If acks don't arrive within `UNRESPONSIVE_TIME` (3s), `RPCProtocol` fires `onDidChangeResponsiveState` → this is how VS Code shows the "Extension host unresponsive" warning.

## Putting it together

`ExtensionHostMain` (`extensionHostMain.ts:161`) is the entry point inside the ext host process. It wraps the transport in an `RPCProtocol`, exposes it through `ExtHostRpcService`, and every `ExtHost*` service uses that service to get its `MainThread*` proxy. On the workbench side, the mirror `MainThread*` objects are registered as the local actors. So a call like `vscode.window.showInformationMessage(...)` becomes a `$showMessage` remote call marshalled through this exact pipeline.

**Key files:**
- `rpcProtocol.ts` — the RPC engine (proxies, req/reply, cancellation, responsiveness)
- `extHost.protocol.ts` — the `MainContext` / `ExtHostContext` proxy identifier tables (the `$` method contract)
- `extensionHostMain.ts` — ext-host bootstrap wiring the protocol into services
- `extensionHostProtocol.ts` — init data + startup handshake message types
- `localProcessExtensionHost.ts` / `remoteExtensionHost.ts` — transport setup per environment
