## The two-canvas architecture

Excalidraw layers **two separate `<canvas>` elements**, each with its own render path:

- **Static scene** (`packages/excalidraw/renderer/staticScene.ts` → `_renderStaticScene`) — draws the actual elements: shapes, text, images, frames, grid.
- **Interactive scene** (`renderer/interactiveScene.ts`) — draws transient UI: selection boxes, transform handles, snap lines, remote collaborator cursors. Redrawn far more often (on every mouse move) but cheap.

Splitting them means dragging a selection handle doesn't force every element to repaint.

## Render entry point and viewport culling

The `Renderer` class (`scene/Renderer.ts`) prepares *what* to draw:

- `getVisibleCanvasElements()` culls to elements passing `isElementInViewport` — only on-screen elements are rendered.
- `getRenderableElementsMap()` builds the map to draw, excluding the element being actively created and any text element currently being edited (those render via an overlaid DOM `<textarea>`).
- This is wrapped in `memoize`, so the visible set is recomputed only when its inputs change.

`_renderStaticScene` then iterates `visibleElements`, and for each calls `renderElement(...)` (from `packages/element/src/renderElement.ts`), handling frame clipping, bound text, and link icons along the way.

## Drawing an element — two cache layers

Excalidraw uses **rough.js** for the hand-drawn look, which is expensive, so there are two caches keyed on the element object via `WeakMap`:

1. **`ShapeCache`** (`packages/element/src/shape.ts`) — caches the rough.js `Drawable` (the geometric "shape" description). `ShapeCache.generateElementShape()` returns the cached shape unless it's missing or the theme changed. The heavy `_generateElementShape()` (the big `switch` over element types) runs only on a miss.

2. **`elementWithCanvasCache`** (`renderElement.ts`) — caches a **pre-rendered off-screen canvas** of each element. `generateElementWithCanvas()` reuses it unless something invalidates it. The main scene render then just `drawImage`s these cached bitmaps onto the visible canvas — a memcpy instead of re-stroking paths. It regenerates when:
   - the zoom value changed (`shouldRegenerateBecauseZoom`),
   - the theme changed,
   - bound-text version, image crop, or containing-frame opacity changed,
   - (for arrows with labels) the angle changed.

## How updates invalidate caches — version & versionNonce

Element edits go through `mutateElement()` (`packages/element/src/mutateElement.ts`). It:

1. Diffs each updated field and only proceeds if something actually changed (`didChange`).
2. Calls **`ShapeCache.delete(element)`** when geometry-affecting fields change (`width`, `height`, `points`, `fileId`) — and `ShapeCache.delete` also clears `elementWithCanvasCache`, so the bitmap is regenerated too.
3. Bumps **`element.version` (+1)** and assigns a new random **`element.versionNonce`**, plus an `updated` timestamp.

The `version`/`versionNonce` pair is the change-detection signal across the whole app: scene-version comparison decides whether a re-render/persist/collab-sync is needed, and `versionNonce` (random, not monotonic) is what lets distributed collaborators deterministically reconcile concurrent edits. Note that because caches are keyed on the element *object reference* via `WeakMap`, replacing an element (e.g. `newElementWith`, which spreads into a new object) naturally misses the cache without any explicit deletion.

## Scheduling: throttled to animation frames

Both scene renders are throttled to the browser's paint cycle. `staticScene.ts` exports `renderStaticSceneThrottled = throttleRAF(_renderStaticScene)`, and `renderStaticScene(config, throttle?)` chooses throttled vs. immediate. Many React state changes within one frame therefore collapse into a single `requestAnimationFrame`-aligned repaint rather than redrawing per state update. `App.tsx` drives this through the `Renderer` and the interactive-scene callback after each component update.

## End-to-end data flow

```
user input → mutateElement() → bumps version/versionNonce,
                               deletes ShapeCache (+ canvas cache) if geometry changed
   → App schedules render (throttleRAF, ~60fps)
   → Renderer: cull to visible elements (memoized) + build renderable map
   → _renderStaticScene: for each visible element → renderElement()
        → ShapeCache.get/generate (rough.js Drawable, cached)
        → generateElementWithCanvas (off-screen bitmap, cached) → drawImage
   → interactiveScene renders selection/handles/snaps on the top canvas
```

The core performance idea: **cache aggressively per-element on object identity, invalidate precisely through `mutateElement`, cull to the viewport, and coalesce paints into animation frames** — with the volatile UI chrome isolated on a second canvas so it never forces the element layer to repaint.
