[assistant] OkHttp's interceptor chain is a layered pipeline built on the **Chain of Responsibility** pattern. Here's how a request flows through it.

## The core mechanism: `RealInterceptorChain`

When you call `call.execute()` or `call.enqueue()`, OkHttp builds a single ordered list of interceptors and wraps the request in a `RealInterceptorChain`. The whole flow is driven by one recursive method, `chain.proceed(request)`:

1. Each interceptor receives the `Chain`.
2. It does whatever it wants **before** the request goes out (rewrite headers, retry logic, etc.).
3. It calls `chain.proceed(request)` to hand off to the *next* interceptor.
4. That call eventually returns a `Response`, bubbling back up.
5. The interceptor can then modify the **response** on the way back out.

So each interceptor sees the request on the way *down* and the response on the way *up*. The `proceed()` call is the seam — everything before it is request-side, everything after is response-side.

```
request  →  A  →  B  →  C  →  [network]
                                  │
response ←  A  ←  B  ←  C  ←──────┘
```

## The order of the built-in interceptors

`RealCall.getResponseWithInterceptorChain()` assembles the list in this order:

1. **Your application interceptors** (added via `OkHttpClient.Builder.addInterceptor`) — run first, see the logical request/response, called once per logical call even across retries/redirects.
2. **`RetryAndFollowUpInterceptor`** — handles connection recovery, retries, redirects, and auth challenges (it's the loop that re-runs the lower chain).
3. **`BridgeInterceptor`** — translates the user request into a network request: adds `Content-Type`, `Content-Length`, `Host`, `Connection: Keep-Alive`, cookies, gzip (`Accept-Encoding`), and transparently decompresses gzip responses.
4. **`CacheInterceptor`** — serves from cache, validates with conditional requests, writes responses to cache.
5. **`ConnectInterceptor`** — opens/reuses a healthy connection to the target (the `ExchangeFinder` / connection pool work).
6. **Your network interceptors** (added via `addNetworkInterceptor`) — see the actual on-the-wire request/response, including redirects and the `Connection` object.
7. **`CallServerInterceptor`** — the terminal interceptor; writes the request bytes and reads the response bytes. It does **not** call `proceed()` — it ends the chain.

## Key consequences of this design

- **Application vs. network interceptors differ** precisely because of *where* they sit. App interceptors run once per `Call` and don't see redirects/retries; network interceptors run once per actual HTTP exchange and see all of them.
- **An interceptor can short-circuit** by returning a `Response` without calling `proceed()` (e.g., a mock/cache layer).
- **`proceed()` must be called exactly once** by non-terminal interceptors — calling it zero times (without returning a response) or multiple times throws.
- The chain carries the `Exchange` (the connection/stream context), which is why `ConnectInterceptor` must run before anything that touches the socket.
