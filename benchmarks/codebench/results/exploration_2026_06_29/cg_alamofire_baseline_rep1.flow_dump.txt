# How Alamofire Builds, Sends, and Validates a Request

Alamofire wraps Apple's `URLSession`. A `Session` (default `AF`) is the top-level coordinator; every request is a `Request` subclass (`DataRequest`, `DownloadRequest`, `UploadRequest`, etc.) that owns a state machine and a `URLSessionTask`. Everything is serialized through a `rootQueue` for thread safety, with URLRequest construction offloaded to a concurrent `requestQueue`.

## 1. Build — turning your call into a `URLRequest`

**Entry point** (`Session.swift:318/367/393`). `AF.request(url, method:parameters:encoding:headers:interceptor:)` doesn't build anything immediately. It wraps your inputs in a lazy `URLRequestConvertible` box and constructs a `DataRequest`:

- The `ParameterEncoding` overload → `RequestConvertible` struct (`Session.swift:286`)
- The `Encodable` + `ParameterEncoder` overload → `RequestEncodableConvertible` (`Session.swift:336`)
- All overloads funnel into the core `request(_ convertible: URLRequestConvertible, ...)` (`Session.swift:393`), which creates the `DataRequest` and calls `performEagerlyIfNecessary`.

The actual `URLRequest` is built later, off the `requestQueue`, when the convertible's `asURLRequest()` runs (`Session.swift:1254`). That does three things:

1. **URL + method + headers** → `URLRequest.init(url:method:headers:)` (`URLConvertible+URLRequestConvertible.swift:97`). `URLConvertible.asURL()` converts a `String`/`URL`/`URLComponents`, throwing `AFError.invalidURL` on failure.
2. **`requestModifier(&request)`** — optional hook to mutate the request (e.g. set timeout/cache policy).
3. **Parameter encoding**:
   - `ParameterEncoding` path (`ParameterEncoding.swift`): `URLEncoding` puts params in the **query string** or **httpBody** depending on `Destination` (`.methodDependent` = query for GET/HEAD/DELETE, body otherwise); `JSONEncoding` serializes to the body with `application/json`.
   - `ParameterEncoder` path (`ParameterEncoder.swift`): the `Encodable` equivalents — `JSONParameterEncoder` and `URLEncodedFormParameterEncoder`.

## 2. Send — adapting, creating the task, resuming

By default `startRequestsImmediately` is on, so `resume()` (`Request.swift:768`) flips the request from `.initialized` → `.resumed`. With no task yet, it calls `delegate.readyToPerform` → `Session.perform` (`Session.swift:1166`).

**`perform`** runs on `rootQueue`, adds the request to `activeRequests`, then dispatches type-specifically (`performDataRequest`, `performUploadRequest`, …) into **`performSetupOperations`** (`Session.swift:1246`), the heart of sending:

1. `convertible.asURLRequest()` builds the `URLRequest` (step 1 above) and `initialRequest.validate()` checks it. Failure → `didFailToCreateURLRequest`.
2. **Adaptation** (`Session.swift:1265–1286`): if an interceptor exists, `adapter.adapt(initialRequest, using:state)` runs to mutate the request asynchronously (e.g. attach an auth token). `Session.adapter(for:)` (`:1315`) composes the session-level and request-level interceptors into a single chained `Interceptor` (`RequestInterceptor.swift`). Adapter failure → `requestAdaptationFailed`.
3. **Task creation** — `didCreateURLRequest` (`Session.swift:1291`): calls `request.task(for:urlRequest, using:session)` (`DataRequest` → `session.dataTask`), stores the pairing in the bidirectional **`RequestTaskMap`** (`Session.swift:1299`), then `request.didCreateTask` (`Request.swift:388`) applies the current state to the task — if `.resumed`, it calls `task.resume()` and the network request goes out.

**Delegate callbacks** (`SessionDelegate.swift`, all on `rootQueue`) feed the response back: auth challenges, `didReceiveResponse`, `didReceive(data:)` (accumulated into `DataRequest.data`), `didFinishCollecting(metrics)`, and finally `didCompleteWithError` → `request.didCompleteTask`.

## 3. Validate — and serialize the response

You attach validation with `.validate(statusCode:)`, `.validate(contentType:)`, or a custom `.validate { req, resp, data in ... }` (`Validation.swift`; the closure type is `(URLRequest?, HTTPURLResponse, Data?) -> ValidationResult`). Each call appends a closure to the request's thread-safe `validators` array (`Request.swift:207`); they don't run yet.

When the task completes, **`didCompleteTask`** (`Request.swift:515`) runs all validators **synchronously** on the underlying queue (`Request.swift:520`). A validator that fails sets `self.error` — e.g. `AFError.responseValidationFailed(.unacceptableStatusCode(...))` or `.unacceptableContentType(...)` (`AFError.swift:99`).

Then **`retryOrFinish`** (`Request.swift:543`) decides:
- **Error present** → ask the interceptor's `RequestRetrier` via `Session.retryResult` (`Session.swift:1349`). It returns a `RetryResult` (`.retry`, `.retryWithDelay`, `.doNotRetry`, …); `RetryPolicy`/`AuthenticationInterceptor` implement common strategies. On retry, `prepareForRetry` bumps the count and the whole flow restarts at `perform` (`Session.swift:1365`).
- **No error / no retry** → `finish()` (`Request.swift:563`), which transitions to `.finished` and enqueues the **response serializers** on the `serializationQueue`.

**Serialization** (`ResponseSerialization.swift`): handlers like `responseDecodable`, `responseData`, `responseString` each run a `ResponseSerializer.serialize(request:response:data:error:)`. Crucially, the **validation error is passed into the serializer as the `error` argument** — so if validation failed, the serializer rethrows it before trying to decode, and you get an `AFDataResponse` whose `.result` is `.failure(AFError)`. Otherwise it preprocesses/decodes the body (decoding errors surface as `.responseSerializationFailed(.decodingFailed(...))`) and the completion handler fires on your chosen queue.

## The one-line model

> `request()` lazily boxes your inputs → on `resume`, `performSetupOperations` builds the `URLRequest`, an interceptor *adapts* it, and a `URLSessionTask` is created/resumed → delegate callbacks collect the data → on completion, **validators run first (setting an error on failure), then a retrier may restart the whole flow**, and finally serializers turn the bytes (or the error) into a typed `AFDataResponse`.

The recurring pattern: a `Request` state machine (`initialized → resumed → finished`, plus `suspended`/`cancelled`) driven entirely through `rootQueue`, with the `RequestTaskMap` bridging Alamofire's `Request` and `URLSession`'s task.
