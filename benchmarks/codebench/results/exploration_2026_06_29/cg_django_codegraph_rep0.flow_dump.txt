[assistant] Django's ORM separates **lazy query building** from **execution**, deferring any database hit until you actually consume results. Here's the full pipeline, grounded in the code:

## 1. Building — the `QuerySet` / `Query` split

A `QuerySet` (`django/db/models/query.py`) is a thin, lazy, chainable wrapper. Each filtering operation (`.filter()`, `.exclude()`, `.annotate()`, `.order_by()`, …) **clones** the queryset and mutates its underlying `Query` object — it never touches the database. The `Query` (`django/db/models/sql/query.py:232`) is the abstract, backend-independent representation of the SQL statement. Its most important state:

- `alias_map` — the joins (`Query.__init__`, query.py:314): which tables are joined and how
- `where` — a `WhereNode` tree of filter conditions
- `annotations`, `select`, `group_by`, `order_by`, `extra`, limits/offsets

So `Author.objects.filter(name="x").order_by("age")` just produces a `Query` with a populated `where` tree and ordering — still zero SQL, zero DB contact.

## 2. Triggering execution — `_fetch_all`

Nothing runs until the queryset is consumed (iterated, `len()`, `list()`, indexed, etc.). That funnels into `QuerySet._fetch_all` (query.py:2238):

```python
def _fetch_all(self):
    if self._result_cache is None:
        self._result_cache = list(self._iterable_class(self))
```

`_iterable_class` is chosen by the queryset type — `ModelIterable` for normal model instances, `ValuesIterable`, `ValuesListIterable`, `FlatValuesListIterable` for `.values()`/`.values_list()`. Results are cached in `_result_cache`, so re-iterating never re-queries.

## 3. Compilation — `Query` → SQL string

`ModelIterable.__iter__` (query.py:92) drives the actual work:

```python
compiler = queryset.query.get_compiler(using=db)
results = compiler.execute_sql(chunked_fetch=..., chunk_size=...)
```

`Query.get_compiler` (query.py:370) asks the **database backend** for the right compiler class — this is where backend-specific SQL dialects come in:

```python
return connection.ops.compiler(self.compiler)(self, connection, using, elide_empty)
```

The `SQLCompiler` (`django/db/models/sql/compiler.py:43`) is what translates the abstract `Query` into an actual SQL string. `SQLCompiler.as_sql()` (compiler.py:757) assembles the clauses — `get_select()`, `get_from_clause()`, the `WHERE`, `get_group_by()`, `get_order_by()`, limits — into a `(sql, params)` tuple. Backends override pieces of this: e.g. MySQL's `SQLDeleteCompiler.as_sql` / `SQLUpdateCompiler.as_sql` (`django/db/backends/mysql/compiler.py`) emit non-standard syntax where it's more efficient.

## 4. Execution — `execute_sql`

`SQLCompiler.execute_sql` (compiler.py:1595) runs the compiled SQL against the DB:

```python
sql, params = self.as_sql()          # compile
if not sql: raise EmptyResultSet
cursor = self.connection.cursor()    # or chunked_cursor() for server-side cursors
cursor.execute(sql, params)
```

The return shape depends on `result_type`:
- **`MULTI`** — wraps the cursor in `cursor_iter` for chunked `fetchmany()` reads (the default path for querysets)
- **`SINGLE`** — one `fetchone()` row (used by `.get()`, aggregates)
- **`ROW_COUNT`** — `cursor.rowcount` (used by `.update()`/`.delete()`)
- **`CURSOR`** — the raw cursor

An `EmptyResultSet` (e.g. `filter(pk__in=[])`) short-circuits to an empty iterator without ever hitting the DB.

## 5. Row → object hydration

Raw DB rows are turned back into Python objects by `SQLCompiler.results_iter` (compiler.py:1565):

```python
results = self.execute_sql(MULTI, ...)
fields = [s[0] for s in self.select[0:self.col_count]]
converters = self.get_converters(fields)   # DB values → Python values
rows = chain.from_iterable(results)
rows = self.apply_converters(rows, converters)
```

Back in `ModelIterable.__iter__`, each row is then materialized:

```python
for row in compiler.results_iter(results):
    obj = model_cls.from_db(db, init_list, row[model_fields_start:model_fields_end], ...)
    # populate select_related() relations via related_populators
    # attach annotations from annotation_col_map
    yield obj
```

This is also where `select_related` joins get unpacked into related objects (`get_related_populators`) and annotations are set as attributes.

## Summary of the layers

| Layer | Responsibility |
|-------|----------------|
| `QuerySet` | Lazy, chainable public API; caches results; picks an iterable class |
| `Query` | Backend-independent abstract query state (joins, where-tree, select, ordering) |
| `SQLCompiler` (via `get_compiler`) | Turns `Query` into a backend-specific `(sql, params)` string in `as_sql()` |
| `execute_sql` | Runs it on a DB cursor, returns rows by `result_type` |
| `results_iter` + `from_db` | Applies converters and hydrates rows into model instances |

The key design idea: **`QuerySet` is lazy and chainable, `Query` holds abstract structure, and the `SQLCompiler` (chosen per-backend) is the only component that produces and executes real SQL** — which is how the same ORM code targets PostgreSQL, MySQL, SQLite, and Oracle.
