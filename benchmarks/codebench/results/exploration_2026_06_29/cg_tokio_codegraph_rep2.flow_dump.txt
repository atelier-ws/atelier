## The core model

Tokio never "polls" a future in a busy loop. Instead, tasks are **event-driven**: a task is only in a run queue when it has been *notified* (woken). A worker thread's job is to pull notified tasks off queues and poll them exactly once each; if the poll returns `Pending`, the task goes to sleep until its waker fires again.

There are two schedulers with the same shape: **multi-thread** (default, work-stealing across N workers) and **current-thread** (single-threaded). Both plug into a shared task/harness layer via the `Schedule` trait (`tokio/src/runtime/task/mod.rs:293`).

## The worker run loop (multi-thread)

Each worker owns a `Core` and spins in `Context::run` (`worker.rs:561`). One iteration = one "tick":

```rust
while !core.is_shutdown {
    core.tick();
    core = self.maintenance(core);
    if let Some(task) = core.next_task(&self.worker) {   // 1. local/global queue
        core = self.run_task(task, core)?;
        continue;
    }
    if let Some(task) = core.steal_work(&self.worker) {  // 2. steal from peers
        core = self.run_task(task, core)?;
    } else {
        core = self.park(core);                          // 3. sleep on the driver
    }
}
```

Where tasks come from, in priority order:

1. **`next_task`** — the worker's own local run queue (a fixed-capacity SPSC ring buffer, `queue.rs`), plus a periodic dip into the **global injection queue**. The `global_queue_interval` (`Core` field, dynamically tuned in `tune_global_queue_interval`, `worker.rs:1307`) forces a check of the global queue every N ticks so injected tasks don't starve.
2. **`steal_work`** (`worker.rs:1141`) — if local work is exhausted, the worker transitions to "searching," picks a **random** peer worker, and steals *half* their queue (`steal_into`). Randomized start index spreads contention. Falls back to the global queue.
3. **`park`** (`worker.rs:810`) — no work anywhere, so it parks on the I/O/timer **driver**, blocking until an event (I/O readiness, timer, or an unpark notification) wakes it. On unpark it runs maintenance and re-enters the loop.

## Running one task + the LIFO slot

`run_task` (`worker.rs:630`) is where polling happens. It sets the `Core` into the thread-local context, then under a coop budget calls `task.run()`. The interesting part is the **LIFO slot** optimization: after polling, it loops draining `core.lifo_slot`:

- When a running task wakes another task *on the same worker* (`schedule_local`, `worker.rs:1353`), that task goes into a single-entry `lifo_slot` rather than the queue tail. This keeps message-passing chains (task A → task B → …) hot in cache.
- To prevent two ping-ponging tasks from starving everything else, the slot is capped at `MAX_LIFO_POLLS_PER_TICK = 3` (`worker.rs:743`); after that the LIFO slot is disabled and the task goes to the queue.
- If the coop budget runs out mid-chain, the LIFO task is pushed to the back of the queue instead (`worker.rs:716`).

## Actually polling the future — the harness

`task.run()` → the harness (`harness.rs`) drives the state machine:

1. `transition_to_running` (`state.rs:117`) atomically locks the `RUNNING` bit and clears the `NOTIFIED` bit — so any wake that arrives *during* the poll is detected afterward.
2. `poll_future` (`harness.rs:521`) builds a `Context` with the task's waker and calls the future's `poll`, wrapped in `catch_unwind` so a panicking task can't take down the worker.
3. On `Poll::Pending`: `transition_to_idle`. If the task was notified again during the poll (`OkNotified`), it's immediately re-scheduled; otherwise it sleeps.
4. On `Poll::Ready`: the output is stored, the task transitions to complete, and the `JoinHandle` is woken.

## How a task gets *back* into a queue (waking)

When a future's waker fires (`wake`/`wake_by_ref`), the state machine (`transition_to_notified_by_val`/`_by_ref`, `state.rs:215`) decides whether to submit the task, then calls the scheduler's `schedule` (the `Schedule` trait impl). For multi-thread that's `schedule_task` (`worker.rs:1327`):

- Woken **from within its own worker** → `schedule_local` (LIFO slot or local queue), and `notify_parked_local` unparks a sleeping worker if the queue grew.
- Woken **from another thread / outside the runtime** → `push_remote_task` onto the global injection queue + `notify_parked_remote` to wake a worker.

The current-thread scheduler's `schedule` (`current_thread/mod.rs:671`) is the same idea minus stealing: same-thread pushes to the local `VecDeque`, otherwise pushes to the inject queue and unparks the driver.

## Cooperative scheduling (the budget)

Every task poll runs inside `coop::budget(...)` (`worker.rs:675`). Each task starts with a fixed budget; tokio's own resource operations (sockets, channels, etc.) decrement it and return `Pending` once it hits zero, even if data is available (`has_budget_remaining`). This forces long-running or always-ready tasks to yield back to the scheduler, preventing one task from monopolizing a worker and starving others. The budget lives in the thread-local `Context` (`context.rs:65`).

## Blocking work

`spawn_blocking` tasks don't go through this at all — they run on a separate dedicated thread pool (`blocking/pool.rs`), whose `Inner::run` (`pool.rs:503`) is a simple lock/condvar loop pulling `Task`s off a shared queue and running them to completion. This keeps blocking code off the async workers.
