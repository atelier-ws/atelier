## How the extension host communicates with the main process

VS Code runs extensions in a **separate process** (the *extension host*) isolated from the main renderer/window process. The two sides talk over a custom **RPC protocol** built on a binary message-passing layer. The design has three layers: **proxies/identifiers**, the **RPCProtocol**, and a swappable **transport**.

### 1. The RPC layer — `RPCProtocol`

`src/vs/workbench/services/extensions/common/rpcProtocol.ts`

Both sides instantiate an `RPCProtocol` over a transport:
- Main side: `extensionHostManager.ts:258` → `new RPCProtocol(protocol, logger)`
- Ext-host side: `src/vs/workbench/api/common/extensionHostMain.ts:176` → `new RPCProtocol(protocol, null, uriTransformer)`

`RPCProtocol` implements `IRPCProtocol` (`proxyIdentifier.ts`). Its core methods:
- `getProxy(identifier)` — returns a lazy JS `Proxy`. Any method call on it (`$someMethod(...)`) is intercepted and turned into a remote request.
- `set(identifier, instance)` — registers a real local object that incoming calls are dispatched to.

By convention, every remotable method is prefixed with **`$`** (e.g. `$executeCommand`). Requests carry a numeric `rpcId` (which actor), a method name, and serialized args.

### 2. Wire format — binary `MessageIO` / `MessageType`

`RPCProtocol` serializes messages itself into `VSBuffer`s rather than relying on raw JSON. The message kinds (`rpcProtocol.ts`) include:

```
RequestJSONArgs / RequestJSONArgsWithCancellation
RequestMixedArgs / ...WithCancellation   // when some args are VSBuffers
Acknowledged, Cancel
ReplyOKEmpty, ReplyOKJSON, ReplyOKJSONWithBuffers, ReplyOKVSBuffer
ReplyErrError, ReplyErrEmpty
```

`MessageIO.serializeRequest` / `deserializeRequest*` pack the request into a length-prefixed buffer. Args that are `VSBuffer`s are sent as raw binary (the "Mixed args" path) to avoid base64 overhead. `URI`s are rewritten via a `uriReplacer`/`uriTransformer` (important for remote/SSH scenarios). Cancellation flows as a separate `Cancel` message keyed by request id.

### 3. The actor naming convention — `MainContext` / `ExtHostContext`

`src/vs/workbench/api/common/extHost.protocol.ts`

This file is the **contract**: it declares every interface that crosses the boundary, plus two big tables of `ProxyIdentifier`s (created via `createProxyIdentifier`):

- **`MainContext`** — services living in the **main/renderer** process, implemented by `MainThread*` classes (e.g. `MainThreadCommands`, `MainThreadDocuments`). The extension host calls *up* into these.
- **`ExtHostContext`** — services living in the **extension host**, implemented by `ExtHost*` classes (e.g. `ExtHostCommands`). The main process calls *down* into these.

So the rule of thumb:
- `MainThreadXyz` runs in the main process and is registered into `MainContext.MainThreadXyz`.
- `ExtHostXyz` runs in the extension host and is registered into `ExtHostContext.ExtHostXyz`.
- Code gets the other side via `rpcProtocol.getProxy(MainContext.X)` or `getProxy(ExtHostContext.Y)`.

The `MainThread*` classes are wired up as **"customers"** (`extHostCustomers.ts`, `@extHostNamedCustomer`) so they're instantiated and registered with the protocol when an extension host connects.

### 4. The transport layer (swappable per host type)

`RPCProtocol` only needs an `IMessagePassingProtocol` (`{ onMessage, send }`). What provides that differs by host:

**Local/Node extension host** — `src/vs/workbench/services/extensions/electron-browser/localProcessExtensionHost.ts`
Despite being a separate Node process, it now connects over an **Electron `MessagePort`** acquired from the shared/utility process, *not* a socket:
```js
writeExtHostConnection(new MessagePortExtHostConnection(), opts.env);
// ...
port.onmessage = (e) => onMessage.fire(VSBuffer.wrap(e.data));
resolve({ onMessage: onMessage.event, send: msg => port.postMessage(msg.buffer) });
```
The actual process is spawned via the `IExtensionHostStarter` utility-process service. After connect, a handshake runs (`MessageType.Ready` → main sends `IExtensionHostInitData` → `MessageType.Initialized`), defined in `common/extensionHostProtocol.ts`.

**Web/worker extension host** — `src/vs/workbench/services/extensions/browser/webWorkerExtensionHost.ts`
Runs in a sandboxed **iframe hosting a Web Worker** (`worker/webWorkerExtensionHostIframe.html`), communicating via `postMessage` / `MessageChannel`. Same `RPCProtocol` on top, different `IMessagePassingProtocol` underneath.

**Remote extension host** (SSH/containers/WSL) — `src/vs/workbench/services/extensions/common/remoteExtensionHost.ts`
Uses a real socket via `PersistentProtocol` (`base/parts/ipc/common/ipc.net.ts`) and `connectRemoteAgentExtensionHost`, which survives reconnects.

### Putting it together — the message flow

```
Extension code → vscode API
   → ExtHost* class → rpcProtocol.getProxy(MainContext.MainThreadX).$method(args)
   → RPCProtocol serializes (MessageIO) → IMessagePassingProtocol.send
   ═══ MessagePort / Worker postMessage / socket ═══
   → RPCProtocol on main deserializes → dispatches to registered MainThreadX.$method
   → reply travels back the same way (ReplyOK* / ReplyErr*)
```

The same machinery runs symmetrically in the other direction (main → ext host via `ExtHostContext`).

```typescript
private _createProxy<T>(rpcId: number, debugName: string): T {
  const handler = {
    get: (target: any, name: PropertyKey) => {
      if (typeof name === 'string' && !target[name] && name.charCodeAt(0) === CharCode.DollarSign) {
        target[name] = (...myArgs: any[]) => {
          return this._remoteCall(rpcId, name, myArgs);
        };
      }
      return target[name];
    }
  };
  return new Proxy(Object.create(null), handler);
}
```
