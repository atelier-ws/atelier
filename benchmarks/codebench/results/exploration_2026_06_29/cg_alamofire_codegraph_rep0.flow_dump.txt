## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

## 1. Build — turning your arguments into a `URLRequest`

Everything starts at `Session.request(...)`. The convenience overloads wrap your inputs (`url`, `method`, `parameters`, `encoder`, `headers`, `requestModifier`) in a lightweight `URLRequestConvertible` struct, then funnel into the core overload (`Session.swift:393`):

```swift
open func request(_ convertible: any URLRequestConvertible, ...) -> DataRequest {
    let request = DataRequest(convertible: convertible, ...)
    performEagerlyIfNecessary(request)
    return request
}
```

Key point: **the `URLRequest` is not built here.** The `Session` only stores a `convertible` — a recipe for making the request — on the `DataRequest`. The actual `URLRequest` is produced lazily when the request is performed, via `RequestConvertible.asURLRequest()` (`Session.swift:294`):

```swift
func asURLRequest() throws -> URLRequest {
    var request = try URLRequest(url: url, method: method, headers: headers)
    try requestModifier?(&request)          // your inout mutation hook
    return try encoding.encode(request, with: parameters)  // parameter encoding sets body/query + Content-Type
}
```

So building = create bare `URLRequest` → apply your `requestModifier` → let the `ParameterEncoding`/`ParameterEncoder` inject parameters and headers.

## 2. Send — perform, adapt, create the task, resume

Once resumed, `Session.perform` (`Session.swift:1166`) hops onto internal queues and dispatches by concrete type (`UploadRequest` before `DataRequest` because of subtyping). Each lands in `performSetupOperations` (`Session.swift:1246`), the heart of sending:

```swift
do {
    initialRequest = try convertible.asURLRequest()   // build (step 1) happens here
    try initialRequest.validate()                     // pre-flight sanity check
} catch { request.didFailToCreateURLRequest(...); return }  // triggers retry

request.didCreateInitialURLRequest(initialRequest)

guard let adapter = adapter(for: request) else {      // no interceptor → straight to task
    request... didCreateURLRequest(initialRequest, for: request); return
}

adapter.adapt(initialRequest, using: adapterState) { result in   // RequestInterceptor.adapt
    let adaptedRequest = try result.get()
    try adaptedRequest.validate()
    request.didAdaptInitialRequest(...)
    self.didCreateURLRequest(adaptedRequest, for: request)
}
```

If a `RequestInterceptor`/adapter is set (auth tokens, etc.), the `URLRequest` is adapted asynchronously before proceeding. Then `didCreateURLRequest` (`Session.swift:1291`) creates the actual `URLSessionTask` — polymorphically, since `DataRequest.task(for:using:)` returns `session.dataTask(with:)` while `DownloadRequest`/`UploadRequest` override it — and records it in the `requestTaskMap`.

Finally `Request.didCreateTask` (`Request.swift:388`) inspects the request's state machine and, if the request is already `.resumed`, calls `task.resume()` to actually put bytes on the wire. (The public `resume()` at `Request.swift:767` handles the reverse ordering, where you resume before the task exists.)

## 3. Validate — deferred closures run at task completion

Validation is **opt-in and deferred**. Calling `.validate()` doesn't check anything immediately — it appends a closure to a `validators` array. `DataRequest.validate(_:)` (`DataRequest.swift:144`):

```swift
public func validate(_ validation: @escaping Validation) -> Self {
    let validator = { [unowned self] in
        guard error == nil, let response else { return }
        let result = validation(request, response, data)
        if case let .failure(error) = result { self.error = ... }   // records failure
        eventMonitor?.request(self, didValidateRequest: ...)
    }
    validators.write { $0.append(validator) }
    return self
}
```

Those stored validators fire in `Request.didCompleteTask` (`Request.swift:515`), **after** the task finishes but **before** `retryOrFinish`:

```swift
let validators = validators.read(\.self)
validators.forEach { $0() }
```

This is the dynamic-dispatch link codegraph flagged: `didCompleteTask → validators handlers`. A validation failure sets the request's `error`, which surfaces in every response handler and can trigger retry.

The built-in `validate()` (no args, `Validation.swift:188`) is sugar for two standard checks:

```swift
validate(statusCode: acceptableStatusCodes)      // default 200..<300
    .validate(contentType: acceptableContentTypes)  // from the Accept header, default */*
```

- **Status code** (`Validation.swift:81`): fails with `.unacceptableStatusCode` if the response code isn't in the range.
- **Content type** (`Validation.swift:105`): parses `response.mimeType` into a `MIMEType`, wildcard-matches it against acceptable types, and fails with `.unacceptableContentType` / `.missingContentType` otherwise (empty bodies pass).

You can also pass custom sequences (`validate(statusCode: 200..<400)`) or a fully custom `Validation` closure. `DataStreamRequest` and `DownloadRequest` have parallel implementations with signatures tailored to their payloads (the download variant reads the file's size off disk to decide emptiness).

## The whole path in one line

`Session.request` (store recipe) → `perform` → `performSetupOperations` → `asURLRequest` (build) + adapt + `URLRequest.validate` → `didCreateURLRequest` → `task(for:)` → `didCreateTask`/`resume` (send) → task completes → `didCompleteTask` runs the deferred `validators` (validate) → `retryOrFinish`.

The elegant part is the deferral: the `URLRequest` is built lazily at send time (so interceptors and modifiers get the last word), and validation is a queue of closures that runs at exactly one point in the lifecycle, feeding the same `error` channel that response serialization and retry policies read from.
