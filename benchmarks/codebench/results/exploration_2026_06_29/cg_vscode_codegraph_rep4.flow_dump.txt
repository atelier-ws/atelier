## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

## How the extension host talks to the main process

VS Code runs extensions in a **separate process** (the extension host) that has no direct access to the workbench (the main/renderer process). All communication happens over an **asynchronous, proxy-based RPC layer** built on a byte-stream message channel. There are four layers.

### 1. The transport: `IMessagePassingProtocol`
At the bottom is a dead-simple duplex channel (`src/vs/base/parts/ipc/common/ipc.ts:99`):

```typescript
export interface IMessagePassingProtocol {
    send(buffer: VSBuffer): void;
    readonly onMessage: Event<VSBuffer>;
    drain?(): Promise<void>;
}
```

It only moves raw `VSBuffer` bytes — it knows nothing about extensions or method calls. The concrete implementation depends on the host type:
- **Local desktop extension host** (`localProcessExtensionHost.ts`) — a Node child process connected via an Electron **MessagePort** (`acquirePort` / `MessagePortExtHostConnection`).
- **Web worker extension host** — a `Worker` MessageChannel.
- **Remote extension host** (`remoteExtensionHost.ts`) — a **socket** (WebSocket/TCP) to a remote server, wrapped by `PersistentProtocol` in `ipc.net.ts`.

### 2. The RPC engine: `RPCProtocol`
`src/vs/workbench/services/extensions/common/rpcProtocol.ts:117` wraps that byte channel and turns it into method calls. Both sides construct one over their end of the protocol. It maintains two registries:
- `_locals` — actor objects that live on this side and can be invoked remotely.
- `_proxies` — JS `Proxy` objects representing actors on the *other* side.

**Outgoing calls** — `getProxy()` returns a `Proxy` whose getter (`_createProxy`, line 251) intercepts any method starting with `$` and routes it to `_remoteCall`:

```typescript
target[name] = (...myArgs: any[]) => {
    return this._remoteCall(rpcId, name, myArgs);
};
```

That's why every cross-process method in VS Code is named with a `$` prefix — it's the marker the proxy uses to know a call must go over the wire. `_remoteCall` serializes the args, assigns a monotonic request id, sends the bytes, and returns a `LazyPromise` (`lazyPromise.ts`) that resolves when the reply comes back.

**Incoming messages** — `_receiveOneMessage` (line 282) decodes the buffer, and for requests dispatches through `_receiveRequest → _invokeHandler`, which looks up the target actor in `_locals` and calls the real method. Replies flow back the same way (`_receiveReply` / `_receiveReplyErr`).

### 3. The wire format: `MessageIO`
Messages are hand-packed binary buffers (not plain JSON envelopes), tagged by a one-byte `MessageType` (line 940): `RequestJSONArgs`, `RequestMixedArgs` (+ `WithCancellation` variants), `Acknowledged`, `Cancel`, `ReplyOKEmpty`/`ReplyOKJSON`/`ReplyOKVSBuffer`/`ReplyOKJSONWithBuffers`, `ReplyErrError`/`ReplyErrEmpty`.

Two serialization strategies (`serializeRequestArguments`, line 731):
- **Simple/JSON args** — when all arguments are plain, everything is one `JSON.stringify`.
- **Mixed args** — used when any argument is a `VSBuffer`, `undefined`, or a `SerializableObjectWithBuffers`. Each argument is encoded individually so binary buffers ride alongside JSON without base64 overhead.

This also handles responsiveness monitoring: a request that isn't acknowledged within `UNRESPONSIVE_TIME` (3s) flips the host into an "unresponsive" state, and cancellation tokens map to `Cancel` messages.

### 4. URI marshalling
Because `URI` objects and file-system semantics differ across local/remote boundaries, an `IURITransformer` (`uriIpc.ts`) rewrites URIs on the way in and out (`transformIncomingURIs` / `transformOutgoing`). The init data sent to a new host is passed through `ExtensionHostMain._transform` (`extensionHostMain.ts:222`) so extension locations, storage paths, etc. are revived correctly for that host's context.

### Bootstrapping
When a host starts, `ExtensionHostMain` (`extensionHostMain.ts:161`) is handed the `IMessagePassingProtocol`, wraps it in a `RPCProtocol`, exposes it as the injectable `IExtHostRpcService`, then bootstraps all the `ExtHost*` services. Extension-facing services obtain typed proxies to the main-thread side (`MainContext.MainThread*`) and register their own actors (`ExtHostContext.ExtHost*`). The main process does the mirror-image setup in `extensionHostManager.ts`. From then on, an API call like `window.showQuickPick` in an extension is just a `$`-method on a proxy that serializes over the channel and runs on the main thread.
