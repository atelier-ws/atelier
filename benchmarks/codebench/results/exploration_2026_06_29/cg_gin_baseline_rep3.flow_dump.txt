## The key insight: middleware and handlers are one flat slice

Gin doesn't keep middleware separate from route handlers. At registration time, everything is flattened into a single `HandlersChain` (a `[]HandlerFunc`) stored on each route.

When you register a route, `routergroup.go:86` calls `combineHandlers`, which concatenates the group's middleware with the route's handlers into one slice:

```go
// routergroup.go:241
mergedHandlers := make(HandlersChain, finalSize)
copy(mergedHandlers, group.Handlers)                          // group middleware first
copy(mergedHandlers[len(group.Handlers):], handlers)          // then route handlers last
```

So a route registered as `r.Use(Logger, Auth)` + `r.GET("/x", Handler)` ends up as a single chain: `[Logger, Auth, Handler]`. This merged chain is stored in the radix tree (`addRoute`) keyed by method + path.

## Per-request: match, then walk the chain

1. **`ServeHTTP` (gin.go:662)** grabs a pooled `*Context`, resets it, and calls `handleHTTPRequest`.

2. **`handleHTTPRequest` (gin.go:690)** selects the tree for the HTTP method, then `root.getValue(rPath, ...)` does the radix-tree lookup. On a match it copies the pre-merged chain onto the context and kicks it off:

   ```go
   // gin.go:719
   c.handlers = value.handlers   // the flattened [Logger, Auth, Handler]
   c.fullPath = value.fullPath
   c.Next()
   ```

3. **`Context.Next()` (context.go:188)** is the engine of the chain. It just advances an index and invokes each handler in order:

   ```go
   func (c *Context) Next() {
       c.index++
       for c.index < safeInt8(len(c.handlers)) {
           if c.handlers[c.index] != nil {
               c.handlers[c.index](c)
           }
           c.index++
       }
   }
   ```

## Why the "before/after" middleware pattern works

The index (`c.index int8`) lives on the context, and `Next()` is re-entrant. A middleware that calls `c.Next()` itself triggers the inner `for` loop to run *all subsequent* handlers before returning to it. That's what gives you the onion model:

```go
func Logger(c *gin.Context) {
    start := time.Now()
    c.Next()                    // run the rest of the chain
    log.Print(time.Since(start)) // resumes here after Handler returns
}
```

Code before `c.Next()` runs on the way in; code after runs on the way out. A middleware that *doesn't* call `Next()` simply returns, and the outer loop in the original `Next()` call advances to the next handler anyway — so calling `Next()` is only required when you need post-processing logic.

## Aborting

`Abort()` (context.go:207) sets `c.index = abortIndex` (`math.MaxInt8 >> 1 = 63`). Since the loop condition is `c.index < len(c.handlers)`, jumping the index past the chain length stops any pending handlers from running — though the *current* handler finishes. `IsAborted()` checks `c.index >= abortIndex`. (This also caps a chain at 63 handlers, asserted in `combineHandlers`.)

## Summary of the flow

```
Register: group middleware + route handlers ──combineHandlers──> one HandlersChain ──> radix tree
Request:  ServeHTTP ──> handleHTTPRequest ──> tree lookup ──> c.handlers = chain ──> c.Next()
          c.Next() walks index 0..n, each middleware optionally calling Next() to nest
          Abort() jumps index to 63 to short-circuit
```

The elegance is that there's no separate middleware dispatcher — routing produces a flat slice, and a single index-walking `Next()` loop handles both middleware and the final handler uniformly.
