## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

[assistant] The extension host talks to the main/renderer process through a **layered RPC-over-IPC stack**. Here's how it works, from the wire up:

## 1. Transport layer — a byte pipe between two OS processes

The extension host runs as a **separate process** (a Node.js utility/child process for desktop; a Web Worker for web; a socket connection for remote). At the bottom is `IMessagePassingProtocol` (`src/vs/base/parts/ipc/common/ipc.ts`), a minimal duplex interface: `send(VSBuffer)` + `onMessage: Event<VSBuffer>`.

Concretely that's backed by:
- A **socket** (named pipe / TCP) — `connect()` in `src/vs/base/parts/ipc/node/ipc.net.ts` wraps it in a `NodeSocket`, and a `PersistentProtocol` adds message framing plus a replay buffer for reconnection.
- Messages crossing the process boundary are wrapped as `IExtHostSocketMessage` (`src/vs/workbench/services/extensions/common/extensionHostProtocol.ts:111`).

## 2. RPC layer — `RPCProtocol`

`RPCProtocol` (`src/vs/workbench/services/extensions/common/rpcProtocol.ts:117`) sits on top of that byte pipe and turns it into typed method calls. It's constructed with the protocol and hooks its receive loop at line 161:

```ts
this._register(this._protocol.onMessage((msg) => this._receiveOneMessage(msg)));
```

Both sides run one. Communication is organized around **proxy identifiers**:
- `MainContext.*` — services that live in the **renderer/main-thread** side (e.g. `MainThreadErrors`, `MainThreadExtensionService`).
- `ExtHostContext.*` — services that live in the **extension host**.

Each side:
- **`getProxy(identifier)`** (`:243`) returns a JS `Proxy`. Any `$`-prefixed method you call on it is trapped and turned into `_remoteCall(rpcId, name, args)` — i.e. a serialized message sent across the wire (`:251-266`). This is why every cross-process API method is named `$something`.
- **`set(identifier, impl)`** (`:268`) registers the *local* object that actually handles incoming calls for that identifier.

## 3. Message format & flow

`_receiveOneMessage` (`:282`) decodes a binary frame: a `MessageType` byte + a request id, then dispatches. Message types include `RequestJSONArgs` / `RequestMixedArgs` (with/without cancellation), `Acknowledged`, `Cancel`, and the reply variants `ReplyOKEmpty/JSON/JSONWithBuffers/VSBuffer` and `ReplyErrError/Empty`.

The request/response cycle (`_receiveRequest`, `:360`):
1. Deserialize args (JSON + raw `VSBuffer` refs), invoke the local handler.
2. Immediately **acknowledge** the request (`:380-382`) — used for liveness.
3. When the handler resolves, send `ReplyOK`; on throw, send `ReplyErr`.

Replies are matched back to `_pendingRPCReplies` by request id. Unacknowledged requests drive **responsiveness detection**: `_onWillSendRequest` / `_checkUnresponsive` flip the host to `Unresponsive` after 3s (`:121`, `:184-221`), which is how VS Code detects a hung extension host.

Cross-cutting concerns handled here:
- **URI marshalling** — `IURITransformer` rewrites `URI` objects on the way out/in (`:85`, `createURIReplacer`), essential for remote scenarios where paths differ between machines.
- **Binary payloads** — `VSBuffer`s are extracted from the JSON via `$$ref$$` placeholders (`stringifyJsonWithBufferRefs`, `:42`) so binary data isn't base64'd through JSON.

## 4. Bootstrapping

On the ext-host side, `ExtensionHostMain` (`src/vs/workbench/api/common/extensionHostMain.ts:161`) is handed the raw `protocol`, wraps it in an `RPCProtocol`, and exposes it as `ExtHostRpcService` in the DI container (`:176`, `:184`). Extension-host services then grab proxies like:

```ts
const mainThreadErrors = rpcService.getProxy(MainContext.MainThreadErrors);
mainThreadErrors.$onUnexpectedError(data);   // → serialized → renderer
```

On the renderer side, `ExtensionHostManager` constructs the mirror `RPCProtocol` and registers all the `MainThread*` handlers.
