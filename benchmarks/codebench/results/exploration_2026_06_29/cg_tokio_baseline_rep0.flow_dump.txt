## The big picture

A tokio task is a `Future` that's been spawned. The runtime's job is to **poll** these futures: call `Future::poll`, let them run until they hit an `.await` that isn't ready, and then set them aside until whatever they're waiting on becomes ready. Tokio never "blocks" on a task — it polls, and a not-ready task yields control back so the thread can run something else.

There are two schedulers: **multi-thread** (the default) and **current-thread** (single-threaded). The interesting machinery is in multi-thread.

## Workers and cores

The multi-thread scheduler starts a fixed pool of **worker threads**. Each worker owns a `Core` (`worker.rs:107`) containing its **local run queue**, a special **LIFO slot**, a tick counter, and stats. The split between `Worker` and `Core` exists so the core can be *handed off* to another thread during `block_in_place`.

Tasks live in several queues:
- **LIFO slot** — a single-task slot checked *before* the run queue (`worker.rs:114-120`)
- **Local run queue** — a bounded, lock-free ring buffer per worker (`queue.rs`)
- **Global/inject queue** — a shared MPMC queue (`inject.rs`) for tasks scheduled from outside a worker, or overflow

## The run loop

Each worker spins in `Context::run` (`worker.rs:561`). Every iteration it:

1. **Increments a tick** and runs `maintenance` periodically (every `event_interval` ticks it polls the I/O/timer driver — `worker.rs:780`).
2. **Picks the next task** via `next_task` (`worker.rs:1062`):
   - Every `global_queue_interval` ticks, it checks the **global queue first** (`worker.rs:1063`). This is deliberate fairness — it prevents the local queue from starving globally-injected tasks.
   - Otherwise it takes a **local task** (`next_local_task`: LIFO slot first, then run queue — `worker.rs:1133`).
   - If the local queue is empty, it pulls a *batch* from the global queue (taking roughly `inject.len() / num_workers + 1`, so other workers still get some — `worker.rs:1106`).
3. If no work is found locally, it tries to **steal** (`steal_work`, `worker.rs:1141`).
4. If stealing finds nothing, it **parks** the thread (`worker.rs:810`) — sleeping on the I/O driver until a wakeup.

## Running a task and the LIFO optimization

`run_task` (`worker.rs:630`) wraps the poll in a **cooperative budget** (`coop::budget`) and calls `task.run()`, which invokes `Future::poll`.

After polling, it checks the **LIFO slot in a loop** (`worker.rs:706`). When a running task wakes another task (the classic message-passing pattern where A sends to B), B goes into the LIFO slot and gets run *immediately* next — great for cache locality and latency. To prevent two tasks from ping-ponging and starving everyone else, the LIFO slot is capped at `MAX_LIFO_POLLS_PER_TICK = 3` (`worker.rs:264`, `743`), after which it's disabled and the task goes to the back of the run queue.

## How a wakeup becomes a scheduled task

When a task's waker is invoked (e.g. I/O is ready, a channel got a message), `schedule_task` runs (`worker.rs:1330`):
- If scheduling **from within the same worker**, it goes local via `schedule_local` (`worker.rs:1355`) — into the LIFO slot (unless it's a `yield_now`, which forces back-of-queue for fairness).
- If scheduling **from another thread / outside the runtime**, it's pushed to the **global inject queue** and a parked worker is notified (`worker.rs:1340`, `push_remote_task`).

## Work-stealing and keeping workers busy

`steal_work` (`worker.rs:1141`) starts at a **random** worker and steals roughly *half* of a victim's run queue. Two throttles keep this from thundering:
- A worker only steals if it can transition into the **searching** state, and tokio limits searchers to roughly half the workers (`worker.rs:1136` note) so they don't all stampede.
- When a worker finds work while searching, it notifies *another* idle worker to come look (`should_notify_others`, `worker.rs:1192`), propagating wake-ups gradually.

If stealing fails everywhere, it falls back to the global queue, and only then parks.

## Cooperative scheduling (budget)

Every poll runs under a **budget** (`task::coop`). Tokio's own leaf resources (I/O, channels, timers) decrement this budget on each ready operation. When it hits zero, they start returning `Pending` *even though data is available*, forcing the task to yield. This prevents a single busy task (e.g. a tight loop reading a always-ready socket) from monopolizing a worker and starving others — fairness without preemption.

---

**In short:** workers run a poll loop pulling from a LIFO slot → local queue → (periodically) global queue → stealing → parking. Wakeups re-enqueue tasks locally (LIFO for locality) or globally (for cross-thread). Work-stealing balances load, periodic global-queue checks and the LIFO cap provide fairness, and a per-poll cooperative budget keeps any one task from hogging a thread.
