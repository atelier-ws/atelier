## Overview

Alamofire is built around a `Session` that owns a `URLSession`, plus a `Request` object that wraps the lifecycle. Everything flows through a few serial `DispatchQueue`s for thread safety: `rootQueue` (all internal state + delegate callbacks), `requestQueue` (building `URLRequest`s), and `serializationQueue` (parsing responses). The pipeline is: **build → adapt → send → validate → serialize**.

---

## 1. Build

**Entry** (`Session.swift`): the convenience `request(_:method:parameters:...)` methods don't build the `URLRequest` directly. They wrap your inputs in a small `URLRequestConvertible` struct and create a `DataRequest`:

- `RequestConvertible` — for dictionary parameters + a `ParameterEncoding` (e.g. `URLEncoding`, `JSONEncoding`).
- `RequestEncodableConvertible` — for `Encodable` parameters + a `ParameterEncoder` (e.g. `JSONParameterEncoder`, `URLEncodedFormParameterEncoder`).

Both implement `asURLRequest()`, which is the actual build step and runs *lazily* later, on `requestQueue`:

```swift
func asURLRequest() throws -> URLRequest {
    var request = try URLRequest(url: url, method: method, headers: headers)
    try requestModifier?(&request)              // your inout hook to tweak the request
    return try encoding.encode(request, with: parameters)  // params → query string or body
}
```

So the order is: URL + method + headers → `requestModifier` → parameter encoding. `URLEncoding` decides query-string vs. body based on HTTP method (`Destination.methodDependent`); `JSONEncoding`/`JSONParameterEncoder` serialize a JSON body and set `Content-Type`.

The `DataRequest` starts in state `.initialized`. Nothing is built or sent yet — it's registered with the session via `performEagerlyIfNecessary()`, which only proceeds immediately if setup is `.eager`; otherwise it waits for `resume()` or a response handler to be attached.

## 2. Send

When the request is performed (`Session.perform`), it's added to `activeRequests` on `rootQueue`, then dispatched to `requestQueue`, which calls `performSetupOperations(for:convertible:)`:

1. `convertible.asURLRequest()` builds the request (step 1 above), then `initialRequest.validate()` sanity-checks it.
2. **Adapt** — if a `RequestInterceptor`/`RequestAdapter` is configured (session-level and request-level adapters are composed), `adapter.adapt(initialRequest, using:state:completion:)` runs asynchronously to mutate the request before it goes out (auth headers, etc.). The adapted request is validated again.
3. Back on `rootQueue`, `didCreateURLRequest` calls the polymorphic `request.task(for:using:)`. For `DataRequest` that's just `session.dataTask(with:)`. The task is stored in `requestTaskMap` (Request ↔ task).
4. `didCreateTask` syncs the task to the request's state — if the request is already `.resumed`, it calls `task.resume()` immediately; `.suspended`/`.cancelled` are handled similarly.

The `Request.State` machine is `initialized → resumed/suspended → cancelled/finished`. `resume()` flips the state and either resumes the live task or, if no task exists yet, triggers `Session.perform` via the delegate.

As the response arrives, `SessionDelegate` (the `URLSessionDelegate`) fires on `rootQueue`: `didReceive(data:)` appends incoming bytes into the request's accumulated `data`, and finally `didCompleteWithError` routes to `request.didCompleteTask(task:with:)`.

## 3. Validate

Validation is opt-in via `.validate()` / `.validate(statusCode:)` / `.validate(contentType:)`. Each appends a closure to a thread-safe `validators` array. A validator is `(URLRequest?, HTTPURLResponse, Data?) -> ValidationResult` (`Result<Void, Error>`). Default `.validate()` accepts status codes 200–299 and content types matching the `Accept` header.

Crucially, validators don't run at send time — they run **when the task completes**, synchronously, in `didCompleteTask`:

```swift
mutableState.write { $0.error = $0.error ?? error }
validators.read(\.self).forEach { $0() }   // sets self.error on failure
retryOrFinish(error: self.error)
```

A failing validator sets `self.error` to an `AFError.responseValidationFailed` (e.g. `.unacceptableStatusCode`, `.unacceptableContentType`). Then `retryOrFinish` consults any `RequestRetrier`: on `.retry`, the request is reset (`prepareForRetry`) and re-enters `perform(...)`; otherwise `finish()` is called and response serialization begins.

## 4. Serialize (response)

Serializers conform to `ResponseSerializer` and are attached via `.responseData()`, `.responseString()`, `.responseDecodable(of:)`, etc. Each is enqueued and runs on `serializationQueue` after `finish()`. The serializer's `serialize(request:response:data:error:)` receives the **error from validation** — so validation failures short-circuit serialization and surface as the response's error.

The result is wrapped into an `AFDataResponse<T>` (`DataResponse<T, AFError>`) carrying the request, response, raw data, metrics, serialization duration, and a `Result<T, AFError>`. The completion handler is finally called on your chosen queue (default `.main`).

## End-to-end

```
request(...)  → DataRequest(.initialized), wrapped in URLRequestConvertible
   ↓ resume()/eager
perform → requestQueue → asURLRequest() (build: url+method+headers → modifier → encode)
   ↓
RequestAdapter.adapt() (optional)  → validate URLRequest
   ↓
session.dataTask(...) created, mapped, resumed
   ↓  (SessionDelegate on rootQueue)
didReceive(data:) accumulates → didCompleteWithError
   ↓
validators run synchronously → set AFError on failure
   ↓
retryOrFinish → (RequestRetrier) retry or finish()
   ↓  (serializationQueue)
serializer.serialize(..., error:) → DataResponse<T> with Result
   ↓
completionHandler on .main
```

Throughout, an `EventMonitor` receives hooks at each step (`didCreateInitialURLRequest`, `didAdaptInitialRequest`, `didCreateTask`, `didReceiveData`, `didValidate`, `didParseResponse`, `requestDidFinish`), which is how logging and metrics plug in.
