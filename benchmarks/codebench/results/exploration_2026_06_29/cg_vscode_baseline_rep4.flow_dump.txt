## How the extension host talks to the main/renderer process in VS Code

Extensions never run in the renderer (UI) process. They run in a **separate extension host**, and all communication happens over an **asynchronous, message-passing RPC protocol**. There is no shared memory and no synchronous access to the DOM — every cross-boundary call is a serialized message and returns a `Promise`.

### The layering (top to bottom)

**1. Typed RPC proxies — `proxyIdentifier.ts` / `rpcProtocol.ts`**

Each side gets typed proxy objects rather than calling each other directly. Two registries of named actors are defined in `extHost.protocol.ts`:
- `MainContext.*` — services living in the renderer that extensions call *into* (e.g. `MainThreadDocuments`).
- `ExtHostContext.*` — objects in the extension host the renderer calls *into* (e.g. `ExtHostDocuments`).

Each is a `ProxyIdentifier` with a unique numeric id (`nid`). You get a proxy with `rpcProtocol.getProxy(identifier)` and register a local implementation with `rpcProtocol.set(identifier, instance)` (`proxyIdentifier.ts:9-44`). The `Proxied<T>` mapped type rewrites every method so its return becomes a `Promise<Dto<...>>` — making it impossible to forget that calls are async (`proxyIdentifier.ts:69`).

**2. `RPCProtocol` — the marshaling engine (`rpcProtocol.ts`)**

This turns a proxy method call into a wire message and back:
- `_remoteCall` (line 461) assigns an incrementing request id, serializes arguments, registers a `LazyPromise` in `_pendingRPCReplies`, and sends the message. The reply (`_receiveReply`, line 403) or error (`_receiveReplyErr`, line 416) resolves that promise by matching the request id.
- The wire format is a hand-rolled binary `MessageBuffer` (line 516): a 1-byte `MessageType`, a 4-byte big-endian request id, then a "mixed array" of arguments tagged by `ArgType` (String / VSBuffer / SerializedObjectWithBuffers / Undefined). Plain args go through JSON; `VSBuffer`s are passed by reference so binary data isn't base64-bloated (`stringifyJsonWithBufferRefs`, line 42).
- `CancellationToken` is special-cased: if it's the last arg, cancellation sends a separate `Cancel` message (line 482-488).
- It also tracks **responsiveness**: every request expects an `Acknowledged` message; if none arrives within 3s, it fires `onDidChangeResponsiveState` → Unresponsive (lines 121, 184-221). That's what powers the "Extension host unresponsive" warnings.
- `IURITransformer` rewrites `URI` objects across the boundary (important for remote/web where paths differ on each side).

**3. `IMessagePassingProtocol` — the transport abstraction (`base/parts/ipc/common/ipc.ts`)**

`RPCProtocol` doesn't know about sockets at all. It takes an `IMessagePassingProtocol` in its constructor (`rpcProtocol.ts:141`) — just `{ onMessage: Event<VSBuffer>, send(VSBuffer) }`. This is the seam that lets the same RPC engine run over completely different underlying channels.

### The actual transport depends on the host kind

There are three flavors of extension host, each providing a different `IMessagePassingProtocol`:

- **Local process** (`electron-browser/localProcessExtensionHost.ts`) — a real Node.js child process (`extensionHostProcess.ts`). Connection is established over an **Electron `MessagePort`** acquired from the shared-process worker (`_establishProtocol`, line 397-447). `MessagePortExtHostConnection` is written into the child's env; the protocol is just `port.onmessage → onMessage.fire(VSBuffer.wrap(...))` and `send → port.postMessage(buffer)` (lines 418-429). Older builds used a Node named pipe / socket with `PersistentProtocol`; current local hosts use MessagePort.

- **Web worker** (`browser/webWorkerExtensionHost.ts`) — extensions run in a `Worker` hosted inside a (cross-origin, for isolation) iframe. Transport is the browser `MessagePort` / `postMessage` (`webWorkerExtensionHostIframe.html`).

- **Remote** (remote-server extension host) — runs on a remote machine; transport is a real network socket wrapped in `PersistentProtocol`, which buffers and replays unacknowledged messages so the connection survives brief network drops.

### Startup handshake

Before any RPC flows, there's a fixed handshake (`_performHandshake`, line 450-502), using the plain `MessageType` control messages rather than RPC:
1. Renderer waits for `MessageType.Ready` from the host.
2. Renderer sends `IExtensionHostInitData` (JSON) — environment, extensions to activate, etc. (line 475-481).
3. Host replies `MessageType.Initialized`; only then is the `RPCProtocol` wired up and proxies become usable.

### End-to-end flow of one call

```
extension calls vscode.workspace.openTextDocument(uri)
  → ExtHostDocuments calls its MainThreadDocuments *proxy*
    → RPCProtocol._remoteCall: serialize args, assign req id, send VSBuffer
      → IMessagePassingProtocol.send  →  [MessagePort / socket / worker]
        → other side RPCProtocol._receiveOneMessage → _invokeHandler
          → real MainThreadDocuments method runs in the renderer
        ← reply message (matched by req id) resolves the LazyPromise
  ← extension's await resolves
```

**The one-line summary:** typed proxies (`MainContext`/`ExtHostContext`) → `RPCProtocol` marshals each call to a binary `VSBuffer` message with a request id → an `IMessagePassingProtocol` carries those buffers over the actual channel (Electron MessagePort for local, browser MessagePort for web worker, network socket for remote) → the far side dispatches to the registered implementation and sends a reply matched back by request id. Everything is async, JSON+VSBuffer serialized, and URI/cancellation-aware.
