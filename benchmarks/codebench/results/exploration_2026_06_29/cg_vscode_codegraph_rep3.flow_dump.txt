## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

[assistant] The extension host and main process (renderer) talk over a layered **message-passing → RPC-proxy** stack. Here's how it fits together.

## The transport layer

At the bottom is a raw `IMessagePassingProtocol` (`src/vs/base/parts/ipc/common/ipc.ts`) — a bidirectional byte-stream that just does `send(VSBuffer)` / `onMessage`. The concrete transport depends on where the extension host runs:

- **Local desktop ext host** (`localProcessExtensionHost.ts`) — a Node child process connected via an Electron **MessagePort** (`acquirePort`, `MessagePortExtHostConnection`).
- **Remote ext host** (`remoteExtensionHost.ts`) — a **socket** to the remote server.
- **Web worker ext host** — a `Worker` `postMessage` channel.

Bootstrapping this connection uses the tiny control messages in `extensionHostProtocol.ts` — e.g. `VSCODE_EXTHOST_IPC_READY`, `VSCODE_EXTHOST_IPC_SOCKET`, and the `MessageType.{Initialized,Ready,Terminate}` handshake bytes (`createMessageOfType`/`isMessageOfType`). Once the socket/port is live, the main side ships over `IExtensionHostInitData` (environment, workspace, extension descriptions, telemetry ids, etc.).

## The RPC layer

Both sides wrap that raw protocol in an **`RPCProtocol`** (`src/vs/workbench/services/extensions/common/rpcProtocol.ts`). This is the actual RPC engine, and it's symmetric — it works identically in both directions. In the ext host it's created in `ExtensionHostMain`'s constructor:

```ts
this._rpcProtocol = new RPCProtocol(protocol, null, uriTransformer);
```

`RPCProtocol` gives you two things:

1. **`getProxy(identifier)`** — returns a lazy JS `Proxy`. Calling a method like `proxy.$asBrowserUri(uri)` gets serialized into a request buffer and sent over the wire; you get back a `Promise` that resolves when the reply arrives. See `ExtensionHostMain.asBrowserUri` for a live example.
2. **`set(identifier, instance)`** — registers a local object so the *other* side can call into it.

The named endpoints are **`ProxyIdentifier`s** (`proxyIdentifier.ts`), organized into two big registries in `extHost.protocol.ts`:
- **`MainContext`** — services living in the renderer/main process (e.g. `MainThreadExtensionService`), called *from* the ext host.
- **`ExtHostContext`** — objects living in the ext host, called *from* the main process.

## Wire format & message loop

`RPCProtocol._receiveOneMessage` is the dispatch heart. Each message begins with a 1-byte `MessageType` + a 4-byte request id, then the payload:

- **Requests**: `RequestJSONArgs` / `RequestMixedArgs` (plus `...WithCancellation` variants). "Mixed" args allow `VSBuffer` and `SerializableObjectWithBuffers` to be sent efficiently rather than JSON-stringified. A request routes through `_receiveRequest` → `_invokeHandler` to call the real local method.
- **Replies**: `ReplyOKEmpty`, `ReplyOKJSON`, or error replies, matched back to the pending promise (`LazyPromise`) by request id.
- **Control**: `Acknowledged` (used for responsiveness detection — if a peer goes silent past `UNRESPONSIVE_TIME` = 3s, it's flagged unresponsive) and `Cancel`.

`CancellationToken` and `URI` are first-class: a `uriTransformer` rewrites URIs crossing the process boundary (important for remote, where file paths differ), via `transformIncomingURIs` on the way in and the symmetric transform on the way out.

## Putting it together

```
Extension code
   │  (typed API, awaits a Promise)
ExtHostRpcService  ── getProxy(MainContext.X) ──┐
   │                                            │
RPCProtocol  ──serialize→ VSBuffer──────────────┤   MessageType + reqId + args
   │                                            │
IMessagePassingProtocol  (MessagePort / socket / worker.postMessage)
   │                                            │
RPCProtocol (main side)  ──_receiveOneMessage──→ _invokeHandler → MainThreadX
```

So: **extensions never touch IPC directly.** They call typed proxy methods; `RPCProtocol` marshals those into length-prefixed buffers over whatever `IMessagePassingProtocol` the host type provides, and dispatches replies back to the awaiting promises. The design is fully symmetric, which is why the main thread can just as easily call into `ExtHostContext` objects (e.g. to fire activation events or push config changes).
