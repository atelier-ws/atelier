**The Main Worker Loop (lines 561-607):**
```rust
fn run(&self, mut core: Box<Core>) -> RunResult {
    while !core.is_shutdown {
        // Increment tick
        core.tick();
        
        // Run maintenance if needed
        core = self.maintenance(core);
        
        // Try local queue first
        if let Some(task) = core.next_task(&self.worker) {
            core = self.run_task(task, core)?;
            continue;
        }
        
        // No local work, try stealing
        if let Some(task) = core.steal_work(&self.worker) {
            core = self.run_task(task, core)?;
        } else {
            // No work found, park
            core = self.park(core);
        }
    }
}
```

**Tick and Maintenance (lines 577-581, 780-797):**
- `core.tick()` increments the tick counter (line 1058)

## The big picture

Tokio has two schedulers behind a common interface (`tokio/src/runtime/scheduler/mod.rs`):

- **Multi-thread** (`scheduler/multi_thread/`) — the default; a pool of worker threads with per-worker run queues, work-stealing, and a shared injection queue.
- **Current-thread** (`scheduler/current_thread/`) — a single thread driving a `VecDeque` of tasks.

A task is a heap-allocated future plus bookkeeping; "scheduling" means deciding which run queue it lands on, and "running" means a worker popping it and polling it once.

## What a task actually is

Each spawned future becomes a `Cell<T, S>` (`task/core.rs:126`), laid out as three contiguous, cache-aligned pieces:

- **`Header`** — atomic `State`, the `vtable`, and an intrusive `queue_next` pointer so the task *is* its own linked-list node in run queues (no separate allocation per enqueue).
- **`Core`** — the scheduler handle, task id, and the future-or-output union (`CoreStage`).
- **`Trailer`** — the `JoinHandle`'s waker and owned-list pointers.

All polymorphism goes through a `Vtable` (`task/raw.rs:24`) with `poll`, `schedule`, `dealloc`, etc. The runtime moves tasks around as type-erased `RawTask`/`Notified` handles and only recovers the concrete type inside the typed `Harness<T, S>`.

The lifecycle is encoded in atomic state bits (`task/state.rs:17`): `RUNNING` (a lock — only one thread polls at a time), `COMPLETE`, `NOTIFIED` (sitting in a run queue), `CANCELLED`, plus a reference count and join-handle bits.

## Spawning

`spawn` → `Handle::spawn` → `bind_new_task` (`multi_thread/handle.rs:55`):

1. `owned.bind(...)` allocates the task, registers it in the runtime's owned-task list, and hands back a `JoinHandle` + a `Notified` (the "this needs to run" token).
2. The `Notified` is scheduled immediately.

## The schedule path (where a task lands)

`schedule_task` (`multi_thread/handle.rs:1327`) decides placement based on *where the wake came from*:

- **From inside a worker on this runtime** → `schedule_local`: push to that worker's own local queue.
- **From anywhere else** (another runtime thread, a foreign thread, `block_on`) → `push_remote_task` pushes onto the shared **injection queue** and notifies a parked worker.

This is also exactly the waker path. A wakeup goes through the waker vtable (`task/waker.rs:118`) → `wake_by_val`/`wake_by_ref` → state transitions to `NOTIFIED` → the scheduler's `Schedule::schedule()` (`handle.rs:105`) → the same `schedule_task` above. So "waking a task" and "scheduling a freshly spawned task" are the same mechanism.

### The three queue tiers

1. **LIFO slot** — a single `Option<Notified>` per worker (`worker.rs:120`). `schedule_local` puts the most-recently-woken task here instead of the queue. This optimizes the common message-passing handoff (task A wakes task B and yields; B runs next on the warm cache).
2. **Local run queue** — a lock-free fixed-size ring buffer, `LOCAL_QUEUE_CAPACITY = 256` (`queue.rs:63`). Single producer (the owning worker), multiple stealers via CAS.
3. **Injection / global queue** — an MPMC growable queue (`inject.rs`) for remote spawns and local-queue overflow. When the local queue fills, `push_back_or_overflow` (`queue.rs:182`) moves ~half (128 tasks) into the injection queue.

## The worker loop (how tasks run)

Each worker thread runs `Worker::run` (`worker.rs:561`) until shutdown:

```
loop {
    core.tick();                 // increment tick counter
    core = maintenance(core);    // periodically poll the I/O driver
    if let Some(task) = core.next_task(worker) { run_task(task); continue; }
    if let Some(task) = core.steal_work(worker) { run_task(task); }
    else { core = park(core); }  // nothing to do — sleep
}
```

**Task selection** — `next_task` (`worker.rs:1062`): every `global_queue_interval` ticks it pulls from the **global queue first** (fairness — stops local work from starving remote/injected tasks); otherwise it drains the **LIFO slot then the local ring buffer**. The interval is dynamically tuned from runtime stats rather than a fixed value.

**Work-stealing** — `steal_work` (`worker.rs:1141`): when a worker's own queues are empty, it transitions to "searching", picks a random victim worker, and steals ~half its queue via `steal_into` (`queue.rs:416`). To avoid a thundering herd, the idle subsystem (`idle.rs`) caps concurrent searchers at ~50% of workers; the last searcher to find work wakes one parked worker. If no peer has work, it falls back to the global queue, then parks.

**Parking** — `park` (`worker.rs:810`, `park.rs`) is a small state machine (`EMPTY → PARKED_DRIVER/CONDVAR → NOTIFIED`). One parked worker holds the I/O driver lock and blocks on epoll/kqueue; others sleep on a condvar. An I/O event or a `notify_parked` wakes a worker, which loops back to find the now-`NOTIFIED` task.

## Running a single task — poll, budget, and the LIFO loop

`run_task` (`worker.rs:630`) wraps the poll in a **cooperative budget**: `coop::budget(|| task.run())`. Each poll session gets **128 units** (`task/coop/mod.rs:116`); budget-aware leaf futures (channels, I/O, timers) call `poll_proceed`, which decrements the counter and forces `Poll::Pending` once it hits zero (`coop/mod.rs:291`). This stops a hot future from monopolizing the worker and starving everything else — even if the future is always ready, it yields after ~128 operations.

`task.run()` → `RawTask::poll` → `Harness::poll` (`harness.rs:153`):

1. `transition_to_running()` acquires the `RUNNING` lock and clears `NOTIFIED`.
2. Build a `Waker` over the task header and poll the future once with it.
3. If it returns `Ready` → `transition_to_complete`, store the output, wake the `JoinHandle`.
4. If `Pending` → `transition_to_idle`. If the task got re-notified *during* its own poll, it's immediately runnable again.

That re-notified case feeds the **LIFO loop** (`worker.rs:689`): the worker keeps polling the same task from the LIFO slot up to `MAX_LIFO_POLLS_PER_TICK = 3` times (`worker.rs:264`), as long as coop budget remains. Past that limit it disables the LIFO slot and pushes the task to the back of the run queue — preventing two tasks from ping-ponging and starving the rest. The LIFO slot is re-enabled at the top of each tick.

## Maintenance / I/O integration

Every `event_interval` ticks (default **61**, `config.rs:13`), `maintenance` (`worker.rs:780`) does a non-blocking `park_yield` — it polls the I/O and timer drivers with a zero timeout. That converts ready I/O events into wakeups (re-scheduling the corresponding tasks), interleaved with task execution so I/O readiness never waits for the pool to go fully idle.

## Current-thread, briefly

`current_thread/mod.rs` is the same idea with the concurrency removed: a `VecDeque` of tasks (`Core`, line 61), the same tick/`global_queue_interval` fairness (default 31), the same coop budget — but no ring buffer, no stealing, no parking coordination. The runtime thread itself drains the queue and drives the driver.
