## Key Insights

1. **Laziness**: Filter operations only mutate the Query object; they don't touch the database until `__iter__()`, `_fetch_all()`, or similar materialize methods are called.

2. **Query → WHERE Tree Path**: 
   - Q objects → QuerySet.filter() → Query.add_q() 
   - → Query._add_q() recursively processes Q tree
   - → Query.build_filter() converts tuples to Lookups
   - → WhereNode tree built with Lookup leaves and WhereNode parents

3. **WHERE to SQL**: 
   - WhereNode.as_sql() recursively compiles children via SQLCompiler.compile()
   - Leaf Lookups call their own as_sql()
   - Connectors (AND/OR/XOR) join child SQL fragments

4. **Three Nested Iterators** (as per comment in QuerySet.__iter__):
   - cursor_iter() yields row chunks (100 at a time)
   - results_iter() applies converters and yields individual rows
   - ModelIterable.__iter__() converts tuples to model instances

5. **Result Caching**: Once _fetch_all() populates _result_cache, all subsequent iterations use the cached list, never hitting the database again.

## The three nested iterators that turn rows into objects

This is the elegant part — results stream through three layers (so a large query needn't load every row at once):

1. **`cursor_iter`** (`compiler.py:2258`) — yields raw row chunks via `cursor.fetchmany(itersize)`.
2. **`results_iter`** (`compiler.py:1565`) — flattens chunks and applies backend **type converters** (e.g. DB string → Python `datetime`).
3. **`ModelIterable.__iter__`** — calls `model_cls.from_db(...)` per row to build the actual model instance, attaching select-related objects and annotations.

That final iterator is what `list(...)` in `_fetch_all` consumes into `_result_cache`.
