## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

## The three phases

### 1. Build

Everything starts at `Session.request(...)` (`Source/Core/Session.swift:367`). The convenience overload wraps your URL, method, parameters, encoder, and headers into a `RequestEncodableConvertible` (a `URLRequestConvertible`), then hands off to the core `request(_:interceptor:)` overload (`:393`), which constructs a `DataRequest` and calls `performEagerlyIfNecessary`.

The actual `URLRequest` isn't built until the request is *performed*. `Session.perform` (`:1166`) dispatches by subtype (`UploadRequest`/`DataRequest`/`DownloadRequest`/…) and ultimately reaches `performSetupOperations` (`:1246`), which is where building happens:

```swift
initialRequest = try convertible.asURLRequest()   // parameters → URLRequest
try initialRequest.validate()                       // sanity-check the URLRequest itself
```

`asURLRequest()` on `RequestEncodableConvertible` applies the method/headers/`requestModifier` and runs your `ParameterEncoder` — e.g. `JSONParameterEncoder.encode` (`Source/Core/ParameterEncoder.swift:75`) serializes `Encodable` params into `httpBody` and sets `Content-Type: application/json`. Encoding failures become `AFError.parameterEncodingFailed`.

### 2. Send

Still in `performSetupOperations`, the built request runs through any interceptor's **adapter** before becoming a task:

```swift
adapter.adapt(initialRequest, using: adapterState) { result in
    let adaptedRequest = try result.get()
    try adaptedRequest.validate()
    ...
    self.didCreateURLRequest(adaptedRequest, for: request)
}
```

`adapt` is dynamically dispatched to whatever `RequestInterceptor` you supplied (16 implementations exist; e.g. `Interceptor.adapt` chains multiple adapters sequentially, `DeflateRequestCompressor`, `Adapter`, etc.). If no adapter, the initial request is used directly.

`Session.didCreateURLRequest` (`:1291`) then:
1. notifies the request (`request.didCreateURLRequest` → event monitors, cURL handlers),
2. creates the actual `URLSessionTask` via `request.task(for:using:)` — overridden per subclass (`DataRequest`/`DownloadRequest`),
3. records it in the `requestTaskMap` and calls `didCreateTask`.

The task is only *resumed* when `Request.resume()` (`Source/Core/Request.swift:768`) runs `task.resume()`. Note resumption is often automatic: adding a response handler via `appendResponseSerializer` (`:589`) calls `resume()` if `shouldAutomaticallyResume`/`startImmediately` is true — this is why you rarely call `.resume()` manually.

### 3. Validate & serialize

You attach validation with `DataRequest.validate(...)` (`Source/Features/Validation.swift`). Rather than running immediately, each validator is **stored as a closure**. The no-arg `validate()` (`:188`) composes the two built-in checks:

```swift
validate(statusCode: acceptableStatusCodes).validate(contentType: contentTypes())
```

- `validate(statusCode:)` → checks 200–299 by default, else `AFError.responseValidationFailed(.unacceptableStatusCode)` (`:81`)
- `validate(contentType:)` → MIME-matches the `Accept` header, with wildcard support (`:105`)

**When does validation actually run?** When the task finishes, `Request.didCompleteTask` (`Source/Core/Request.swift:515`) fires all stored validators, letting each set `mutableState.error`:

```swift
let validators = validators.read(\.self)
validators.forEach { $0() }        // runs statusCode/contentType checks
...
retryOrFinish(error: self.error)
```

So validation failures surface as an `error` on the request *before* serialization. Then response serializers (queued by `response(...)` / `responseDecodable(...)` via `_response`, `DataRequest.swift:243`) run:

```swift
let result = Result {
    try responseSerializer.serialize(request:, response:, data:, error: self.error)
}
```

Crucially, the serializer receives `self.error` — so if validation failed, the serializer sees that error and the `DataResponse.result` is a `.failure`. On a serialization/validation error, `_response` also consults the interceptor's **retrier** (`delegate.retryResult`) and may retry the whole request instead of completing.

`processNextResponseSerializer` (`:613`) walks the serializer queue one at a time on the serialization queue, then invokes all completion handlers and cleans up.

## The flow in one line

`Session.request` → `perform` → `performSetupOperations` (build `URLRequest` via `ParameterEncoder`, **adapt** via interceptor) → `didCreateURLRequest` (create `URLSessionTask`) → `resume` (send) → `didCompleteTask` (**run stored validators** → set error) → response serializers consume `error`+`data` → completion handler gets `DataResponse`, or retrier reschedules.

The key design idea: **validation and serialization are both deferred closures** — you declare them up front on the `DataRequest`, but they only execute once the network task completes, chained through the request's serialization queue.
