## The two phases: lazy build, then execute

A `QuerySet` is *lazy*. Chaining `.filter()`, `.exclude()`, `.annotate()`, etc. does **not** hit the database — each call clones the QuerySet and mutates an underlying `Query` object (`django/db/models/sql/query.py:232`). SQL is only generated and run when you actually consume the results.

### Phase 1 — Building the abstract `Query`

Every QuerySet wraps a `django.db.models.sql.query.Query` — an in-memory, database-agnostic representation of the query. It holds the pieces a SELECT is assembled from:

- `where` — a `WhereNode` tree (`query.py:324`)
- `alias_map` / `table_map` / `alias_refcount` — the join graph (`query.py:309-322`)
- `annotations`, `extra`, `select_related`, `group_by`, ordering, slicing marks, etc.

Because QuerySets are immutable-by-convention, each operation calls `Query.chain()` → `Query.clone()` (`query.py:388-446`), a lightweight alternative to `deepcopy` that shallow-copies the dicts and clones only the mutable structures (like the `where` tree). This is why `qs2 = qs.filter(...)` never disturbs `qs`.

At this stage no SQL exists yet — just a structured description of intent.

### Phase 2 — Execution, triggered by consumption

Iterating (or `len()`, `list()`, `bool()`, indexing…) calls `QuerySet.__iter__` (`query.py:420`), which calls `_fetch_all()`:

```python
def _fetch_all(self):
    if self._result_cache is None:
        self._result_cache = list(self._iterable_class(self))   # query.py:2238
```

`_result_cache` is the result cache — that's why iterating a QuerySet twice only runs one query. The `_iterable_class` (default `ModelIterable`) drives three nested iterators, documented right in the `__iter__` docstring:

**1. `ModelIterable.__iter__` → `get_compiler` → `execute_sql`** (`query.py:92`)

```python
compiler = queryset.query.get_compiler(using=db)
results = compiler.execute_sql(chunked_fetch=..., chunk_size=...)
```

`Query.get_compiler()` (`query.py:370`) is the key handoff from abstract to concrete: it picks the **backend-specific** compiler via `connection.ops.compiler(...)`. This is how the same `Query` produces different SQL on PostgreSQL vs. MySQL (e.g. MySQL overrides `SQLDeleteCompiler.as_sql` / `SQLUpdateCompiler.as_sql`).

**2. `SQLCompiler.as_sql`** (`compiler.py:757`) — turns the `Query` into a SQL string + params. It calls `pre_sql_setup()` (`compiler.py:83`), which does `setup_query()`, computes `get_order_by()`, splits the WHERE tree into `where`/`having`/`qualify` via `split_having_qualify()`, and computes GROUP BY. Then `as_sql` assembles the clauses in order — `SELECT`, `DISTINCT`, the compiled select columns, `FROM` (`get_from_clause()`), `WHERE`, `GROUP BY`, `HAVING`, `ORDER BY`, `LIMIT/OFFSET`, `FOR UPDATE` — returning `(" ".join(result), tuple(params))`. Note SQL is always built as **string + parameter list**, never interpolated, which is what prevents SQL injection.

**3. `SQLCompiler.execute_sql`** (`compiler.py:1595`) — runs it:

```python
sql, params = self.as_sql()
cursor = self.connection.cursor()      # or chunked_cursor() for server-side
cursor.execute(sql, params)
```

The return shape depends on `result_type`: `MULTI` wraps `cursor.fetchmany()` in `cursor_iter` (fetching `GET_ITERATOR_CHUNK_SIZE` = 100 rows at a time); `SINGLE` does `fetchone()`; `ROW_COUNT` returns `cursor.rowcount`; `CURSOR` hands back the raw cursor. For non-chunked reads it eagerly `list()`s everything into memory.

**Back in `ModelIterable`:** rows come back as plain tuples. `compiler.results_iter()` (`compiler.py:1565`) applies per-column **converters** (DB value → Python value). Then each row is turned into a model instance via `model_cls.from_db(...)`, with `select_related` data hydrated by `related_populators` and annotations attached from `annotation_col_map` (`query.py:131-164`).

## Summary of the pipeline

```
QuerySet.filter()/... ──(clone + mutate)──▶ Query (abstract, DB-agnostic)
        │  lazy — no DB yet
        ▼  consumption (iter/len/list/bool)
QuerySet._fetch_all ──▶ ModelIterable.__iter__
        ├─ Query.get_compiler(using)          → backend-specific SQLCompiler
        ├─ SQLCompiler.as_sql()               → (sql_string, params)  [pre_sql_setup assembles clauses]
        ├─ SQLCompiler.execute_sql()          → cursor.execute(sql, params); fetchmany in 100-row chunks
        ├─ results_iter()                     → apply DB→Python converters, one tuple at a time
        └─ model_cls.from_db(...)             → model instances (+ select_related, annotations)
                                              → cached in _result_cache
```

Key design points: the **`Query` is the abstract model, the `SQLCompiler` is the backend-specific renderer**, `get_compiler()` is the seam between them, everything flows as `(sql, params)` for safety, results stream in 100-row chunks, and `_result_cache` makes re-iteration free.
