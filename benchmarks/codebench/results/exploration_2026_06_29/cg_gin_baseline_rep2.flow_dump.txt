[assistant] Here's how gin routes a request through its middleware chain, traced through the actual code.

## 1. Chains are pre-built at registration time, not per-request

The key design decision: gin doesn't assemble a middleware stack per request. It flattens middleware + handlers into a single flat slice (`HandlersChain`) when the route is registered, and stores that slice in the routing tree.

When you call `router.Use(...)`, middleware is appended to a `RouterGroup`'s `Handlers` slice (`routergroup.go:65`). When you register a route, `handle()` merges the group's middleware with the route's handlers via `combineHandlers` (`routergroup.go:86-91`):

```go
func (group *RouterGroup) combineHandlers(handlers HandlersChain) HandlersChain {
	finalSize := len(group.Handlers) + len(handlers)
	mergedHandlers := make(HandlersChain, finalSize)
	copy(mergedHandlers, group.Handlers)              // middleware first
	copy(mergedHandlers[len(group.Handlers):], handlers) // route handler(s) last
	return mergedHandlers
}
```

So a route ends up as one ordered slice like `[mw1, mw2, ..., finalHandler]`, and `addRoute` stores it in the per-method radix tree (`gin.go:364-377`). The "last handler is the main one" convention shows up in `HandlersChain.Last()` (`gin.go:60-65`).

## 2. Per request: pool a Context, look up the chain

`ServeHTTP` (`gin.go:662`) grabs a pooled `*Context` (sync.Pool, to avoid allocations), resets it, and calls `handleHTTPRequest`.

`handleHTTPRequest` (`gin.go:690`) picks the radix tree for the HTTP method, then `root.getValue(rPath, ...)` matches the path and returns the stored `handlers` chain plus any path params. On a match (`gin.go:719-724`):

```go
c.handlers = value.handlers
c.fullPath = value.fullPath
c.Next()
```

So routing's job is just to **select which pre-built chain to attach to the context**.

## 3. Walking the chain: `Next()` and the index

The actual chain execution is in `Context.Next()` (`context.go:188-196`):

```go
func (c *Context) Next() {
	c.index++
	for c.index < safeInt8(len(c.handlers)) {
		if c.handlers[c.index] != nil {
			c.handlers[c.index](c)
		}
		c.index++
	}
}
```

`c.index` starts at `-1` (set in `reset()`, `context.go:107`). The first `Next()` increments it to 0 and runs handlers in order.

The clever part is **re-entrancy**. A middleware that itself calls `c.Next()` runs the *rest* of the chain inline, then regains control when that nested call returns. This is what enables wrap-around middleware:

```go
func Logger(c *gin.Context) {
	start := time.Now()
	c.Next()                    // runs downstream handlers
	log.Print(time.Since(start)) // runs after they finish
}
```

A middleware that *doesn't* call `Next()` still works — when it returns, the `for` loop in the original `Next()` advances `index` and runs the next handler. So `Next()` is both the loop driver and the recursion hook.

## 4. Aborting

`Abort()` (`context.go:207-209`) sets `c.index = abortIndex` (`math.MaxInt8 >> 1`, i.e. 63). Since the loop condition is `c.index < len(c.handlers)` and chains are capped below `abortIndex` (`combineHandlers` asserts this), bumping index past that ceiling makes the loop terminate — no remaining handlers run. The current handler still finishes (Abort doesn't return early on its own; you typically `return` after it).

## 5. The 404 / 405 paths use the same mechanism

When no route matches, gin still routes through a chain. It sets `c.handlers = engine.allNoRoute` (or `allNoMethod`) and calls `serveError`, which calls `c.Next()` (`gin.go:758`, `764-766`). Those `allNoRoute`/`allNoMethod` chains are global middleware combined with the no-route handlers (`rebuild404Handlers`, `gin.go:356-362`) — which is why `Use()`'d middleware runs even for 404s.

## Summary

| Stage | Where | What happens |
|-------|-------|--------------|
| Registration | `combineHandlers` | middleware + handler flattened into one slice |
| Storage | `addRoute` → radix tree | chain stored per (method, path) |
| Lookup | `handleHTTPRequest` → `getValue` | matching chain attached to Context |
| Execution | `Context.Next()` | index-driven loop, re-entrant for wrap-around |
| Short-circuit | `Abort()` | sets index past the chain length |

The elegance is that middleware and the final handler are the *same type* (`HandlerFunc`) in the *same flat slice*, and `Next()`'s index-based loop handles both linear pass-through and nested wrap-around middleware with no special casing.
