[assistant] Here's how Django's ORM goes from a `QuerySet` to live model instances. The key insight is that there are **three distinct objects** involved, and a hard separation between **building** the query (lazy) and **executing** it (when you iterate).

## The three layers

| Object | File | Role |
|--------|------|------|
| `QuerySet` | `django/db/models/query.py` | High-level, lazy, chainable API. Holds a result cache. |
| `Query` | `django/db/models/sql/query.py` | The abstract query *representation* — a tree of joins, a `WhereNode`, select columns, etc. SQL-dialect-agnostic. |
| `SQLCompiler` | `django/db/models/sql/compiler.py` | Turns a `Query` into an actual SQL string + params for one specific database backend, executes it, and converts rows. |

A `QuerySet` *wraps* a `Query` (`self._query = query or sql.Query(self.model)`, `query.py:333`).

## Phase 1 — Building the query (stays lazy)

Every chaining method (`filter`, `exclude`, etc.) **clones** rather than mutates, which is why QuerySets are reusable:

```
QuerySet.filter(name='foo')          query.py:1648
  → _filter_or_exclude(False, ...)
    → _chain() → _clone()            clones the QuerySet + Query.chain()
    → _filter_or_exclude_inplace()
      → Query.add_q(Q(name='foo'))   query.py:1663
```

Inside the `Query`, `add_q` → `_add_q` → `build_filter` (`query.py:1488`) does the real work:
- resolves the lookup (`name__icontains` → field + lookup type),
- sets up any JOINs needed (`setup_joins`, recorded in `Query.alias_map`),
- builds a `Lookup` object,
- and adds it as a node to **`Query.where`**, a `WhereNode` tree (AND/OR of conditions).

At this point **nothing has touched the database.** The `Query` is just accumulated state: `where`, `select`, `alias_map` (tables/joins), `annotations`, `group_by`, `order_by`, and `low_mark`/`high_mark` (LIMIT/OFFSET).

## Phase 2 — Triggering evaluation

The query executes only when something needs the rows. The trigger funnels through `_fetch_all` (`query.py:2238`):

```python
def _fetch_all(self):
    if self._result_cache is None:
        self._result_cache = list(self._iterable_class(self))
```

This is reached via `__iter__`, `__len__`, `__getitem__` (non-slice), `bool()`, etc. The `_result_cache` is why iterating a second time hits no database. `_iterable_class` is usually **`ModelIterable`** (it's `ValuesIterable`, `FlatValuesListIterable`, etc. for `.values()`/`.values_list()`).

## Phase 3 — Compiling and executing

`ModelIterable.__iter__` (`query.py:92`) drives execution:

```python
compiler = queryset.query.get_compiler(using=db)   # picks backend-specific compiler
results  = compiler.execute_sql(chunked_fetch=..., chunk_size=...)
```

`Query.get_compiler` (`query.py:370`) asks the **backend** for its compiler class — `connection.ops.compiler(self.compiler)(...)` — which is how the same `Query` produces PostgreSQL vs SQLite SQL.

Inside `execute_sql` (`compiler.py:1595`):
1. `as_sql()` runs `pre_sql_setup` → `setup_query` → **`get_select()`**, which compiles each select expression and records `self.select`, `self.klass_info` (which model + which column indices map to it), and `self.annotation_col_map`. It then assembles SELECT / WHERE (`self.compile(self.where)`) / GROUP BY / ORDER BY / LIMIT into a single SQL string and a flat params list.
2. It gets a **PEP-249 cursor** from the backend: `self.connection.cursor()` (or `chunked_cursor()` for server-side iteration).
3. `cursor.execute(sql, params)` — params are passed **separately**, so the DB driver handles escaping (this is the ORM's SQL-injection defense).
4. For a `MULTI` result it returns a `cursor_iter` that lazily `fetchmany()`s rows in chunks (default 100) rather than loading everything at once.

## Phase 4 — Rows back into model instances

`ModelIterable` loops over `compiler.results_iter(results)` (`compiler.py:1565`), which applies **field converters** (`get_converters`) to turn raw DB values into Python types (e.g. a string into a `datetime`/`Decimal`). Then per row:

```python
obj = model_cls.from_db(db, init_list, row[model_fields_start:model_fields_end])
for rel_populator in related_populators:   # select_related joins
    rel_populator.populate(row, obj)
for attr_name, col_pos in annotation_col_map.items():   # annotations
    setattr(obj, attr_name, row[col_pos])
yield obj
```

`init_list` is the list of field attnames (from `klass_info["select_fields"]`), `model_cls.from_db` builds the instance, `select_related` objects are stitched on from the same row, annotations become attributes, and each instance is **yielded** — so it streams.

## End-to-end

```
Model.objects.filter(name='foo')
  → QuerySet.filter            clone QuerySet, Query.add_q → WhereNode  (lazy, no DB)
for obj in qs:

    → compiler.execute_sql

    → compiler.results_iter       raw rows → Python values via converters
```

The clean takeaway: **`QuerySet` = lazy public API**, **`Query` = backend-independent query tree**, **`SQLCompiler` = backend-specific SQL generator + executor**. Chaining mutates only the `Query` tree (via clones); the database is touched exactly once, lazily, and rows stream back through converters into model instances.
