[assistant] Alamofire's request lifecycle runs through `Session`, with the actual `URLRequest` built lazily and validation deferred until the task completes. Here's the full path (all in `Source/Core` unless noted).

## 1. Build — assemble a `URLRequestConvertible`, not a request yet

`AF.request(...)` forwards to `Session.request(...)` (`Session.swift`). The convenience overloads don't build a `URLRequest` immediately — they capture the inputs in a small `URLRequestConvertible` struct:

- **`RequestConvertible`** — for dictionary `Parameters` + a `ParameterEncoding` (`URLEncoding.default`).
- **`RequestEncodableConvertible<Parameters: Encodable>`** — for `Encodable` parameters + a `ParameterEncoder` (`URLEncodedFormParameterEncoder.default`).

Each struct's `asURLRequest()` is where building *actually* happens, and it runs later:
```swift
var request = try URLRequest(url: url, method: method, headers: headers)
try requestModifier?(&request)                       // caller's inout mutation
return try encoding.encode(request, with: parameters) // or encoder.encode(_:into:)
```
The terminal `request(_ convertible: URLRequestConvertible, ...)` constructs a `DataRequest` (a state machine, created in the inert `.initialized` state), wires the `Session` as its `delegate`, and returns it. With the default `requestSetup == .lazy`, nothing is performed yet; `.eager` would call `perform` right away via `performEagerlyIfNecessary`.

## 2. Send — `resume()` → `perform` → setup → task

The request starts when `resume()` is called (`Request.swift`) — automatically once the first response handler is added (if `startRequestsImmediately`), or manually. `resume()` transitions to `.resumed`; if no `URLSessionTask` exists yet it calls `delegate?.readyToPerform(request:)`, which lands in `Session.perform`:

- **`perform`** (`Session.swift:1166`) hops to the serial `rootQueue`, inserts into `activeRequests`, then dispatches to `requestQueue` and type-switches (`UploadRequest` before `DataRequest` because of subclassing) into `performDataRequest`, etc. — all of which call **`performSetupOperations`**.
- **`performSetupOperations`** (`:1246`):
  1. `initialRequest = try convertible.asURLRequest()` — the build from step 1 finally executes — then `try initialRequest.validate()` checks the `URLRequest` itself (valid method/URL). A throw routes to `didFailToCreateURLRequest`.
  2. Notifies `request.didCreateInitialURLRequest`.
  3. If an adapter exists (the session and per-request `RequestInterceptor`s composed via `Interceptor`), it `adapt`s the request asynchronously and re-validates the adapted result; otherwise this is skipped.
  4. Calls `Session.didCreateURLRequest(_:for:)`.
- **`Session.didCreateURLRequest`** (`:1291`): fires `request.didCreateURLRequest` (event monitors, cURL handler), then `request.task(for: urlRequest, using: session)` creates the concrete `URLSessionTask` (data/upload/download subclass override), stores it in `requestTaskMap`, and calls `request.didCreateTask`, which `resume()`s the task if the request is already in `.resumed`.

From there `URLSession` does the network I/O; `SessionDelegate` handles the `URLSessionDelegate` callbacks (incoming data, redirects, auth challenges / server-trust evaluation, metrics, completion).

## 3. Validate — chained closures, deferred to task completion

`.validate()`, `.validate(statusCode:)`, and `.validate(contentType:)` (`Features/Validation.swift`) are chainable and **do not run when called** — they append a closure to `Request.validators` (a `Protected<[() -> Void]>`). `DataRequest.validate(_:)` (`DataRequest.swift:144`) wraps the user `Validation` into:
```swift
guard error == nil, let response else { return }
let result = validation(request, response, data)
if case let .failure(error) = result { self.error = ... }   // first failure wins
eventMonitor?.request(self, didValidateRequest: ..., withResult: result)
```

The validators fire synchronously in **`Request.didCompleteTask`** (`Request.swift:515`) once the task finishes, *before* retry/finish:
```swift
let validators = validators.read(\.self)
validators.forEach { $0() }
...
retryOrFinish(error: self.error)
```

The built-in checks (`Validation.swift`):
- **Status code**: `acceptableStatusCodes.contains(response.statusCode)` — default `200..<300`; else `AFError.responseValidationFailed(.unacceptableStatusCode)`.
- **Content type**: parses the response MIME type and matches it against the acceptable list using `MIMEType.matches` (wildcard `*/*`, `type/*` supported). An empty body is skipped; a missing content type with no wildcard accept → `.missingContentType`; a mismatch → `.unacceptableContentType`. The default acceptable list comes from the request's own `Accept` header (falling back to `["*/*"]`).

A validation failure sets `request.error`, which then propagates to the response serializers/handlers and can drive a retry via the `RequestRetrier` in `retryOrFinish`.

**In short:** build is lazy (inputs → a `URLRequestConvertible` whose `asURLRequest()` runs at perform time), send is an async pipeline on serial queues (`resume → perform → performSetupOperations → adapt → create & resume task`, with `SessionDelegate` driving I/O), and validation is a list of deferred closures executed synchronously at task completion that set the request's error.
