### 2. MULTI-THREADED SCHEDULER ARCHITECTURE

**File location**: `/tmp/codebench_workspaces/20260629T234322Z-36eed4b76d68/cg_tokio_baseline_rep2/tokio/src/runtime/scheduler/multi_thread/`

The multi-threaded scheduler is a **work-stealing thread pool** with these key components:

#### **Worker Threads and Core**

Each worker thread executes the `run()` function (worker.rs:508) which:
1. Takes ownership of a `Core` from an `AtomicCell`
2. Enters the runtime context
3. Calls `cx.run(core)` in an infinite loop until shutdown

**Core struct** (worker.rs:111-165):
```rust
struct Core {
    /// Task execution scheduling tick counter
    tick: u32,
    
    /// LIFO slot (next task slot) - optimization for task locality
    lifo_slot: Option<Notified>,
    lifo_enabled: bool,
    
    /// Worker-local run queue (fixed-size 256-element circular buffer)
    run_queue: queue::Local<Arc<Handle>>,
    
    is_searching: bool,          // Worker looking for work to steal
    is_shutdown: bool,           // Runtime shutting down
    is_traced: bool,             // Tracing enabled
    
    had_driver: park::HadDriver, // Did we just park on I/O driver?
    enable_eager_driver_handoff: bool,
    
    park: Option<Parker>,        // Used to sleep/wake thread
    stats: Stats,                // Performance metrics
    global_queue_interval: u32,  // How often to check global queue
    rand: FastRand,              // RNG for work stealing
}
```

#### **The Main Worker Loop** (worker.rs:561-627)

The core worker loop `Context::run()` executes:

```
Loop (while !is_shutdown):
  1. Increment tick
  2. Run maintenance (if tick % event_interval == 0)
     - Calls park_yield() to poll drivers without blocking
  3. Try to get next task from local queues:
     a) Check LIFO slot first (next_task → next_local_task)
     b) Check local run_queue
     c) Every global_queue_interval ticks, check injection queue
  4. If task found: run_task(task, core) [execution in cooperative budget]
     - Polls task with coop::budget wrapper
     - May run LIFO slot tasks up to MAX_LIFO_POLLS_PER_TICK (3) times
  5. If no local work, try to steal from other workers (steal_work)
  6. If no work found anywhere: park() thread until notified
```

**Key intervals**:
- `tick` counter increments each loop iteration
- `global_queue_interval`: How often to check the injection queue (tuned, default ~31 ticks)
- `event_interval`: Controls maintenance frequency (from config)

1. **Local Queue** (`Local<T>`, lines 29-398): Per-worker, 256-element fixed-size circular buffer
   - Only the worker thread pushes (single producer)
   - Other workers steal from it (MPMC stealers)
   - Uses atomic `head` (64-bit) packing two indices: `(steal, real)`
   - Uses `tail` (32-bit) for write position
   - When full, triggers overflow

2. **LIFO Slot** (`Option<Notified>` in Core): Optimization for message-passing patterns
   - Last notified task stored here, checked before run_queue
   - Improves cache locality and reduces latency
   - Can be disabled via config (`disable_lifo_slot`)
   - Capped at 3 polls per tick to prevent starvation

3. **Injection Queue** (`inject::Shared<T>`, worker.rs:176): Global MPMC queue
   - Used when local queues overflow OR tasks spawned from outside worker threads
   - Used when pulling work during global_queue_interval checks
   - Growable linked-list based MPMC queue

#### **Queue Operations**:

**Local queue push_back_or_overflow** (queue.rs:188-223):
- If queue has space: direct push to circular buffer
- If queue full: move half of queue to injection queue (overflow)
- This keeps frequently-used tasks in fast local queue

**Local queue pop** (queue.rs:361-398):
- Pops from head in FIFO order
- Uses CAS to atomically claim task, handling concurrent stealers

### 4. WORK-STEALING MECHANISM

**File**: `/tmp/codebench_workspaces/20260629T234322Z-36eed4b76d68/cg_tokio_baseline_rep2/tokio/src/runtime/scheduler/multi_thread/queue.rs:416-562`

#### **How stealing works** (`steal_into` and `steal_into2`):

When a worker runs out of local work and is in "searching" state (worker.rs:1141-1169):

1. **Searching limit**: Only if less than 50% of workers are already searching (idle.rs:104-114)
2. **Random victim selection**: Pick random worker, iterate through all workers
3. **Skip self**: Don't try to steal from own queue
4. **Attempt steal**: Call `target.steal.steal_into(&mut self.run_queue, &mut self.stats)`

#### **steal_into algorithm** (queue.rs:417-468):
```
1. Check destination queue isn't already half-full
2. Call steal_into2 to claim tasks atomically
3. If n tasks stolen:
   - Return one task immediately
   - Push remaining n-1 tasks to destination queue
   - Update stats
```

#### **steal_into2 implementation** (queue.rs:472-562):
```
1. Load source queue head (atomic, 64-bit: steal_idx || real_idx)
2. Check no concurrent steal in progress (steal_idx == real_idx)
3. Load source queue tail
4. Calculate n = (tail - real) - (tail - real)/2  [steal half]
5. Atomically CAS head to claim tasks (sets real_idx to steal_to)
   - This blocks other stealers until complete
6. Copy n tasks from source buffer to destination buffer
7. Issue release CAS to complete the steal (real_idx == steal_idx again)
```

**Work-stealing order for idle worker** (worker.rs:1141-1169, Core::steal_work):
1. Try to steal from other workers' local queues (preferring ones with tasks)
2. Fallback to global injection queue (Handle::next_remote_task)

### 5. LIFO SLOT OPTIMIZATION

**Location**: worker.rs:119-124, 1132-1134

The LIFO (Last-In-First-Out) slot is a critical optimization:

```rust
/// When a task is scheduled from a worker, it is stored in this slot. The
/// worker will check this slot for a task **before** checking the run queue.
/// This effectively results in the **last** scheduled task to be run next (LIFO).
/// This is an optimization for improving locality which benefits message 
/// passing patterns and helps to reduce latency.
lifo_slot: Option<Notified>,
```

**Benefits**:
- **Message-passing pattern**: When Task A wakes Task B, B executes immediately after A yields
- **Better cache locality**: Working with recently-touched data
- **Reduced latency**: Less context switching for dependent tasks

**Implementation** (handle.rs:1353-1385, `schedule_local`):
```rust
if is_yield || !core.lifo_enabled {
    // Yield forces to back of queue
    core.run_queue.push_back_or_overflow(task, ...);
} else {
    // Normal schedule: check LIFO slot
    let prev = core.lifo_slot.take();  // Evict old LIFO task
    if let Some(prev) = prev {
        core.run_queue.push_back_or_overflow(prev, ...);
    }
    core.lifo_slot = Some(task);  // New task in LIFO slot
    return prev.is_some();  // Only notify if slot was full
}
```

**Starvation protection** (worker.rs:743-746):
```rust
if lifo_polls >= MAX_LIFO_POLLS_PER_TICK {  // MAX_LIFO_POLLS_PER_TICK = 3
    core.lifo_enabled = false;  // Disable LIFO to let other tasks run
}
```

```rust
fn with_budget<R>(budget: Budget, f: impl FnOnce() -> R) -> R {
    let maybe_guard = context::budget(|cell| {
        let prev = cell.get();
        cell.set(budget);  // Set 128 unit budget
        ResetGuard { prev }  // Auto-restore on drop
    });
    f()  // Execute task poll
    // Guard drops, restores previous budget
}
```

## The two schedulers

A runtime carries one of two schedulers, dispatched through an enum in `scheduler/mod.rs`:

- **`CurrentThread`** — single-threaded, a simple `VecDeque` run queue.
- **`MultiThread`** — the work-stealing thread pool (the default `#[tokio::main]` runtime).

Both implement the `Schedule` trait (`task/mod.rs:293`), whose key method is `fn schedule(&self, task: Notified<Self>)`. That trait is the seam between a woken task and "put it on a queue."

## What a task actually is

When you `spawn` a future, tokio doesn't keep a `Box<dyn Future>` lying around. It allocates a single `Cell<T, S>` (`task/core.rs`) holding three parts, cache-padded against false sharing:

- **`Header`** — hot data: an atomic `state` bitfield, a `queue_next` pointer (so the task *is* its own intrusive linked-list node in the injection queue), and a `vtable`.
- **`Core`** — the future itself, or its output once complete (same memory, reused).
- **`Trailer`** — cold data: the `JoinHandle` waker, owner list pointers.

`spawn` returns three handles to this one allocation — a `Task` (owned-list ownership), a `Notified` (the "I'm runnable, put me on a queue" token), and a `JoinHandle`. The state starts as `NOTIFIED | JOIN_INTEREST` plus a refcount of 3, one for each handle (`task/state.rs:61`). When the refcount hits 0 the allocation is freed.

## The multi-threaded worker loop

Each worker thread owns a `Core` (`multi_thread/worker.rs:111`) and runs `Context::run()` in a loop. Per iteration it:

1. Bumps a `tick` counter.
2. Every `event_interval` ticks, runs **maintenance** — polls the I/O/timer driver without blocking, so ready I/O wakes the right tasks.
3. Picks the next task, in priority order:
   - the **LIFO slot** (see below),
   - the worker's **local run queue**,
   - the **global injection queue** — but every `global_queue_interval` ticks (~31, dynamically tuned) it checks the global queue *first* so injected tasks don't starve.
4. Runs the task inside a cooperative budget.
5. If there's no local work, tries to **steal**.
6. If there's nothing anywhere, **parks** the thread.

### Three queues

- **Local run queue** (`queue.rs`) — a fixed 256-slot ring buffer, single-producer (the owning worker) / multi-consumer (stealers). Head packs two indices `(steal, real)` into one 64-bit atomic so stealing and popping coordinate via CAS. When it fills, half its tasks **overflow** into the global queue.
- **LIFO slot** — a single `Option<Notified>` checked *before* the run queue. When a task wakes another task from within a worker, the woken task goes here and runs *next*. This is the message-passing optimization: if A sends to B and yields, B runs immediately on the warm cache instead of going to the back of the queue. To prevent two tasks ping-ponging forever, it's capped at `MAX_LIFO_POLLS_PER_TICK = 3`, after which the slot is disabled for that tick (`worker.rs:743`).
- **Global injection queue** (`inject.rs`) — an MPMC linked list. Receives tasks spawned from *outside* any worker (e.g. from `main`, or from another runtime's thread) and overflow tasks.

### Work stealing

When a worker runs dry it marks itself "searching" and tries to steal (`queue.rs:417`, `worker.rs:1141`):

- Only if **fewer than half** the workers are already searching — this throttles a thundering herd (`idle.rs:104`).
- It picks a random victim and walks the worker list, calling `steal_into`, which atomically claims **half** the victim's queue: CAS the victim's `steal` index forward, copy the tasks out, then CAS to release. It returns one task to run immediately and pushes the rest onto its own local queue.
- If no worker has anything, it falls back to the global queue.

### Parking / unparking

Idle coordination lives in `idle.rs`, which packs `(unparked_count, searching_count)` into one atomic. When a worker finds nothing, it transitions to parked and sleeps on its `Parker`. When a task is scheduled remotely (or the *last* searching worker is about to sleep), `notify_parked` pops a sleeper and unparks exactly one worker — enough to handle the new work without waking everyone.

## Waking and re-scheduling

A `Waker` is built from the task header via a raw vtable (`task/waker.rs:118`): `clone` bumps the refcount, `drop` drops it, `wake`/`wake_by_ref` drive the state machine. When `wake` runs:

1. The state transitions toward `NOTIFIED` (`state.rs:215`). If the task is currently `RUNNING`, it's just marked notified and the running poll will re-queue it on exit; if `IDLE`, it's marked notified and a refcount is taken to put it back on a queue; if `COMPLETE`, the wake is dropped.
2. If it needs queuing, the vtable's `schedule` is called → `Handle::schedule_task` (`multi_thread/handle.rs:1327`), which decides **local vs remote**:
   - **Local** (we're on a worker of the same runtime that holds a core): goes into the LIFO slot, or the back of the run queue if it's a `yield`.
   - **Remote** (any other thread): pushed onto the injection queue, then `notify_parked_remote` wakes a sleeper.

## Running the future and the coop budget

To actually run a task, the worker wraps the poll in `coop::budget` (`task/coop/mod.rs`). Each poll gets a budget of **128 units**; tokio's own leaf resources (channels, I/O, timers) decrement it on each ready operation. When it hits zero, those resources start returning `Pending` even though data is available, forcing the task to yield. This is what stops a single hot task (e.g. a busy socket in a tight loop) from monopolizing a worker and starving everyone else — it's cooperative scheduling layered on top of `async`/`await`. `tokio::task::yield_now()` is built on the same mechanism.

The harness (`task/harness.rs:153`) handles the poll: transition to `RUNNING`, poll the future; on `Ready` it stores the output, marks `COMPLETE`, and wakes the `JoinHandle`; on `Pending` it either re-queues (if woken during the poll) or drops its notified ref and waits for the next wake.

## Current-thread, for contrast
