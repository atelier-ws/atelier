[assistant] Django's ORM splits query handling into three cooperating layers, and the whole thing is **lazy** — no SQL touches the database until you iterate. Here's the full path.

## 1. Building: the `QuerySet` wraps a `Query`

A `QuerySet` (`django/db/models/query.py`) is a thin, lazy, chainable wrapper. Every `.filter()`, `.exclude()`, `.annotate()`, `.order_by()` **clones** the queryset and mutates its underlying `Query` object (`django/db/models/sql/query.py:232`) — an abstract, backend-agnostic representation of the query as data structures, not SQL text.

`Query` holds:
- `where` — a tree of lookup nodes (`WhereNode`), built up by `add_q` / `add_filter` (`query.py:1660`). `.filter(name="x")` becomes `self.add_q(Q(("name", "x")))`.
- `select`, `annotations`, `group_by`, ordering, `low_mark`/`high_mark` (LIMIT/OFFSET), join/alias bookkeeping, etc.

Because chaining only edits this in-memory structure, you can compose a queryset freely with zero DB traffic.

## 2. Trigger: iteration forces evaluation

The query runs only when the result is actually needed — iterating, `len()`, `list()`, `bool()`, slicing without step, etc. All routes converge on `QuerySet._fetch_all` (`query.py:2238`):

```python
def _fetch_all(self):
    if self._result_cache is None:
        self._result_cache = list(self._iterable_class(self))
```

The results are cached in `_result_cache`, so re-iterating the *same* queryset does **not** re-hit the DB. `_iterable_class` selects the row shape: `ModelIterable` (model instances), `ValuesIterable` (dicts), `ValuesListIterable` (tuples), `FlatValuesListIterable`.

## 3. Compiling: `Query` → SQLCompiler → SQL string

The iterable (e.g. `ModelIterable.__iter__`, `query.py:92`) asks the `Query` for a backend-specific compiler:

```python
compiler = queryset.query.get_compiler(using=db)   # query.py:370
results  = compiler.execute_sql(...)
```

`get_compiler` picks the compiler class via `connection.ops.compiler(...)` — that's the indirection that lets each backend (PostgreSQL, MySQL, SQLite, Oracle) customize SQL generation.

`SQLCompiler.as_sql()` (`compiler.py:757`) is where the abstract `Query` becomes an actual SQL string plus a parameter tuple. It calls `pre_sql_setup()` (`compiler.py:83`) to finalize the SELECT columns, `get_order_by()`, split WHERE/HAVING/QUALIFY, and compute GROUP BY, then assembles `SELECT ... FROM ... WHERE ... ORDER BY ... LIMIT ...`. Individual expressions/lookups are turned into SQL by `compile(node)` (`compiler.py:574`), which dispatches to each node's own `as_sql`. **Values are never interpolated into the string** — they're collected as separate `params` for the DB driver to bind, which is what makes the ORM injection-safe.

## 4. Executing: cursor + row processing

`SQLCompiler.execute_sql` (`compiler.py:1595`) runs it:

```python
sql, params = self.as_sql()
cursor = self.connection.cursor()        # or chunked_cursor()
cursor.execute(sql, params)
...
return list(cursor_iter(cursor, ...))    # for MULTI: fetchmany() in chunks
```

The `result_type` controls fetching: `MULTI` (fetchmany, chunked), `SINGLE` (fetchone), `ROW_COUNT`, `CURSOR`, `NO_RESULTS`. `EmptyResultSet` short-circuits to an empty iterator without ever hitting the DB.

## 5. Row → Python objects

`compiler.results_iter()` (`compiler.py:1565`) takes the raw DB rows and applies **converters** (`get_converters`/`apply_converters`) to turn DB values into Python types (dates, decimals, etc.). Then the iterable builds the final objects — for `ModelIterable`, each row goes through `model_cls.from_db(...)` to instantiate a model, and `related_populators` fill in `select_related` joins and annotations (`query.py:131-164`).

## The three nested iterators

Django's own docstring at `query.py:420` sums up the default read path:

1. **`execute_sql()`** — pulls ~100 rows at a time (`GET_ITERATOR_CHUNK_SIZE`) via `cursor.fetchmany()`, in chunks.
2. **`results_iter()`** — yields one row at a time as a tuple, applying value converters.
3. **`iterator()` / the iterable class** — turns each tuple into a model instance (or dict/tuple).

## Summary of the flow

```
QuerySet.filter()/.annotate()   → clone + mutate Query (where-tree, select, ...)   [no DB]
        │  iterate / len / list / bool
        ▼
QuerySet._fetch_all → _iterable_class(self)          (query.py:2238)
        ▼
Query.get_compiler(using=db)                          (query.py:370)
        ▼
SQLCompiler.as_sql() → (sql string, params)           (compiler.py:757)  ← backend-specific
        ▼
SQLCompiler.execute_sql() → cursor.execute(sql, params); fetchmany  (compiler.py:1595)
        ▼
results_iter() → converters                           (compiler.py:1565)
        ▼
model_cls.from_db(row) → model instances (cached in _result_cache)
```

Related variants reuse the same machinery through different compiler subclasses: `SQLInsertCompiler`, `SQLUpdateCompiler`, `SQLDeleteCompiler`, `SQLAggregateCompiler` (all in `compiler.py`), each overriding `as_sql`/`execute_sql` for their statement type. Raw SQL (`Model.objects.raw()`) bypasses compilation via `RawQuery` (`query.py:156`), which executes the given SQL string directly but still routes rows through the converter/`from_db` machinery.

The key design ideas: **laziness** (build as data, execute on demand), **result caching** (one DB hit per queryset), **backend abstraction** (Query is portable; the compiler is backend-specific), and **parameterized SQL** (values bound by the driver, never string-formatted).
