## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

## 1. Build the request

Everything starts at `Session.request(...)` (`Source/Core/Session.swift:318`). The convenience overload just packages your inputs (URL, method, parameters, encoding, headers, `requestModifier`) into a `RequestConvertible` value, then calls the `URLRequestConvertible` overload (`Session.swift:393`):

```swift
let request = DataRequest(convertible: convertible,
                          underlyingQueue: rootQueue,
                          serializationQueue: serializationQueue,
                          eventMonitor: eventMonitor,
                          interceptor: interceptor,
                          shouldAutomaticallyResume: shouldAutomaticallyResume,
                          delegate: self)
performEagerlyIfNecessary(request)
return request
```

Note the `DataRequest` is returned **before** any network work happens — the actual URLRequest hasn't been created yet. `AF.request(...)` is just `Session.default.request(...)`.

## 2. Send the request

The heavy lifting is in `Session.performSetupOperations` (`Session.swift:1246`), run on the `requestQueue`:

1. **Create & validate the URLRequest** — `convertible.asURLRequest()` turns your convertible into a real `URLRequest`, then `initialRequest.validate()` is called immediately. (This is a *request-side* sanity check — e.g. GET requests with a body — distinct from response validation.) On failure it routes to `request.didFailToCreateURLRequest`.
2. **Adapt** — if a `RequestInterceptor`/adapter is configured, `adapter.adapt(...)` gets a chance to mutate the request (auth headers, etc.), and the adapted request is `validate()`d again.
3. **Create the task** — `Session.didCreateURLRequest(_:for:)` (`Session.swift:1291`) runs on the `rootQueue`:
   ```swift
   request.didCreateURLRequest(urlRequest)
   guard !request.isCancelled else { return }
   let task = request.task(for: urlRequest, using: session)
   mutableState.write { $0.requestTaskMap[request] = task }
   request.didCreateTask(task)
   ```
   `task(for:using:)` is overridden per subclass — `DataRequest` returns `session.dataTask(with:)` (`DataRequest.swift:119`). The `RequestTaskMap` is the bidirectional bookkeeping between `Request`s and `URLSessionTask`s.
4. **Resume** — `didCreateTask` (`Request.swift:388`) appends the task and, based on the request's state machine, resumes it. If you called `.resume()` (`Request.swift:768`) before the task existed, the state is already `.resumed`, so the task resumes as soon as it's created. With `startRequestsImmediately`/`shouldAutomaticallyResume`, adding the first response handler triggers the resume automatically.

Response data arrives via the `SessionDelegate`, which looks up the `Request` in the `requestTaskMap` and feeds it `DataRequest.didReceive(data:)` (`DataRequest.swift:86`), accumulating bytes into `dataMutableState.data`.

## 3. Validate the response

Validation is **deferred and closure-based**. Calling `.validate()` doesn't validate anything immediately — it appends a validator closure to a protected `validators` array. `DataRequest.validate(_:)` (`DataRequest.swift:144`):

```swift
let validator: @Sendable () -> Void = { [unowned self] in
    guard error == nil, let response else { return }
    let result = validation(request, response, data)
    if case let .failure(error) = result {
        self.error = error.asAFError(or: .responseValidationFailed(reason: .customValidationFailed(error: error)))
    }
    eventMonitor?.request(self, didValidateRequest: request, response: response, data: data, withResult: result)
}
validators.write { $0.append(validator) }
```

The built-in checks in `Source/Features/Validation.swift` are just convenience wrappers that build such closures:
- `validate(statusCode:)` (`Validation.swift:160`) → checks `acceptableStatusCodes.contains(response.statusCode)`, else `.unacceptableStatusCode`.
- `validate(contentType:)` (`Validation.swift:175`) → MIME-type matching against `Accept`, with wildcard handling.
- `validate()` with no args applies the default 200–299 range plus content-type from the `Accept` header.

These validators actually **run at completion**, in `Request.didCompleteTask` (`Request.swift:515`):

```swift
mutableState.write { $0.error = $0.error ?? error }
let validators = validators.read(\.self)
validators.forEach { $0() }          // <-- validation happens here
eventMonitor?.request(self, didCompleteTask: task, with: error)
retryOrFinish(error: self.error)
```

So the order is: task completes → transport error (if any) is recorded → all validators run in sequence (each may set `self.error`) → `retryOrFinish(error:)`.

## 4. Finish (and retry)

`retryOrFinish` (`Request.swift:543`) asks the interceptor's retrier whether to retry. `.retry`/`.retryWithDelay` loops back through `prepareForRetry()` → `reset()` → re-perform; otherwise `finish(error:)` (`Request.swift:563`) sets `isFinishing` and calls `processNextResponseSerializer()`, which runs your `.response`/`.responseDecodable` handlers on the `serializationQueue`. Because validation ran before finishing, a validation failure surfaces as the `AFError` your response handler receives.

## The pipeline in one line

`Session.request` → `DataRequest` created → `performSetupOperations` (asURLRequest → request-validate → adapt → make `dataTask`) → `resume()` → `didReceive(data:)` accumulates → `didCompleteTask` runs response validators → `retryOrFinish` → `finish` runs serializers.

A recurring design theme worth noting: the whole lifecycle is a set of `did...` callbacks hopping between the `rootQueue` (state/task) and `serializationQueue` (results), with all mutable state guarded by `Protected` wrappers — and validation/serialization are lazily-registered closures that fire at fixed lifecycle points, not eagerly.
