## The two-loop architecture

Excalidraw separates **mutation** (changing element data) from **rendering** (drawing to canvas), connected by a callback chain. The key insight: elements are mutated *in place* for collaboration efficiency, and version/nonce changes flow up to React, which re-renders the canvases.

## Rendering: layered canvases

The scene is split across **three stacked `<canvas>` layers**, each redrawn independently so frequent interactions don't force a full scene repaint:

| Layer | File | What it draws | Trigger |
|---|---|---|---|
| **Static** | `renderer/staticScene.ts` | grid + the actual shapes | `renderStaticScene()` in a `useEffect`, throttled via `throttleRAF` |
| **Interactive** | `renderer/interactiveScene.ts` | selection boxes, transform handles, collaborator cursors | `AnimationController.tick()` per animation frame |
| **New element** | `renderer/renderNewElementScene.ts` | the shape currently being drawn | during creation drag |

`scene/Renderer.ts` orchestrates; `_renderStaticScene(...)` and `_renderInteractiveScene(...)` are the internal implementations. Frame throttling (`renderer/animation.ts`) picks `requestAnimationFrame` (throttled) vs `setTimeout(…,0)` based on `isRenderThrottlingEnabled()`.

## Drawing one element + two caches

`packages/element/src/renderElement.ts` → `renderElement(element, elementsMap, …, rc, context, …)` draws each element with **RoughJS** for the hand-drawn look. Two WeakMap caches keep this cheap:

- **`ShapeCache`** (`element/src/shape.ts`) — caches the RoughJS *drawable* (geometry) per element. Keyed by element, invalidated on theme change/export.
- **`elementWithCanvasCache`** (`renderElement.ts`) — caches a *pre-rendered bitmap* per element. Invalidated on zoom/theme/bound-text/crop changes.

WeakMaps mean cache entries are GC'd automatically when an element object is replaced.

## Updating: mutation → re-render chain

```
App.mutateElement(el, updates)
  └─ Scene.mutateElement(el, updates, {informMutation:true})
       └─ mutateElement()         // in-place: version++, versionNonce=random, updated=ts
       │                          //          ShapeCache.delete(el) if width/height/points/fileId changed
       └─ if version changed && informMutation:
            Scene.triggerUpdate()
              ├─ sceneNonce = random()        // scene-wide cache-bust key
              └─ callbacks.forEach(cb => cb()) // registered via scene.onUpdate(this.triggerRender)
                   └─ App.triggerRender → this.setState({})
                        └─ React re-renders → StaticCanvas useEffect → renderStaticScene()
```

The mechanism that ties it together:

- **`version` / `versionNonce`** (`element/src/types.ts`) — `version` is a sequential counter; `versionNonce` is a random int. Both bump on every change, used for collaboration reconciliation *and* as render-cache keys.
- **`sceneNonce`** — regenerated on every `triggerUpdate()`. `Renderer.getRenderableElements()` uses it as the memoization key, so a stale scene nonce can't return a cached element list.
- **Two mutation flavors**: `mutateElement()` (in-place, fast, for live dragging) vs `newElementWith()` (immutable copy, when a fresh object identity is needed).

## The complete loop

User action → `mutateElement` bumps version + invalidates `ShapeCache` if geometry changed → `Scene.triggerUpdate` rolls `sceneNonce` and fires callbacks → `setState({})` re-renders App → `StaticCanvas` effect calls `renderStaticScene` → `renderElement` redraws via RoughJS, hitting `ShapeCache`/`elementWithCanvasCache` for anything unchanged. Meanwhile the interactive canvas runs its own RAF loop for selection/handles/cursors, independent of this.

**Design takeaway:** in-place mutation keeps collaboration and dragging fast; version/nonce counters bridge mutation to both React's re-render and the WeakMap render caches; three canvas layers ensure that high-frequency UI feedback never triggers an expensive full-scene repaint.
