**State Layout** (state.rs:6-61):
```rust
pub(super) struct State {
    val: AtomicUsize,
}

// Bitfield structure:
const RUNNING: usize = 0b0001;           // Task being polled/cancelled
const COMPLETE: usize = 0b0010;          // Future completed, never unset
const NOTIFIED: usize = 0b100;           // Task in run queue
const JOIN_INTEREST: usize = 0b1_000;    // JoinHandle exists
const JOIN_WAKER: usize = 0b10_000;      // Join waker stored
const CANCELLED: usize = 0b100_000;      // Task marked for cancellation

// Ref-count occupies remaining bits, shifted by REF_COUNT_SHIFT
const INITIAL_STATE: usize = (REF_ONE * 3) | JOIN_INTEREST | NOTIFIED;
// Initial: 3 ref-counts + JOIN_INTEREST + NOTIFIED bits set
// Ref-counts: OwnedTask (1), Notified (1), JoinHandle (1)
```

**State Transitions** (state.rs:117-333):

1. **IDLE → RUNNING** (lines 117-145):
   - `transition_to_running()`: Acquire RUNNING bit (lock), clear NOTIFIED, check CANCELLED
   - Success: Continue to poll
   - Cancelled: Transition to Complete
   - Failed: Task already running/complete, drop ref-count

2. **RUNNING → IDLE** (lines 151-181):
   - `transition_to_idle()`: Release RUNNING bit, check for re-notification
   - If re-notified: Increment ref-count (create new Notified)
   - If not notified: Decrement ref-count (consume the Notified)

3. **RUNNING → COMPLETE** (lines 184-192):
   - `transition_to_complete()`: XOR RUNNING and COMPLETE bits atomically
   - Returns snapshot showing join interest/waker status

4. **COMPLETE → TERMINAL** (lines 198-207):
   - `transition_to_terminal(count)`: Subtract `count` ref-counts atomically
   - Returns true if ref-count hits zero (task can be deallocated)

5. **NOTIFIED Transitions** (lines 215-278):
   - `transition_to_notified_by_val()`: Consumes a ref-count
     - If RUNNING: Set NOTIFIED, decrement ref-count (caller will requeue)
     - If COMPLETE or already NOTIFIED: Decrement ref-count (no submission needed)
     - Else (IDLE): Set NOTIFIED, increment ref-count (create new Notified to submit)
   - `transition_to_notified_by_ref()`: Does not consume ref-count
     - Increments ref-count only if transitioning from IDLE → NOTIFIED

6. **NOTIFIED + CANCEL** (lines 303-333):
   - `transition_to_notified_and_cancel()`: Used for remote abort
   - Sets CANCELLED bit
   - If IDLE: Also set NOTIFIED and increment ref-count to submit cancellation

**Reference Counting** (state.rs:476-511):

fn ref_inc(&self) {
    // Relaxed ordering: new references can only come from existing references
    let prev = self.val.fetch_add(REF_ONE, Relaxed);
    if prev > isize::MAX as usize {
        process::abort();  // Overflow protection
    }
}

fn ref_dec(&self) -> bool {
    // AcqRel: Establish synchronization on last decrement
    let prev = Snapshot(self.val.fetch_sub(REF_ONE, AcqRel));
    prev.ref_count() == 1  // Return true if this was the final ref
}

```rust
pub(super) fn poll(self) {
    match self.poll_inner() {  // Poll the future
        PollFuture::Notified => {
            // Future yielded, was re-notified during poll
            self.core().scheduler.yield_now(Notified(self.get_new_task()));
            self.drop_reference();
        }
        PollFuture::Complete => {
            // Future completed
            self.complete();  // Handle waker, hooks, release from scheduler
        }
        PollFuture::Dealloc => {
            self.dealloc();  // Deallocate task
        }
        PollFuture::Done => (),  // Task finished, nothing more to do
    }
}

fn poll_inner(&self) -> PollFuture {
    match self.state().transition_to_running() {
        TransitionToRunning::Success => {
            // Acquired RUNNING lock
            let waker_ref = waker_ref::<S>(&header_ptr);  // Create waker for this task
            let cx = Context::from_waker(&waker_ref);     // Build Context
            let res = poll_future(self.core(), cx);       // Poll the future
            
            if res == Poll::Ready(()) {
                return PollFuture::Complete;
            }
            
            // Transition back to IDLE, check if re-notified
            match self.state().transition_to_idle() {
                TransitionToIdle::OkNotified => PollFuture::Notified,
                TransitionToIdle::Ok => PollFuture::Done,
                TransitionToIdle::OkDealloc => PollFuture::Dealloc,
                TransitionToIdle::Cancelled => {
                    cancel_task(self.core());
                    PollFuture::Complete
                }
            }
        }
        TransitionToRunning::Cancelled => {
            // Task was cancelled before we acquired RUNNING
            cancel_task(self.core());
            PollFuture::Complete
        }
        TransitionToRunning::Failed => PollFuture::Done,      // Already running
        TransitionToRunning::Dealloc => PollFuture::Dealloc,  // Deallocate
    }
}
```

impl Budget {
    const fn initial() -> Budget {
        Budget(Some(128))  // 128 units per task poll
    }

    const fn unconstrained() -> Budget {
        Budget(None)  // No limit
    }

    fn has_remaining(self) -> bool {
        self.0.map_or(true, |budget| budget > 0)
    }
}

pub(crate) fn budget<R>(f: impl FnOnce() -> R) -> R {
    with_budget(Budget::initial(), f)
}

fn with_budget<R>(budget: Budget, f: impl FnOnce() -> R) -> R {
    // Store budget in thread-local context, execute closure, restore old budget
    struct ResetGuard { prev: Budget }
    impl Drop for ResetGuard {
        fn drop(&mut self) {
            let _ = context::budget(|cell| cell.set(self.prev));
        }
    }
    
    let maybe_guard = context::budget(|cell| {
        let prev = cell.get();
        cell.set(budget);
        ResetGuard { prev }
    });
    f()
}
```

    fn decrement(&mut self) -> BudgetDecrement {
        if let Some(num) = &mut self.0 {
            if *num > 0 {
                *num -= 1;
                let hit_zero = *num == 0;
                BudgetDecrement { success: true, hit_zero }
            } else {
                BudgetDecrement { success: false, hit_zero: false }
            }
        } else {
            BudgetDecrement { success: true, hit_zero: false }
        }
    }
}

pub fn poll_proceed(cx: &mut Context<'_>) -> Poll<RestoreOnPending> {
    context::budget(|cell| {
        let mut budget = cell.get();
        let decrement = budget.decrement();

        if decrement.success {
            // Budget available, allow progress
            let restore = RestoreOnPending::new(cell.get());
            cell.set(budget);
            if decrement.hit_zero {
                inc_budget_forced_yield_count();  // Metrics
            }
            Poll::Ready(restore)
        } else {
            // Budget exhausted, yield to scheduler
            register_waker(cx);  // Store waker for re-run
            Poll::Pending
        }
    }).unwrap_or(Poll::Ready(RestoreOnPending::new(Budget::unconstrained())))
}
```

**Integration with Task Polling:**

The budget is set up by the scheduler *before* calling `task.poll()`:
1. Scheduler calls `task.poll()` via vtable
2. Inside `harness.poll()`, context sets budget to `Budget::initial()` (128 units)
3. The future and its internal futures can call `poll_proceed()` to check/consume budget
4. When budget hits zero, `poll_proceed()` returns `Poll::Pending` with a waker
5. Scheduler re-queues the task (via waker) after other tasks run
6. This prevents one task from starving others

## The big picture

Tokio is a **work-stealing scheduler**. There are N worker threads (one `Core` each), each with its own local run queue, plus a shared global "injection" queue. A spawned future is packed into a single heap allocation and driven by polling. When a future's `Waker` is called, the task is re-queued onto a scheduler, and a worker eventually polls it again. Cooperative budgeting keeps any one task from hogging a worker.

---

## 1. What a task *is* — one allocation, three handles

When you call `tokio::spawn(fut)`, the future doesn't just get pushed somewhere. It's wrapped into a `Cell<T, S>` — a single heap allocation laid out as (`task/core.rs:126`):

```rust
struct Cell<T: Future, S> {
    header: Header,   // hot: atomic state + vtable + owner id  (offset 0)
    core:   Core<T,S>,// the scheduler S + task id + the future-or-output
    trailer: Trailer, // cold: JoinHandle waker, owned-list pointers, hooks
}
```

`Header` is first so any `*mut Cell` can be cast to `*mut Header` regardless of the concrete future type. The `Header` holds a `&'static Vtable` (`raw.rs:24`) of monomorphized function pointers — `poll`, `schedule`, `dealloc`, `shutdown`, etc. This vtable is how the runtime drives a heterogeneous set of futures through type-erased `NonNull<Header>` pointers.

`new_task` produces **three references** to that one allocation (`task/mod.rs:323`), all counted in a single atomic refcount:
- `Task<S>` → stored in the scheduler's `OwnedTasks` list (ownership/cleanup),
- `Notified<S>` → the "this is runnable, put me in a queue" handle,
- `JoinHandle<T::Output>` → returned to you.

The initial atomic state is `(REF_ONE * 3) | JOIN_INTEREST | NOTIFIED` (`task/state.rs`) — three refs, marked runnable from birth.

---

## 2. Spawning: where the task first lands

The spawn path: `handle.spawn` → `scheduler::spawn` → `OwnedTasks::bind` → `schedule_option_task_without_yield(notified)` (`multi_thread/handle.rs:91-99`). That last call routes the brand-new `Notified` into a run queue via `schedule_task`. The routing decision (`worker.rs:1320`):

- **If spawning from within a worker thread** of *this* runtime (it holds a `Core`) → schedule onto that worker locally.
- **Otherwise** (spawning from outside, e.g. from `main` or another runtime) → push onto the **global injection queue** and notify a parked worker (`worker.rs:1341`).

---

## 3. The worker loop — task selection priority

Each worker runs `Context::run` (`worker.rs:561`). Per iteration:

```rust
while !core.is_shutdown {
    core.tick();
    core = self.maintenance(core);          // periodic I/O poll (see §7)
    if let Some(task) = core.next_task(&worker) {   // local work
        core = self.run_task(task, core)?;
        continue;
    }
    if let Some(task) = core.steal_work(&worker) {  // steal from peers
        core = self.run_task(task, core)?;
    } else {
        core = self.park(core);             // sleep until woken
    }
}
```

The **priority order** for finding the next task:

1. **LIFO slot** — the single most-recently-woken task (`next_local_task`, `worker.rs:1133`: `self.lifo_slot.take().or_else(|| self.run_queue.pop())`).
2. **Local run queue** — FIFO, per-worker.
3. **Global injection queue** — but only checked *periodically*, every `global_queue_interval` ticks (`next_task`, `worker.rs:1063`). On those ticks the global queue is checked **first**, before local, to prevent injection-queue starvation.
4. **Work-stealing** from other workers' queues.
5. **Park** (sleep) if everything is empty.

When pulling from the global queue while otherwise idle, the worker grabs a *batch* — roughly `inject.len() / num_workers + 1` tasks, capped at half the local-queue capacity — returning one to run and pushing the rest local (`worker.rs:1109-1126`). This amortizes the lock and spreads work.

---

## 4. The LIFO slot — locality optimization, with anti-starvation

When a running task wakes another task and schedules it locally (and it's not a `yield`), that task goes into the **LIFO slot** rather than the back of the queue (`schedule_local`, `worker.rs:1360`):

```rust
let prev = core.lifo_slot.take();
if let Some(prev) = prev {
    core.run_queue.push_back_or_overflow(prev, ...); // evict old occupant
}
core.lifo_slot = Some(task);
```

This is the classic message-passing optimization: task A sends to channel, wakes task B; B runs *immediately next* on the same core while A's data is still cache-hot. But unconditional LIFO can starve everyone else if two tasks keep waking each other. So:

- `MAX_LIFO_POLLS_PER_TICK = 3` (`worker.rs:264`). After 3 consecutive LIFO polls the worker sets `lifo_enabled = false`, forcing tasks back to the normal queue (`worker.rs:743`).
- The LIFO slot also respects the **coop budget**: if budget is exhausted, the slotted task is pushed to the run queue instead of polled immediately (`worker.rs:716`).
- One more subtlety: the LIFO-slot task **cannot be stolen**. When another worker steals this worker's core, the slot task is moved into the run queue first so it becomes stealable (`worker.rs:452`).

---

## 5. The local run queue — fixed ring buffer with overflow

The local queue (`multi_thread/queue.rs`) is a lock-free fixed-capacity ring buffer: **256 slots** (`LOCAL_QUEUE_CAPACITY`, 4 under loom tests). It uses split atomic head/tail indices, where the head packs *two* head values to coordinate with concurrent stealers.

On a local push when full, `push_back_or_overflow` runs **overflow**: it moves **half the queue plus the new task to the global injection queue** in one shot (`queue.rs` `push_overflow`). Overflow only ever moves the *second half*, which is why the batch-from-inject code (§3) deliberately keeps pulled tasks in the *first half* — so a task isn't pulled from global and immediately overflowed back.

---

## 6. Work-stealing and waking workers

When a worker's own queues are empty, `steal_work` (`worker.rs:1141`) tries to steal:

```rust
if !self.transition_to_searching(worker) { return None; } // throttle
let start = self.rand.fastrand_n(num);           // random victim
for i in 0..num {
    let target = &shared.remotes[(start + i) % num];
    if let Some(task) = target.steal.steal_into(&mut self.run_queue, ...) {
        return Some(task);
    }
}
```

Two throttles prevent thundering herds:
- **At most ~half the workers search at once** (`Idle::transition_worker_to_searching`, `idle.rs`). A worker that can't enter the searching state just parks.
- **Random starting victim** so stealers don't all pile onto worker 0.

`steal_into` atomically claims **half** of the victim's tasks via a CAS on the victim's head, copying them into the thief's local buffer and returning one to run immediately (`queue.rs:417`). The dual-head encoding lets it detect a competing stealer and back off (return 0).

**Parking/unparking** is coordinated by the `Idle` struct (`idle.rs`), which packs `(num_searching, num_unparked)` into one atomic. The key fairness rule: a newly-scheduled task only bothers to wake a sleeping worker if **no worker is currently searching** (`worker_to_notify`) — if someone's already hunting for work, they'll find it, so don't wake another thread. The last worker to find work-while-searching is responsible for waking a replacement, keeping at least one thread looking for work whenever work exists. Workers park on the **I/O/timer driver** if they hold it (so I/O readiness wakes them), otherwise on a condvar (`park.rs`).

## 7. Running a task: poll, the Waker, and re-scheduling

`run_task` (`worker.rs:630`) wraps the poll in the coop budget:

```rust
coop::budget(|| {
    task.run();           // vtable poll -> Harness::poll
    // then drain the LIFO slot up to MAX_LIFO_POLLS_PER_TICK...
});
```

Inside `Harness::poll` (`harness.rs:153`):
1. `transition_to_running()` — CAS the RUNNING bit, clear NOTIFIED, check for cancellation (`state.rs:117`).
2. Build a `Waker` from the task header pointer + a global `WAKER_VTABLE` (`waker.rs`). The waker *is* the task: its data pointer is the `Header`.
3. Poll the future with that `Context`.
4. On `Poll::Pending` → `transition_to_idle`. If the task was notified again *during* its own poll, it's immediately re-queued; otherwise it sleeps until its waker fires. On `Poll::Ready` → `transition_to_complete`, store output, wake the `JoinHandle`.

**Waking is how rescheduling happens.** When some I/O resource or channel calls `waker.wake()`:
- `wake_by_val` / `wake_by_ref` call `state.transition_to_notified_*` (`state.rs:215`). If the task was idle, this sets NOTIFIED, bumps the refcount to mint a fresh `Notified`, and calls the vtable's `schedule` (`raw.rs:346`):
  ```rust
  scheduler.schedule(Notified(Task::from_raw(ptr)));
  ```
- That lands in `Schedule::schedule` for `Arc<Handle>` (`multi_thread/handle.rs:110`) → `schedule_task` → back into the same LIFO-slot / local-queue / inject routing from §2–4.
- If the task was already RUNNING or COMPLETE, waking just flips a bit / drops the ref — no duplicate enqueue. This is what makes wakes idempotent and race-free.

The `Schedule` trait (`task/mod.rs:293`) is the clean boundary: `schedule`, `release` (remove from `OwnedTasks`), `yield_now` (re-queue to the *back*, never the LIFO slot), `hooks`.

## 8. Fairness: cooperative budget and maintenance ticks

Two mechanisms keep the runtime fair and responsive:

**Coop budget** (`task/coop/mod.rs`). Before each poll the worker installs a budget of **128 units** in a thread-local. Tokio's leaf resources (sockets, channels, timers) call `poll_proceed`, which decrements the budget; when it hits zero, the resource returns `Poll::Pending` *as if not ready* even when it is, after registering the waker (`coop/mod.rs:342`). The task yields, gets re-queued, and other tasks get a turn. This stops a future that's draining a never-empty socket from monopolizing the worker forever.

**Maintenance / global-queue ticks.** Two periodic counters:
- `event_interval` ticks → `maintenance` parks with a zero timeout to **poll the I/O driver** without sleeping (`worker.rs:780`), so I/O readiness is checked even under steady CPU load.
- `global_queue_interval` ticks → check the global queue *before* local work (§3). This interval is auto-tuned from observed poll times (`tune_global_queue_interval`, `worker.rs:1307`).

### In one sentence

A spawned future becomes a single ref-counted allocation driven through a vtable; worker threads each pull from a LIFO slot → local queue → (periodically) global queue → stolen work, poll it under a 128-unit coop budget, and re-enqueue it whenever its `Waker` fires — with LIFO-slot locality, half-queue work-stealing, and at-most-half-searching parking rules balancing throughput against fairness.
