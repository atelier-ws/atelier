## The big picture

Excalidraw renders to **two stacked `<canvas>` layers**, decoupling stable content from volatile UI:

| Layer | Component | Draws | Cadence |
|---|---|---|---|
| **Static** | `StaticCanvas.tsx` | the actual elements (shapes, text, images, arrows) | only when elements/zoom/scroll change, RAF-throttled |
| **Interactive** | `InteractiveCanvas.tsx` | selection box, transform/binding handles, remote cursors, snap guides | continuously, via an `AnimationController` |

This split means dragging a selection handle doesn't force every shape to redraw — only the cheap UI layer repaints.

## The render pipeline (static layer)

1. **Entry:** `renderStaticScene(config, throttle?)` in `renderer/staticScene.ts`. The throttled path is `renderStaticSceneThrottled = throttleRAF(_renderStaticScene)` — it coalesces rapid calls into at most one paint per animation frame.
2. **Element selection:** `scene/Renderer.ts` → `getRenderableElements()` filters to visible, non-deleted elements. It's **memoized on a `canvasNonce`** derived from the scene nonce, so if nothing changed the same array is returned and downstream work is skipped.
3. **Per element:** `_renderStaticScene` iterates and calls `renderElement()` (`element/src/renderElement.ts`), which switches on element type.

## Two-level caching is the core trick

- **`ShapeCache`** (`element/src/shape.ts`) — a `WeakMap<element, {shape, theme}>` holding the **Rough.js `Drawable`** (the hand-drawn geometry). Rough.js generators (`.rectangle`, `.ellipse`, `.polygon`, `.curve`) are expensive, so the result is cached per element + theme. Bypassed during export.
- **`elementWithCanvasCache`** (`renderElement.ts`) — a `WeakMap<element, ...>` holding a fully-rendered **off-screen canvas** per element. On a normal frame, `renderElement` just `drawImage`s this cached bitmap onto the main context (`drawElementFromCanvas`) — no re-drawing of the shape at all.

  The cache key bundles `theme`, `scale`, `angle`, `zoomValue`, bound-text version, image crop, frame opacity, etc. Any of these changing regenerates the off-screen canvas (`generateElementCanvas`, which respects browser limits of ~16M px / 32K dimensions). During **export**, caching is bypassed and elements draw directly via `drawElementOnCanvas` + `rc.draw()`.

There's a separate SVG path (`staticSvgScene.ts` → `renderSceneToSvg`) for export.

## How updates flow and trigger re-renders

The element + the version fields drive everything:

- Each element carries **`version`** (monotonic counter), **`versionNonce`** (random — deterministic tiebreaker for collaboration reconciliation), and **`updated`** (timestamp).
- **`mutateElement(element, map, updates)`** (`element/src/mutateElement.ts`) mutates in place *only if* something actually changed, then bumps all three fields. (`newElementWith` is the immutable variant.)

The mutation → repaint chain:

```
scene.mutateElement(el, updates, { informMutation: true })
  └─ mutateElement() bumps version / versionNonce / updated
  └─ scene.triggerUpdate()          // Scene.ts: regenerates sceneNonce, fires callbacks
       └─ App.triggerRender()        // registered via scene.onUpdate(this.triggerRender)
            └─ this.setState({})      // empty setState → React re-render
                 └─ StaticCanvas useEffect re-runs → renderStaticScene(...)
                      └─ Renderer sees a new canvasNonce → cache misses → repaint
```

So the **`sceneNonce`** (regenerated on every meaningful mutation) is the signal that busts the renderer's memoization. Because the per-element caches are keyed on the element object and its version-derived fields, *only the mutated element* regenerates its bitmap; everything else hits cache.

## History / undo-redo (orthogonal to rendering)

`Store` + `StoreSnapshot` (`element/src/store.ts`) capture state independently. `maybeClone` detects changes via reference equality plus `hashElementsVersion` (a djb2 hash of `versionNonce`s), and structurally clones **only changed elements**. `StoreDelta.calculate/inverse/applyTo` produce undo/redo deltas; on replay, `version`/`versionNonce` are excluded so reapplied changes get fresh versions (keeping collaboration sane). Capture timing is governed by `CaptureUpdateAction` (`IMMEDIATELY` / `NEVER` / `EVENTUALLY`).

## Summary

The performance story is **layer separation + nonce-gated memoization + two WeakMap caches**. Mutations bump version fields and a scene nonce; the nonce invalidates the renderer's memoized element set and `setState` schedules a RAF-throttled repaint; the shape cache and per-element off-screen-canvas cache ensure unchanged elements are blitted rather than redrawn, while the interactive layer animates UI chrome independently on top.
