## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

[assistant] The extension host communicates with the main renderer/process through a **layered RPC protocol built on message-passing**. Here's how it works, from the transport up to the API surface:

## 1. The transport (physical channel)

The extension host runs as a separate process (or a web worker). At startup, `_createExtHostProtocol()` (`src/vs/workbench/api/node/extensionHostProcess.ts:180`) establishes one of three transports and wraps it in an `IMessagePassingProtocol`:

- **MessagePort** — the desktop default: a `MessagePortMain` is transferred from the renderer; messages flow via `port.postMessage`/`port.on('message')`.
- **Socket** (`VSCODE_EXTHOST_IPC_SOCKET`) — used for remote/reconnectable scenarios. The renderer sends the ext host a `net.Socket` handle, wrapped in a `NodeSocket` or `WebSocketNodeSocket` and driven by a `PersistentProtocol` that supports reconnection with a grace period.
- **Named pipe** — `net.createConnection(pipeName)` wrapped in a `NodeSocket` + `PersistentProtocol`.

All three converge on the tiny `IMessagePassingProtocol` interface (`src/vs/base/parts/ipc/common/ipc.ts:99`):

```ts
interface IMessagePassingProtocol {
  send(buffer: VSBuffer): void;
  readonly onMessage: Event<VSBuffer>;
  drain?(): Promise<void>;
}
```

So everything above the transport just sends/receives opaque `VSBuffer` byte blobs — the transport details are abstracted away.

## 2. The RPC layer

Both sides construct an `RPCProtocol` (`src/vs/workbench/services/extensions/common/rpcProtocol.ts:117`) over that protocol. This is the actual request/response engine, and it's symmetric — the renderer's side lives in `ExtensionHostManager`, the ext host's side is created in `ExtensionHostMain` (`src/vs/workbench/api/common/extensionHostMain.ts:176`).

**Calling a remote method** — `getProxy(identifier)` returns a JS `Proxy` (`rpcProtocol.ts:251`). Any `$`-prefixed method access is trapped and turned into `_remoteCall(rpcId, method, args)`, which:
1. Serializes the arguments (`MessageIO.serializeRequest`), assigns an incrementing `req` id.
2. Stores a `LazyPromise` in `_pendingRPCReplies[req]`.
3. Sends the bytes via `this._protocol.send(msg)`.

**Receiving** — `_receiveOneMessage` (`rpcProtocol.ts:282`) reads a one-byte `MessageType` + `req` id and dispatches. For a request it calls `_invokeHandler` → `_doInvokeHandler`, which looks up the local actor in `this._locals[rpcId]` and invokes the real method, then serializes the resolved value back as a `ReplyOK*` (or `ReplyErr`) message. Replies resolve the pending promise on the caller's side.

Notable mechanics baked in:
- **Acknowledgements + responsiveness tracking** — each request is acked; if acks don't arrive within `UNRESPONSIVE_TIME` (3s), the ext host is flagged `Unresponsive` (this is what surfaces the "extension host unresponsive" UI).
- **Cancellation** — a trailing `CancellationToken` argument is detected and turned into a separate `Cancel` message.
- **URI transformation** — URIs are rewritten on the way in/out (`transformIncomingURIs` / `_uriReplacer`) so file paths make sense across the local/remote boundary.

## 3. The named-actor API (the contracts)

The `rpcId`s correspond to named proxy identifiers defined in `extHost.protocol.ts` — the `MainContext.*` actors (things the ext host calls in the renderer, e.g. `MainThreadExtensionService`) and `ExtHostContext.*` actors (things the renderer calls in the ext host). Each `MainThread*` class registers itself via `rpcProtocol.set(identifier, this)`. The whole VS Code extension API is implemented as these `$`-prefixed methods flowing over the RPC protocol.

## Summary of the stack

```
Extension API (MainThread* / ExtHost* actors, $-methods)
        ↓  getProxy / set
RPCProtocol   — req/reply, acks, cancellation, URI transform, serialization
        ↓  send(VSBuffer) / onMessage
IMessagePassingProtocol
        ↓
MessagePort  |  Socket (PersistentProtocol, reconnectable)  |  Named pipe
```

So: **separate process, connected by a MessagePort/socket/pipe, over which a symmetric `RPCProtocol` marshals `$`-prefixed method calls between named "main thread" and "ext host" actors** — an async message-passing RPC, never shared memory.
