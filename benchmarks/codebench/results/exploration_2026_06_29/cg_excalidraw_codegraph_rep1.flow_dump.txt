## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

[assistant] Excalidraw uses a **retained-mode scene model with a React-driven, multi-canvas rendering pipeline**. Here's how the pieces fit together.

## The core data model: `Scene`

`Scene` (`packages/element/src/Scene.ts`) is the single source of truth. It holds the element array plus derived indexes (`elementsMap`, `nonDeletedElements`, `frames`) that are recomputed together in `replaceAllElements` (Scene.ts:271). Two key primitives drive updates:

- **`triggerUpdate()`** (Scene.ts:303) — bumps `sceneNonce = randomInteger()` and fires every registered callback. The nonce is the cheap "did anything change?" signal used downstream to invalidate render caches.
- **`onUpdate(cb)`** — subscription mechanism; `App` registers a callback that schedules a re-render.

## How elements are updated: `mutateElement`

There are three layers of `mutateElement`, each thinner:

1. `App.mutateElement` (App.tsx:4644) — the public API surface, delegates to the scene.
2. `Scene.mutateElement` (Scene.ts:411) — calls the low-level mutate, then compares `version` before/after. It only calls `triggerUpdate()` if the element is actually in the scene, its version changed, **and** `informMutation` is true. This version check is what prevents redundant renders.
3. `element/src/mutateElement.ts:37` — the raw in-place mutation that bumps `version`/`versionNonce` and invalidates the element's cached shape (`ShapeCache`).

So the update flow is: **`mutateElement` → version bump + shape-cache invalidation → `triggerUpdate` → nonce change → callback → React re-render.**

## The render pipeline

Excalidraw splits rendering across **separate canvas layers**, the main two being the **static canvas** (committed elements) and the **interactive canvas** (selection, handles, cursors). `App.render()` mounts `<StaticCanvas>` and `<InteractiveCanvas>`.

**Which elements to draw** — `Renderer` (`scene/Renderer.ts:40`) computes this. `getRenderableElements` (Renderer.ts:208) builds a `canvasNonce` from the scene nonce and calls the **memoized** `_getRenderableElements`, which:
- filters out the element being text-edited and any new in-frame element (`getRenderableElementsMap`),
- viewport-culls via `isElementInViewport` (`getVisibleCanvasElements`),
- optionally reorders selected elements into a highlighted frame during drag.

Because it's memoized on the nonce + viewport props, an unchanged scene returns the same object references.

**When to repaint** — `StaticCanvas` (`components/canvases/StaticCanvas.tsx:33`) is a `React.memo` component. Its custom `areEqual` (line 108) bails out unless `canvasNonce`, `scale`, `elementsMap`, or `visibleElements` changed (or a relevant `appState` prop). On render, its `useEffect` calls `renderStaticScene`. This is the bridge from React's declarative world to imperative canvas drawing.

**The actual draw** — `renderStaticScene` (staticScene.ts:491) → throttled to animation frames via `renderStaticSceneThrottled` (`throttleRAF`) when throttling is enabled → `_renderStaticScene` (staticScene.ts:229):
1. `bootstrapCanvas` clears + fills background, applies zoom scale,
2. draws the grid,
3. iterates `visibleElements`, calling **`renderElement`** per element (with frame-clipping, bound-text, and link-icon handling), embeddables painted on top.

**Per-element drawing + caching** — `renderElement` (`element/src/renderElement.ts:780`) draws each element by rendering into an **off-screen per-element canvas** (`generateElementCanvas`, line 204) which is then blitted to the main canvas. `drawElementOnCanvas` uses **Rough.js** (`rough.canvas`) for the hand-drawn look. The rough "shape" (path/drawable) itself is cached in **`ShapeCache`** (`element/src/shape.ts:81`), keyed per element and invalidated on mutation. So there are two cache layers: the rough shape, and the rasterized element canvas (keyed on version, zoom, theme, etc.).

## Putting it together

```
mutateElement ──► version++ / ShapeCache invalidate
       │
       ▼
Scene.triggerUpdate ──► sceneNonce = random ──► onUpdate callback
       │
       ▼
App re-renders ──► Renderer.getRenderableElements (memoized on nonce)
       │
       ▼
<StaticCanvas> React.memo (areEqual checks nonce/maps)
       │  useEffect
       ▼
renderStaticScene ─(throttleRAF)─► _renderStaticScene
       │  per visible element
       ▼
renderElement ──► generateElementCanvas (Rough.js + ShapeCache) ──► drawImage to canvas
```

The design's efficiency hinges on **nonces and version counters** cascading through **layered memoization** (renderable-elements memo → `React.memo` → element-canvas cache → shape cache), so a mutation only re-rasterizes what actually changed, and `throttleRAF` caps repaints at the display's frame rate.
