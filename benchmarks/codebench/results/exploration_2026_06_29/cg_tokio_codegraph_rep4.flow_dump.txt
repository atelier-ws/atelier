## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

## The big picture

Tokio's runtime is a set of **worker threads**, each running an identical loop that pulls tasks off queues and polls their futures. A "task" is a spawned future wrapped in a reference-counted allocation (`RawTask`) with a vtable; scheduling just means placing that task's handle onto a queue and possibly waking a sleeping worker.

## Where tasks live: a hierarchy of queues

Each worker (`Core` in `multi_thread/worker.rs`) has three tiers of work, checked in priority order:

1. **LIFO slot** (`lifo_slot`) — a single-task fast path. When a running task wakes another task, the woken task goes here (`schedule_local`, worker.rs:1353). This exploits locality: task B often depends on data task A just produced, so running it *immediately* on the same core keeps it cache-hot.
2. **Local run queue** (`run_queue`) — a bounded, lock-free ring buffer, FIFO, owned by this worker. Other workers may *steal* from it but only this worker pushes/pops its own.
3. **Global inject queue** (`shared.inject`) — a shared, mutex-guarded queue for tasks spawned from *outside* any worker (e.g. `runtime.spawn(...)` from the main thread) or that overflowed a local queue.

## The worker run loop

`Context::run` (worker.rs:561) is the heart:

```
loop while not shutdown:
  tick()                          // increment counter
  maintenance()                   // periodic shutdown/trace checks
  if let Some(task) = next_task() { run_task(task); continue }   // local work
  if let Some(task) = steal_work() { run_task(task) }            // steal
  else park()                     // sleep until woken
```

**`next_task`** (worker.rs:1062) decides local-vs-global by tick: every `global_queue_interval` ticks it checks the global queue *first* (to avoid starving injected tasks), otherwise it drains the LIFO slot + local queue first, and only dips into the global queue in batches when local work is exhausted. The batch size is `inject.len() / num_workers + 1` — take a fair share, leave the rest for others.

## Work stealing

When a worker's own queues are empty, `steal_work` (worker.rs:1141) kicks in:

- It first tries to transition into the **searching** state via `transition_to_searching`. Only if *fewer than half* the workers are already searching does it proceed — this deliberately prevents a thundering herd of stealers.
- It picks a **random** starting worker and scans all peers, calling `steal_into` to move roughly half of a victim's queue into its own.
- Falls back to the global queue if no peer had work.

This is the classic Chase-Lev-style work-stealing deque design that keeps all cores busy without central coordination.

## Running a task = polling its future

`run_task` (worker.rs:630) → `task.run()` → the task's vtable `poll` → `Harness::poll` (harness.rs:153) → `poll_future` (harness.rs:521), which calls `future.poll(cx)` inside a `catch_unwind` panic guard and a **cooperative budget** (`coop::budget`).

- `Poll::Pending` → nothing to do; the task is now owned by whatever waker it registered. When that waker fires, `schedule` re-queues the task.
- `Poll::Ready` → `complete()` stores the output for the `JoinHandle` and releases the task.

### The LIFO fast-path loop
After polling, `run_task` loops on the LIFO slot (worker.rs:693): if the task woke another task into `lifo_slot`, that task runs *right away* on the same core. To prevent two tasks ping-ponging each other from starving everything else, it's capped at `MAX_LIFO_POLLS_PER_TICK = 3` (worker.rs:264, 743), after which the LIFO slot is disabled for the tick and the task goes to the normal queue.

### Cooperative scheduling
Each poll runs under a **budget** (`coop::budget`). Tokio's leaf resources (I/O, channels) decrement it and return `Pending` once exhausted, forcing long-running tasks to yield so they can't monopolize a worker (worker.rs:716 checks `has_budget_remaining`).

## How scheduling wakes a sleeping worker

When a task is scheduled (`schedule_task`, worker.rs:1327):
- If we're *on the worker thread that owns the runtime*, it goes to that core's LIFO slot / local queue (`schedule_local`).
- Otherwise it's pushed to the global inject queue and `notify_parked_remote` wakes a parked worker.

**Parking** (`park`, worker.rs:810): a worker with no work calls `transition_to_parked`, then blocks on the I/O/timer driver (`park_internal`). The driver serves double duty — it's both the "sleep until notified" mechanism *and* the thing that polls OS I/O readiness. When a peer schedules work or the driver gets an event, the worker unparks, re-enters searching, and resumes the loop. The `idle` structure tracks which workers are asleep so exactly one gets woken per new batch of work (avoiding wasteful wakeups).

## Blocking work

`spawn_blocking` and `block_in_place` are separate: blocking tasks run on a dedicated thread pool (`blocking/pool.rs`) whose `run` loop (pool.rs:503) is a plain mutex+condvar queue — no stealing, no polling, just run-to-completion — so blocking code never stalls the async workers.

---

**In one sentence:** each worker thread loops pulling tasks from a LIFO slot → local queue → global queue → stealing from peers, polls each task's future under a cooperative budget until `Ready` or `Pending`, and sleeps on the I/O driver when idle — with wakers re-queueing tasks and idle-worker notifications tying it all together.
