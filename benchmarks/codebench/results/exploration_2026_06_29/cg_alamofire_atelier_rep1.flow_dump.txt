## 1. Build — assembling and encoding the `URLRequest`

Everything starts at `Session.request(...)` (`Source/Core/Session.swift`). There are three overloads, but they funnel into the one taking a `URLRequestConvertible`. The convenience overloads wrap your inputs in an internal struct that knows how to lazily produce a `URLRequest`:

```swift
struct RequestConvertible: URLRequestConvertible {
    func asURLRequest() throws -> URLRequest {
        var request = try URLRequest(url: url, method: method, headers: headers)
        try requestModifier?(&request)              // your inline tweaks
        return try encoding.encode(request, with: parameters)  // params → query/body
    }
}
```

- `URLConvertible.asURL()` turns a `String`/`URL`/`URLComponents` into a `URL`.
- Parameters are encoded by either the legacy `ParameterEncoding` (`URLEncoding`, `JSONEncoding` in `ParameterEncoding.swift`) or the modern `Encodable`-based `ParameterEncoder` (`URLEncodedFormParameterEncoder`, `JSONParameterEncoder` in `ParameterEncoder.swift`), which also sets `Content-Type`.

The actual conversion is deferred. In `Session.performSetupOperations()` (`Session.swift`) Alamofire calls `convertible.asURLRequest()`, runs a basic `URLRequest.validate()` (sane method/URL), and then — if a `RequestInterceptor`/`RequestAdapter` is set — calls `adapter.adapt(initialRequest, using:completion:)` to inject things like auth headers, re-validating the adapted result. The `DataRequest` object itself is created earlier (`DataRequest.init` stores the `convertible` for this lazy flow).

## 2. Send — turning the `URLRequest` into a `URLSessionTask`

Once an adapted `URLRequest` exists, `DataRequest.task(for:using:)` creates the task:

```swift
override func task(for request: URLRequest, using session: URLSession) -> URLSessionTask {
    session.dataTask(with: request)
}
```

- `RequestTaskMap` (`RequestTaskMap.swift`) keeps a bidirectional `Request ↔ URLSessionTask` map so delegate callbacks can find their owning request.
- `SessionDelegate` (`SessionDelegate.swift`) receives the `URLSessionDataDelegate` callbacks and forwards them:
  - `didReceive response` → `request.didReceiveResponse(...)`
  - `didReceive data` → `DataRequest.didReceive(data:)` appends into `self.data`
  - `didFinishCollecting metrics` → stores `URLSessionTaskMetrics`
  - `didCompleteWithError` → `request.didCompleteTask(task, with:)`

Response **serialization** (`response(...)`, `responseDecodable(...)` in `DataRequest.swift`) doesn't run inline — each handler enqueues a serializer closure via `appendResponseSerializer()`, and those run on the `serializationQueue` only *after* the task completes and validation passes.

## 3. Validate — status codes and content types

`.validate()` doesn't run immediately. Each `validate` call appends a closure to `Request.validators` (a thread-safe `Protected<[() -> Void]>`). On task completion, `Request.didCompleteTask()` runs them all before deciding to retry or finish:

```swift
let validators = validators.read(\.self)
validators.forEach { $0() }      // run validation
...
retryOrFinish(error: self.error)
```

The default `DataRequest.validate()` (`Source/Features/Validation.swift`) checks two things:

- **Status code** must be in `200..<300`; otherwise → `AFError.responseValidationFailed(reason: .unacceptableStatusCode(code:))`.
- **Content-Type** must match the `Accept` header (with wildcard support like `application/*`); otherwise → `.unacceptableContentType(...)` or `.missingContentType(...)`.

A failing validator sets `self.error`, which short-circuits serialization and surfaces as the `AFError` in your completion handler.

## End-to-end ordering

1. `Session.request` → create `DataRequest`, store the `URLRequestConvertible`.
2. `performSetupOperations` → `asURLRequest()` + parameter encoding → adapter → final `URLRequest`.
3. `DataRequest.task(for:using:)` → `URLSessionDataTask`, registered in `RequestTaskMap`.
4. `SessionDelegate` streams data into `DataRequest.data`; `didCompleteTask` fires on completion.
5. Validators run → set `self.error` on failure → `retryOrFinish`.
6. Response serializers run on `serializationQueue`; completion handlers dispatch (default `.main`).
