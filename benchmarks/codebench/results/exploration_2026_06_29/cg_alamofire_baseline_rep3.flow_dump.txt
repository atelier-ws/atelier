## The big picture

Everything is orchestrated by `Session`, which serializes all mutating work onto a single serial `rootQueue`. A `Request` is a state machine that wraps one or more `URLSessionTask`s; a `SessionDelegate` bridges `URLSession`'s callbacks back into the request. Here's the lifecycle in three phases.

---

## 1. Build

**Entry points** (`Source/Core/Session.swift`) — three `request(...)` overloads that all converge on a core method:

- `request(_ convertible: URLConvertible, method:parameters:encoding:headers:...)` — classic params + `ParameterEncoding`
- `request<Parameters: Encodable>(_:parameters:encoder:...)` — Codable params + `ParameterEncoder`
- `request(_ convertible: URLRequestConvertible, interceptor:...)` — the core overload that actually constructs the `DataRequest`

The first two wrap their inputs in lightweight structs — `RequestConvertible` and `RequestEncodableConvertible` — that conform to `URLRequestConvertible`. The key abstraction is two protocols (`Source/Core/URLConvertible+URLRequestConvertible.swift`):

```swift
public protocol URLConvertible: Sendable { func asURL() throws -> URL }
public protocol URLRequestConvertible: Sendable { func asURLRequest() throws -> URLRequest }
```

`String`/`URL`/`URLComponents` conform to `URLConvertible`; the convertible structs' `asURLRequest()` is where the actual `URLRequest` gets built: construct `URLRequest(url:method:headers:)`, apply the `requestModifier`, then encode parameters via either:

- **`ParameterEncoding`** (`ParameterEncoding.swift`) — `URLEncoding` (query string or form body, method-dependent) / `JSONEncoding`, for `[String: Any]` parameters.
- **`ParameterEncoder`** (`ParameterEncoder.swift`) — `JSONParameterEncoder` / `URLEncodedFormParameterEncoder`, for `Encodable` parameters.

Importantly, **this encoding is lazy** — the struct just stores the inputs. The `DataRequest` is created immediately holding the convertible, but `asURLRequest()` isn't called until the request is performed.

---

## 2. Send

`Session.request(...)` creates the `DataRequest` (with `delegate: self`, the `rootQueue`, a serialization queue, the interceptor) and, if auto-resume is set, calls `perform`. The pipeline (`Session.swift`):

1. **`perform(_:forRetry:)`** — on `rootQueue`, registers the request in `activeRequests`, then hops to the concurrent `requestQueue`.
2. **`performSetupOperations(...)`** — calls `convertible.asURLRequest()` (the lazy build above) and `initialRequest.validate()`. Failure here → `didFailToCreateURLRequest`. Success → `didCreateInitialURLRequest`.
3. **Adapt** — if a `RequestInterceptor`/`RequestAdapter` exists, `adapter.adapt(initialRequest, using:completion:)` runs (async; this is where e.g. auth tokens get injected). Session + request interceptors compose into a single `Interceptor`. On completion → `didCreateURLRequest`.
4. **`didCreateURLRequest`** — back on `rootQueue`. Calls `task = request.task(for: urlRequest, using: session)` — `DataRequest` overrides this to return `session.dataTask(with:)` (Upload/Download return their respective task types). Stores the `request ↔ task` mapping in `requestTaskMap`, then `request.didCreateTask(task)`.
5. **`didCreateTask`** (`Request.swift`) — applies the request's current **state** to the new task.

**The state machine** (`Request.swift`, `enum State`): `.initialized → .resumed ⇄ .suspended → .cancelled/.finished` (last two terminal). Because task creation is async, `didCreateTask` switches on current state: if `.resumed`, call `task.resume()`; if `.cancelled`, resume-then-cancel (to still gather metrics); etc. This decouples user-facing `resume()`/`suspend()`/`cancel()` from whether the underlying task exists yet.

**`SessionDelegate`** (`SessionDelegate.swift`) receives `URLSession` callbacks, looks up the `Request` via the task map, and forwards: data chunks → `didReceiveResponse`, metrics → `didGatherMetrics`, and completion → `didCompleteTask(task, with: error)`.

---

## 3. Validate

Validation is opt-in via fluent calls on `DataRequest`/`DownloadRequest` (`Source/Features/Validation.swift`):

```swift
public typealias Validation = @Sendable (URLRequest?, HTTPURLResponse, Data?) -> ValidationResult
// ValidationResult = Result<Void, any Error & Sendable>

.validate(statusCode: 200..<300)
.validate(contentType: ["application/json"])
.validate()   // default: status 200..<299 + content types from the Accept header (or */*)
```

- **Status code**: checks the response code is in the acceptable sequence → else `.unacceptableStatusCode`.
- **Content type**: parses media types via a `MIMEType` helper that handles wildcards (`application/*`, `*/*`); empty/nil bodies pass automatically → else `.unacceptableContentType` / `.missingContentType`.

**Storage & timing** — each `validate(...)` appends a closure to `Request.validators` (a thread-safe `Protected<[() -> Void]>`). They don't run when you call them; they run during **`didCompleteTask`** (`Request.swift`):

```swift
let validators = validators.read(\.self)
validators.forEach { $0() }   // each sets self.error on failure
...
retryOrFinish(error: self.error)
```

So the order is: **task completes → validators run (may set `error`) → `retryOrFinish` → `finish` → response serializers run**. A validation failure becomes an `AFError.responseValidationFailed(reason:)` (`AFError.swift`, with reasons `unacceptableStatusCode`, `unacceptableContentType`, `customValidationFailed`, etc.). Because the error is set *before* serialization, every response handler (`responseDecodable`, `responseData`, …) surfaces it as `.failure(AFError)` in its `DataResponse.result`.

**Retry ties in here too**: `retryOrFinish` — if there's an error and a `RequestRetrier` approves via `retry(_:for:dueTo:completion:)` — calls `prepareForRetry()` (bumps `retryCount`, `reset()`s state) and re-enters `perform(_, forRetry: true)`, looping back through the whole build/send pipeline. Otherwise `finish()` runs the serializers and transitions the request to `.finished`.

---

### One-line summary

`Session.request` → lazy `URLRequestConvertible.asURLRequest()` (parameter encoding) → adapt via interceptor → `URLSessionTask` created and resumed per the `Request` state machine → `SessionDelegate` routes completion → validators run and may set an `AFError` → retry-or-finish → response serializers deliver the typed result. The `rootQueue` is the single serialization point holding it all together.
