## The key idea: everything is one flat slice of handlers

Gin doesn't have a separate "middleware stack" and "handler." Middleware and the final route handler are the **same type** — `HandlerFunc` — stored together in a single `HandlersChain` (`gin.go:57`, just `[]HandlerFunc`). The `Context` holds the slice plus a cursor:

```go
handlers HandlersChain
index    int8        // cursor into handlers
```

### 1. Building the chain (registration time)

When you register a route, gin merges the group's middleware with the route's handlers via `combineHandlers` (`routergroup.go:241`):

```go
mergedHandlers := make(HandlersChain, finalSize)
copy(mergedHandlers, group.Handlers)              // middleware first
copy(mergedHandlers[len(group.Handlers):], handlers) // route handler(s) last
```

So `Use(A, B)` + `GET("/x", H)` produces the flat slice `[A, B, H]`. This is precomputed and stored in the radix tree node — there's no per-request assembly of the chain. The assertion `finalSize < abortIndex` enforces a hard cap because `index` is an `int8`.

### 2. Dispatch (request time)

In `handleHTTPRequest` (`gin.go:690`), gin walks `engine.trees` to find the tree for the HTTP method, looks up the path in the radix tree (`root.getValue`), and on a match assigns the precomputed chain to the context and starts it:

```go
c.handlers = value.handlers
c.fullPath  = value.fullPath
c.Next()
```

If no route matches it instead sets `c.handlers` to `allNoRoute`/`allNoMethod` (global middleware + the 404/405 handler) and still calls `c.Next()` — so even error paths run through the same mechanism.

### 3. Running the chain — `Next()` (`context.go:188`)

```go
func (c *Context) Next() {
    c.index++
    for c.index < safeInt8(len(c.handlers)) {
        if c.handlers[c.index] != nil {
            c.handlers[c.index](c)
        }
        c.index++
    }
}
```

`index` starts at `-1`. The first `Next()` advances to `0` and loops forward, invoking each handler in order.

The elegant part is **re-entrancy**: when a middleware itself calls `c.Next()` *inside its body*, that nested call advances the cursor and runs all remaining handlers to completion before returning to the middleware. This is what gives you "around" behavior:

```go
func Logger(c *gin.Context) {
    start := time.Now()
    c.Next()                       // runs B, H... everything downstream
    log.Print(time.Since(start))   // resumes here after they finish
}
```

A middleware that *doesn't* call `Next()` still works — control returns to the outer loop, which advances the cursor and runs the next handler anyway. `Next()` is only needed when you want to run code *after* downstream handlers.

### 4. Aborting (`context.go:188–208`)

`abortIndex` is `math.MaxInt8 >> 1` (63). `Abort()` simply sets `index = abortIndex`:

```go
func (c *Context) Abort() { c.index = abortIndex }
```

Since the loop condition is `c.index < len(c.handlers)` and a chain can never reach 63 entries (the registration-time assertion guarantees it), jamming the cursor to 63 makes the loop terminate — no remaining handlers run. `IsAborted()` is just `index >= abortIndex`. Note Abort doesn't stop the *current* handler; you typically `return` after calling it.

## Summary of the flow

1. **Register:** group middleware + route handlers are flattened into one `HandlersChain` and stored in the radix tree.
2. **Match:** `handleHTTPRequest` finds the method tree, looks up the path, copies that chain onto the `Context`.
3. **Execute:** `c.Next()` walks the slice with an `int8` cursor; nested `Next()` calls produce wrap-around behavior, and `Abort()` short-circuits by jumping the cursor past the end.

The whole design is just *one slice + one integer cursor* — which is why it's allocation-free per middleware and very fast.
