# How Django's ORM builds and executes a query from a QuerySet

Django splits the work across three layers, each in its own object:

| Layer | Object | File | Job |
|---|---|---|---|
| Public API | `QuerySet` | `django/db/models/query.py` | Lazy, chainable wrapper; holds a `Query` and a result cache |
| Query AST | `Query` | `django/db/models/sql/query.py` | Backend-agnostic tree of WHERE clauses, joins, aliases, selected columns |
| SQL generation | `SQLCompiler` | `django/db/models/sql/compiler.py` | Turns the `Query` into a SQL string + params, executes it, yields rows |

The flow is **build → compile → execute → reconstruct**.

## 1. The QuerySet is lazy and immutable

A `QuerySet` wraps a `sql.Query` instance (`self.query`) and a `self._result_cache` that starts as `None`. Nothing touches the database until you iterate, slice with a step, call `len()`, `bool()`, `list()`, etc.

Every chaining method (`filter`, `exclude`, `annotate`, `order_by`…) calls **`_clone()`** (`query.py:2218`), which copies the QuerySet and its underlying `Query`, mutates the clone's `query`, and returns it. The original is untouched — this is why QuerySets are reusable and chainable.

```python
qs = Book.objects.filter(author="X")   # clone #1, no DB hit
qs2 = qs.exclude(year=2020)            # clone #2, no DB hit
```

Evaluation is triggered through **`_fetch_all()`** (`query.py:2238`):

```python
def _fetch_all(self):
    if self._result_cache is None:
        self._result_cache = list(self._iterable_class(self))
    if self._prefetch_related_lookups and not self._prefetch_done:
        self._prefetch_related_objects()
```

`__len__` and `__iter__` (`query.py:416`, `420`) both call `_fetch_all` first, so the query runs **once** and is cached in `_result_cache`. `self._iterable_class` is normally `ModelIterable` (or `ValuesIterable`, `FlatValuesListIterable`, etc., swapped in by `.values()` / `.values_list()`). `iterator()` (`query.py:561`) deliberately bypasses the cache for streaming.

## 2. Building the Query tree

When you call `.filter(author="X")`, the QuerySet calls into its `Query`:

- **`add_q(q_object)`** (`sql/query.py:1663`) — entry point for a `Q` object. It walks the boolean tree and produces a `WhereNode`.
- **`build_filter(...)`** (`sql/query.py:1488`) — handles a single `field__lookup=value` term: it splits the lookup path, resolves the field, and creates the join chain.
- **`setup_joins(...)`** (`sql/query.py:1891`) and **`get_initial_alias()`** (`sql/query.py:1113`) — the alias/join machinery. `setup_joins` follows relations across tables, registering table aliases (`T1`, `T2`, …) in the alias map and reference-counting them so unused joins can be trimmed.

So after all the chaining, `self.query` is an in-memory AST describing *what* to fetch, with no SQL text yet.

## 3. Compiling to SQL

When evaluation begins, `ModelIterable.__iter__` (`query.py:92`) asks the Query for a compiler:

```python

```

**`Query.get_compiler()`** (`sql/query.py:370`) asks the *backend* for the right compiler class — `connection.ops.compiler(self.compiler)` — which is how the same Query produces PostgreSQL vs. SQLite vs. MySQL dialects.

**`SQLCompiler.as_sql()`** (`compiler.py:757`) assembles the statement:
1. **`pre_sql_setup()`** (`compiler.py:83`) resolves the SELECT columns (`get_select`), ORDER BY, and GROUP BY, populating `self.select` and `self.klass_info`.
2. **`get_from_clause()`** (`compiler.py:1141`) emits the FROM and JOINs from the alias map (only aliases still reference-counted survive).
3. The `WhereNode` is compiled via `self.compile(self.where)`, yielding the WHERE fragment and its params.
4. These pieces are concatenated into `SELECT … FROM … WHERE … ORDER BY …` plus a flat list of parameters, with LIMIT/OFFSET added if the QuerySet was sliced.

Returns `(sql, params)` — a parameterized string (placeholders, never interpolated values, which is what makes it injection-safe).

## 4. Executing and fetching rows

**`SQLCompiler.execute_sql()`** (`compiler.py:1595`) runs it:

```python

...

```

The `result_type` controls fetching — `MULTI` wraps `cursor.fetchmany()` in `cursor_iter` for chunked reads (`compiler.py:1655`), `SINGLE` does `fetchone()`, `ROW_COUNT` returns `cursor.rowcount`. If the WHERE provably matches nothing, `as_sql` raises `EmptyResultSet` and Django returns an empty iterator **without hitting the DB** (`compiler.py:1615`).

**`results_iter()`** (`compiler.py:1565`) wraps the raw cursor rows, flattening chunks with `chain.from_iterable` and applying backend **converters** (`get_converters` / `apply_converters`) — e.g. turning DB strings into `datetime`/`Decimal`/`UUID`.

## 5. Reconstructing model instances

Back in `ModelIterable.__iter__` (`query.py:92`), the compiler exposes `select`, `klass_info`, and `annotation_col_map` (filled during `as_sql`). `klass_info` says which columns map to the model and which to `select_related` relations. For each row:

```python

```

**`Model.from_db()`** (`base.py:618`) builds the instance from the column slice, related populators attach `select_related` objects from the same row (no extra query), and annotation columns are set as attributes. The fully-formed object is `yield`ed, collected into `_result_cache`, and returned to your code.
