## The core: `RPCProtocol`

The heart is `RPCProtocol` in `src/vs/workbench/services/extensions/common/rpcProtocol.ts`. It wraps a generic `IMessagePassingProtocol` (just `send(VSBuffer)` + `onMessage`) and turns it into a bidirectional remote-procedure-call channel. Both sides instantiate one (see `extensionHostManager.ts`: `new RPCProtocol(protocol, logger)`).

## Proxies and the `$` convention

Services are identified by typed `ProxyIdentifier`s created in `extHost.protocol.ts`:

- **`MainContext`** — ~90 services living on the main side (e.g. `MainThreadCommands`, `MainThreadAuthentication`)
- **`ExtHostContext`** — ~60 services living in the extension host

Each `createProxyIdentifier('Name')` gets a unique numeric id. You obtain a remote object with `rpcProtocol.getProxy(MainContext.MainThreadCommands)`, which lazily builds a JS `Proxy`. Its `get` trap intercepts any property whose name **starts with `$`** and replaces it with a function that calls `this._remoteCall(rpcId, name, args)`. So the `$`-prefix is the marker for "this is a remote method" — `mainThreadCommands.$executeCommand(...)` becomes a wire message instead of a local call. Return values are always wrapped to `Promise<...>` since the call is now async.

## Serialization

Arguments are serialized by the `MessageIO` class. It distinguishes simple JSON args from "mixed" args that contain binary data: `VSBuffer` instances (VS Code's cross-platform `Uint8Array`/`Buffer` wrapper) are pulled out via `stringifyJsonWithBufferRefs` and replaced with `{ $$ref$$: index }` references, then restored on the other end with `parseJsonAndRestoreBufferRefs`. Each `ProtocolMessage` carries a 1-byte type, a 4-byte message id, a 4-byte ack id, then the payload. Message types include `RequestJSONArgs`, `RequestMixedArgs`, `Acknowledged`, `Cancel`, `ReplyOKJSON`, `ReplyErrError`, etc.

## Request/reply flow

For an extension calling a main-thread API:

1. Extension calls `mainProxy.$someMethod(args)` → proxy trap fires `_remoteCall`.
2. A request id is allocated (`++_lastMessageId`), a `LazyPromise` is registered in `_pendingRPCReplies`, args are serialized, and the message is sent over the protocol.
3. The main side receives it, **immediately sends an `Acknowledged`** message, then invokes the real handler.
4. When the handler resolves, it serializes and sends back `ReplyOKJSON`/`ReplyOKVSBuffer` (or `ReplyErrError`) tagged with the same request id.
5. The extension side looks up the pending reply by id and resolves the `LazyPromise`, so the extension's `await` resumes.

Cancellation tokens flow through as `Cancel` messages. The acks also feed **responsiveness tracking**: if requests go unacknowledged for ~3 seconds, `RPCProtocol` fires an "unresponsive" state (this is what surfaces the "Extension Host unresponsive" warning).

## The transport underneath

`RPCProtocol` doesn't care how bytes move — that's the `IMessagePassingProtocol`, which differs by host type:

- **Local desktop extension host** (`electron-browser/localProcessExtensionHost.ts`): the preferred transport is an Electron **`MessagePort`**. The renderer acquires the port via `acquirePort()` (`base/parts/ipc/electron-browser/ipc.mp.ts`), the connection details are passed to the spawned ext-host process through the environment (`MessagePortExtHostConnection`), and messages flow as `port.postMessage(buffer)` / `port.onmessage → VSBuffer.wrap(e.data)`. Alternative connection types exist for **named pipes / Unix domain sockets (`IPC`)** and raw **Node sockets (`Socket`)**.
- **Web worker extension host** (`browser/webWorkerExtensionHost.ts`): standard DOM `postMessage` between the worker and the main window.
- **Remote extension host** (SSH/containers/WSL): a Node socket carried by a `PersistentProtocol` that supports reconnection.
