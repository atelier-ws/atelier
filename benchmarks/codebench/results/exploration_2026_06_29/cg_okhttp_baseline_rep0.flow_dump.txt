## How OkHttp processes a request through its interceptor chain

OkHttp uses a **recursive, index-based chain of responsibility**. Each interceptor gets a `Chain` object, does work, calls `chain.proceed()` to hand off to the next interceptor, and gets a `Response` back to post-process. The request flows "down" the stack and the response bubbles back "up."

### 1. Assembling the chain

When you execute a call, `RealCall.getResponseWithInterceptorChain()` builds a flat, ordered list of interceptors (`RealCall.kt:208`):

```kotlin
val interceptors = mutableListOf<Interceptor>()
interceptors += client.interceptors          // your application interceptors
interceptors += RetryAndFollowUpInterceptor()
interceptors += BridgeInterceptor()
interceptors += CacheInterceptor()
interceptors += ConnectInterceptor
if (!forWebSocket) {
  interceptors += client.networkInterceptors  // your network interceptors
}
interceptors += CallServerInterceptor

val chain = RealInterceptorChain(
  call = this, interceptors = interceptors, index = 0,
  exchange = null, request = originalRequest,
)
val response = chain.proceed(originalRequest)
```

The **order matters** — this is the exact top-to-bottom path a request takes:

| Interceptor | Role |
|---|---|
| **Application interceptors** (`client.interceptors`) | Your code. Run once even if there are retries/redirects. Can short-circuit. |
| `RetryAndFollowUpInterceptor` | Recovers from failures, follows redirects and auth challenges (the retry loop). |
| `BridgeInterceptor` | App ↔ network translation: adds `Content-Length`, `Host`, `Accept-Encoding: gzip`, cookies; gunzips the response. |
| `CacheInterceptor` | Serves from / writes to the HTTP cache; may short-circuit without hitting the network. |
| `ConnectInterceptor` | Opens a connection / `Exchange` to the server. |
| **Network interceptors** (`client.networkInterceptors`) | Your code, with access to the live connection and the actual on-wire request/response. Skipped for WebSockets. |
| `CallServerInterceptor` | The terminal interceptor — writes the request and reads the response off the socket. |

### 2. The core mechanism: `RealInterceptorChain.proceed()`

The entire chain is driven by this method (`RealInterceptorChain.kt:312`):

```kotlin
override fun proceed(request: Request): Response {
  check(index < interceptors.size)
  calls++
  ...
  // Call the next interceptor in the chain.
  val next = copy(index = index + 1, request = request)
  val interceptor = interceptors[index]

  val response = interceptor.intercept(next)
    ?: throw NullPointerException("interceptor $interceptor returned null")
  ...
  return response
}
```

The key trick: the chain is **immutable** and progresses via an `index`. Each call to `proceed()`:
1. Makes a **copy** of itself with `index + 1` (`next`) — so the next interceptor can't reach back to earlier ones.
2. Grabs `interceptors[index]` and calls `interceptor.intercept(next)`.
3. That interceptor does its pre-work, calls `next.proceed(...)` (recursing one level deeper), gets the `Response`, does its post-work, and returns it.

So the call stack literally nests one frame per interceptor. `CallServerInterceptor` (the last one) doesn't call `proceed()` — it produces the real `Response`, which then unwinds back up through every interceptor in reverse.

### 3. The `Interceptor` contract

```kotlin
fun interface Interceptor {
  fun intercept(chain: Chain): Response
}
```

A typical interceptor is shaped like this — work *before* and *after* `proceed()`:

```kotlin
override fun intercept(chain: Interceptor.Chain): Response {
  val request = chain.request()
  // ... pre-processing (modify request) ...
  val response = chain.proceed(request)   // hand off down the chain
  // ... post-processing (inspect/modify response) ...
  return response
}
```

### Guardrails on network interceptors

When `exchange != null` (i.e. the chain is past `ConnectInterceptor`, in network-interceptor territory), `proceed()` enforces two invariants (`RealInterceptorChain.kt:317` and `:336`):

- A network interceptor **must retain the same host and port** (it can't redirect to a different server mid-connection).
- A network interceptor **must call `proceed()` exactly once** (it can't short-circuit or replay), tracked via the `calls` counter.

Application interceptors (where `exchange == null`) have neither restriction — they may skip `proceed()` entirely or call it multiple times.

### Summary of the flow

```
request →  [app interceptors]
        →  RetryAndFollowUp  →  Bridge  →  Cache  →  Connect
        →  [network interceptors]  →  CallServer  →  (socket)
                                                       ↓
response ←  ←  ←  ←  ←  ←  ←  ←  ←  ←  ←  ←  ←  ←  ←  ← ←
```

Each `proceed()` descends one index deeper; the terminal `CallServerInterceptor` produces the response, which unwinds back up through every interceptor's post-processing code. The immutable index-based design is what guarantees correct ordering and prevents an interceptor from re-entering an earlier stage.

@Throws(IOException::class)
override fun proceed(request: Request): Response {
  check(index < interceptors.size)

  calls++

  if (exchange != null) {
    check(exchange.finder.routePlanner.sameHostAndPort(request.url)) {
      "network interceptor ${interceptors[index - 1]} must retain the same host and port"
    }
    check(calls == 1) {
      "network interceptor ${interceptors[index - 1]} must call proceed() exactly once"
    }
  }

  // Call the next interceptor in the chain.
  val next = copy(index = index + 1, request = request)
  val interceptor = interceptors[index]

  @Suppress("USELESS_ELVIS")
  val response =
    interceptor.intercept(next) ?: throw NullPointerException(
      "interceptor $interceptor returned null",
    )

  if (exchange != null) {
    check(index + 1 >= interceptors.size || next.calls == 1) {
      "network interceptor $interceptor must call proceed() exactly once"
    }
  }

  return response
}
```

**How It Works:**
- The `proceed()` method gets the interceptor at the current `index`
- It creates a new chain with `index + 1` (via `copy()`) and passes it to the current interceptor
- The current interceptor then calls `intercept(next)`, which eventually triggers the next interceptor in the chain
- Each interceptor can modify the request before calling `chain.proceed()`, and can modify the response after it returns
- The chain terminates when the last interceptor (CallServerInterceptor) executes and returns a Response

@Throws(IOException::class)
internal fun getResponseWithInterceptorChain(): Response {
  // Build a full stack of interceptors.
  val interceptors = mutableListOf<Interceptor>()
  interceptors += client.interceptors                    // Index 0+: Client (Application) Interceptors
  interceptors += RetryAndFollowUpInterceptor()         // Index N: Retry and Follow-Up
  interceptors += BridgeInterceptor()                   // Index N+1: Bridge
  interceptors += CacheInterceptor()                    // Index N+2: Cache
  interceptors += ConnectInterceptor                    // Index N+3: Connect
  if (!forWebSocket) {
    interceptors += client.networkInterceptors          // Index N+4+: Network Interceptors (conditional)
  }
  interceptors += CallServerInterceptor                 // Index M: Call Server (final)

  val chain =
    RealInterceptorChain(
      call = this,
      interceptors = interceptors,
      index = 0,
      exchange = null,
      request = originalRequest,
    )

  var calledNoMoreExchanges = false
  try {
    val response = chain.proceed(originalRequest)
    if (isCanceled()) {
      response.closeQuietly()
      throw IOException("Canceled")
    }
    return response
  } catch (e: IOException) {
    calledNoMoreExchanges = true
    throw noMoreExchanges(e) as Throwable
  } finally {
    if (!calledNoMoreExchanges) {
      noMoreExchanges(null)
    }
  }
}
```

**Interceptor Order (in execution sequence):**

1. **Client/Application Interceptors** (0 to N-1): User-added interceptors via `client.interceptors`
2. **RetryAndFollowUpInterceptor** (N): Handles retries and HTTP redirects
3. **BridgeInterceptor** (N+1): Adds required headers (User-Agent, Content-Length, etc.)
4. **CacheInterceptor** (N+2): Implements HTTP caching logic
5. **ConnectInterceptor** (N+3): Establishes network connections
6. **Network Interceptors** (N+4 to M-1): User-added network interceptors via `client.networkInterceptors` (skipped for WebSockets)
7. **CallServerInterceptor** (M): Final interceptor that sends the request over the network

The chain starts with `index = 0` and `exchange = null`. It progresses through each interceptor sequentially, incrementing the index by 1 with each call to `proceed()`.
