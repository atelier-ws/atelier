## VS Code: extension host ↔ main process

**Process layout.** VS Code (Electron) has a *main process* (Node.js, owns windows, the menu, the lifecycle) and one *renderer process* per window (the UI/workbench, Chromium). Extensions don't run in either — they run in a separate **extension host** process, forked from the main process, so a slow or crashing extension can't freeze the UI.

**Transport.** Communication is **message-passing RPC over a duplex stream**, never shared memory:
- The channel is typically a **Node IPC pipe / socket** (`MessagePort` / named pipe / TCP for remote scenarios). For remote/WSL/SSH/web setups the extension host can live on another machine, so everything must be serializable.
- Messages are length-prefixed and serialized (JSON, with `VSBuffer` for binary). The framing layer is `IMessagePassingProtocol`.

**RPC layer (`RPCProtocol`).** On top of the raw stream sits a proxy system:
- Each side registers actors and gets typed **proxies** (`getProxy(identifier)`). Calling a proxy method serializes `{ requestId, actorId, method, args }`, sends it, and returns a `Promise`.
- The other side dispatches to the real implementation and sends back a reply (or error/cancel) keyed by `requestId`.
- Interfaces are split as `MainThreadXxx` (runs in renderer, the "main thread" side) and `ExtHostXxx` (runs in the extension host). The public `vscode` API objects in an extension are thin wrappers that call `ExtHost*`, which RPCs to `MainThread*` to actually touch the UI/editor.

**Consequences of the boundary:**
- Every API call crossing it is **async** — that's why so much of the `vscode` API returns `Thenable`/`Promise`.
- Arguments must be **serializable**; you can't pass live object references (hence handle/ID indirection for things like `TextDocument`).
- Cancellation, large payloads, and ordering are all handled by the protocol (cancellation tokens map to cancel messages, etc.).
