## The big picture

Everything funnels through a `Session` (the global `AF` is just a shared `Session`). A `Session` owns three serial queues that keep the whole thing thread-safe:

- **`rootQueue`** — coordinates all shared mutable state.
- **`requestQueue`** — does the async work of building `URLRequest`s.
- **`serializationQueue`** — runs response serializers.

The work happens in three phases: **build** (make a `URLRequest`), **send** (wrap it in a `URLSessionTask` and `resume()` it), and **validate/finish** (check the response, then serialize). Three abstractions carry it: `URLRequestConvertible` (something that can produce a `URLRequest`), the `Request` object (a state machine tracking the lifecycle), and the `RequestInterceptor` (which can `adapt` requests and `retry` them).

---

## 1. Building the request

When you call `AF.request(url, method:, parameters:, encoding:)` (`Session.swift:318`), Alamofire doesn't build a `URLRequest` immediately. Instead it captures your arguments in a lightweight `URLRequestConvertible` struct and hands that to the core `request(_:)` method:

- `RequestConvertible` (`Session.swift:286`) — for the legacy `Parameters` + `ParameterEncoding` API.
- `RequestEncodableConvertible` (`Session.swift:336`) — for the `Encodable` + `ParameterEncoder` API.

The core `request(_ URLRequestConvertible)` (`Session.swift:393`) creates the **`DataRequest`** object, storing the convertible inside it, and returns it right away. The actual `URLRequest` is built lazily/asynchronously later via `performEagerlyIfNecessary` → `perform` → `performDataRequest` → **`performSetupOperations`** (`Session.swift:1246`).

Inside `performSetupOperations`, on the `requestQueue`:

1. **`convertible.asURLRequest()`** is called (`Session.swift:1254`). This is where the `URLRequest` is actually assembled:
   - Base `URLRequest` from URL + `HTTPMethod` + `HTTPHeaders` (the `URLRequest(url:method:headers:)` initializer, `URLConvertible+URLRequestConvertible.swift:97`; the string/URL becomes a `URL` via `asURL()`).
   - Your `requestModifier` closure runs, if provided.
   - **Parameter encoding**: `encoding.encode(request, with: parameters)` or `encoder.encode(parameters, into: request)`. This is what writes the query string or `httpBody` + `Content-Type`. E.g. `URLEncoding` builds a query string for GET-style or sets a form body (`ParameterEncoding.swift:163`); `JSONEncoding`/`JSONParameterEncoder` serialize to JSON and set `application/json` (`ParameterEncoding.swift:275`, `ParameterEncoder.swift:75`).
2. The raw request is validated (`initialRequest.validate()`) and recorded via `request.didCreateInitialURLRequest` (appended to the `Request`'s `requests` array — `Request.swift:300`).
3. **Adaptation**: if an interceptor with an adapter is attached, `adapter.adapt(initialRequest, using: state, completion:)` runs asynchronously (`Session.swift:1273`). `Interceptor` chains multiple adapters recursively, short-circuiting on the first failure (`RequestInterceptor.swift:260`). This is where you'd inject auth headers, etc. The adapted request is recorded via `didAdaptInitialRequest`.
4. The final request flows to `didCreateURLRequest(_:for:)` (`Session.swift:1291`).

## 2. Sending the request

`didCreateURLRequest(_:for:)` (`Session.swift:1291`), back on the `rootQueue`:

1. Asks the request to build the task: **`request.task(for: urlRequest, using: session)`** — for a `DataRequest` this is `session.dataTask(with:)` (`DataRequest.swift:119`).
2. Records the pairing in the **`RequestTaskMap`** (`RequestTaskMap.swift`), a bidirectional `Request ↔ URLSessionTask` map so delegate callbacks can route back.
3. Calls **`request.didCreateTask(task)`** (`Request.swift:388`), which syncs the task to the request's current state: if the `Request` is already `.resumed`, it calls **`task.resume()`** — **this is the moment bytes go on the wire.** (`.suspended` → `task.suspend()`; `.cancelled` → resume-then-cancel to still gather metrics.)

The `Request.State` machine (`Request.swift:32`) governs all this: `.initialized → .resumed ↔ .suspended`, with `.cancelled` and `.finished` terminal. Calling `.resume()` before the task exists just flags `needsToPerform` and asks the session to build it (`Request.swift:768`).

**Completion** comes back through `SessionDelegate`. `URLSession` calls `urlSession(_:task:didCompleteWithError:)` (`SessionDelegate.swift:222`), which looks up the `Request` via the task map and ultimately calls **`request.didCompleteTask(task, with: error)`** (`Request.swift:515`).

## 3. Validating and finishing

You opt into validation by chaining `.validate()` on the request. Each `validate(...)` call (`DataRequest.swift:144`) appends a closure to the request's thread-safe **`validators`** array (`Request.swift:207`) — it does *not* run immediately.

- A bare `.validate()` (`Validation.swift:188`) chains two built-ins: `validate(statusCode: 200..<300)` and `validate(contentType:)` derived from the request's `Accept` header (defaulting to `*/*`).
- A validation closure has signature `(URLRequest?, HTTPURLResponse, Data?) -> ValidationResult` where `ValidationResult` is `Result<Void, Error>`.

The validators **run inside `didCompleteTask`** (`Request.swift:520`), synchronously on the `underlyingQueue`, right after the task completes and **before** `retryOrFinish`. If a validator returns `.failure`, it sets `self.error` to an `AFError.responseValidationFailed` — with a reason like `.unacceptableStatusCode(code:)`, `.unacceptableContentType(...)`, `.missingContentType(...)`, or `.customValidationFailed(error:)` (`AFError.swift:100`).

Then **`retryOrFinish(error:)`** (`Request.swift:543`) decides what's next:

- No error (or cancelled) → `finish()`.
- Error → asks the interceptor: `retrier.retry(request, dueTo: error, completion:)` (`Session.swift:1349`, chained in `RequestInterceptor.swift:315`). On `.retry`/`.retryWithDelay`, `retryRequest` calls `prepareForRetry()` (bumps `retryCount`) and re-enters `perform(...)` — **looping back to phase 1**. On `.doNotRetry`, it calls `finish()`.

Crucially, the validation error is stored in `self.error` *before* response serializers run on the `serializationQueue`. So when a serializer executes (`DataRequest.swift:243`), it receives that error and short-circuits — meaning **a failed validation prevents your `responseDecodable`/`responseJSON` handler from treating the body as success.**

## End-to-end, in one line each

```
AF.request(...)                → wrap args in URLRequestConvertible, make DataRequest
performSetupOperations         → convertible.asURLRequest()  [headers + param encoding]
  → adapter.adapt(...)         → interceptor injects/modifies the URLRequest
didCreateURLRequest            → session.dataTask(with:), map task↔request
didCreateTask                  → task.resume()              ← REQUEST GOES OUT
SessionDelegate didComplete    → request.didCompleteTask(...)
  → validators.forEach { }     ← VALIDATION runs, may set AFError
  → retryOrFinish              → retry (loop back) or finish
serializationQueue             → serializer sees self.error, produces your typed Result
```

The throughline: a `Request` is a state machine that accumulates `URLRequest`s, and the `Session` orchestrates queue-hopping between building, sending, validating, and (if needed) retrying — with `RequestInterceptor` as the single hook for both adapting outgoing requests and deciding retries.
