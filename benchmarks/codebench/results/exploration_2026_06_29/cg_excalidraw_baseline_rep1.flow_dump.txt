## The big picture: three layered canvases

Excalidraw doesn't render shapes with React/DOM — React only manages the UI chrome (toolbars, menus). Elements live on stacked HTML `<canvas>` layers, drawn imperatively:

- **Static canvas** (`renderer/staticScene.ts`) — all committed elements. Heavy, cached, throttled.
- **Interactive canvas** (`renderer/interactiveScene.ts`) — selection boxes, handles, cursors, snap lines. Cheap, redrawn often.
- **New-element canvas** (`renderer/renderNewElementScene.ts`) — the shape currently being drawn.

These are mounted via `<StaticCanvas>` / `<InteractiveCanvas>` in `components/App.tsx:2331/2381`.

## The data model: the `Scene`

`packages/element/src/Scene.ts` is the source of truth. It holds:
- `elements: readonly OrderedExcalidrawElement[]` — the array (each element is a plain object with `version`, `versionNonce`, `updated`).
- `elementsMap` / `nonDeletedElementsMap` — `Map`s for O(1) lookup, used by the renderers.
- `sceneNonce` — a random integer regenerated on every scene change; it's purely a **renderer cache-invalidation token** (not tied to element versions).

Anything that mutates the scene ends by calling `triggerUpdate()` (`Scene.ts:303`), which regenerates `sceneNonce` and fires registered callbacks.

## How an update propagates (the core loop)

1. **Mutation** — code calls `scene.mutateElement(el, updates)` → `mutateElement()` in `packages/element/src/mutateElement.ts`. It applies the changes and, if anything actually changed:
   - bumps `element.version = version + 1` and assigns a fresh `element.versionNonce = randomInteger()` (`mutateElement.ts:139-140`)
   - **invalidates the cached shape**: `ShapeCache.delete(element)` (`mutateElement.ts:136`), but only when geometry-affecting props (width/height/points/fileId) change.

2. **Notify** — `Scene.mutateElement` (`Scene.ts:415`) skips the notification if the version didn't change, otherwise calls `triggerUpdate()`.

3. **Re-render request** — App subscribes once in its constructor: `this.scene.onUpdate(this.triggerRender)` (`App.tsx:3147`). `triggerRender` (`App.tsx:4655`) usually just calls `this.setState({})` to schedule a React render; `force=true` instead calls `scene.triggerUpdate()` to force a redraw even with no change.

4. **Draw** — on React re-render, App passes the current elements + appState into `renderStaticScene(...)`. The static renderer is wrapped in `throttleRAF`:

   ```ts
   // renderer/staticScene.ts:482
   export const renderStaticSceneThrottled = throttleRAF((config) => _renderStaticScene(config));
   export const renderStaticScene = (renderConfig, throttle?) =>
     throttle ? renderStaticSceneThrottled(renderConfig) : _renderStaticScene(renderConfig);
   ```

   `throttleRAF` coalesces bursts of updates into one paint per animation frame. `sceneNonce` lets the renderer cheaply decide whether the static layer even needs repainting.

## How an individual element is drawn — the two-level cache

`packages/element/src/renderElement.ts` + `shape.ts` do the per-element work, with **two cache layers** keyed on object identity (`WeakMap`):

**Level 1 — `ShapeCache` (`shape.ts:81`)**: turns an element into a RoughJS `Drawable` ("shape"). `generateElementShape()` returns the cached shape unless missing/exporting/theme-changed; otherwise it calls `_generateElementShape()` (the big switch over element types → `getEllipseShape`, `getFreedrawShape`, etc.). The cache is a `WeakMap<element, {shape, theme}>`, so it's automatically tied to the element object's identity. `ShapeCache.delete` also clears the level-2 cache.

**Level 2 — `elementWithCanvasCache` (`renderElement.ts:603`)**: each element is rendered once to its own **off-screen canvas** (`generateElementWithCanvas`), then blitted onto the static canvas. Regeneration is triggered only when something visible changed (`renderElement.ts:631`):

## Why this design is fast

- **Identity-based `WeakMap` caches** mean cache invalidation is implicit: because `mutateElement` produces a new logically-versioned element and explicitly deletes its cache entries only when geometry changes, a moved/recolored element often reuses its cached bitmap.
- **`throttleRAF`** caps the static layer to one paint per frame regardless of how many mutations fire.
- **Layer separation** means dragging a selection only repaints the cheap interactive canvas; the expensive static canvas (with its per-element bitmap caches) is untouched until elements actually change.

So the chain is: **`mutateElement` → bump `version`/`versionNonce`, drop shape cache → `Scene.triggerUpdate` (new `sceneNonce`) → `App.triggerRender`/`setState` → `renderStaticScene` (throttled to rAF) → per-element `ShapeCache` + `elementWithCanvasCache` lookups → blit to canvas.**
