## The big picture: three stacked canvases

Excalidraw doesn't draw everything onto one `<canvas>`. It layers **three separate canvases** (composited in `App.tsx`), each with its own update cadence. This split is the central performance idea:

| Canvas | Component | Renders | Cadence |
|---|---|---|---|
| **Static** | `StaticCanvas.tsx` | Shapes, text, images, grid, frames | Rarely — only when elements/zoom/scroll change |
| **New element** | `NewElementCanvas.tsx` | The single shape currently being drawn | While dragging out a new element |
| **Interactive** | `InteractiveCanvas.tsx` | Selection boxes, handles, snap lines, binding highlights, remote cursors | Continuously, every animation frame |

The rationale: the expensive content (all your shapes) lives on the static canvas and stays still, while the cheap-but-frequently-changing UI overlay (selection handles, cursors, animations) lives on the interactive canvas and repaints every frame. Neither blocks the other.

## The rendering side

Each canvas component owns a render function in `packages/excalidraw/renderer/`:

- **`renderStaticScene()`** (`staticScene.ts`) — sets up the canvas, applies the zoom transform, draws the grid, then paints each visible element via `renderElement()`.
- **`renderInteractiveScene()`** (`interactiveScene.ts`) — draws selection borders, transform handles, linear-element point handles, binding highlights, and collaborator cursors. Returns animation state for the next frame.
- **`renderNewElementScene()`** (`renderNewElementScene.ts`) — draws just the in-progress element in isolation.

**How a shape actually gets drawn** (`packages/element/src/`):
- `renderElement()` (`renderElement.ts`) dispatches by element type (rectangle, ellipse, arrow, text, freedraw, image…).
- Shapes are generated with **RoughJS** (the hand-drawn look) and cached in **`ShapeCache`** (`shape.ts`), a `WeakMap` keyed by element + theme.
- For interactive rendering, elements are pre-rendered to off-screen canvases and then composited with `context.drawImage()` (`drawElementFromCanvas`) — far cheaper than re-running RoughJS every frame.

**Throttling:** the interactive canvas is driven by an `AnimationController` (`animation.ts`) running a `requestAnimationFrame` loop. Static/new-element renders are wrapped in `throttleRAF()` when `isRenderThrottlingEnabled()` is true (React ≥18), collapsing multiple updates into one paint per frame.

## The update side — how a change reaches the screen

State lives in the **`Scene`** class (`packages/element/src/Scene.ts`): an ordered `elements` array plus an `elementsMap` for O(1) lookup, and a `sceneNonce` (a random int).

The flow when you mutate an element (e.g. dragging it):

1. **`mutateElement()`** (`mutateElement.ts`) updates the element's properties in place. If anything actually changed, it bumps `element.version` (+1), regenerates `element.versionNonce` (random), and stamps `updated`. If geometry changed (width/height/points), it calls **`ShapeCache.delete(element)`** so the shape is regenerated.
2. **`Scene.mutateElement()`** sees the version changed and calls **`triggerUpdate()`**, which regenerates `sceneNonce` and fires all subscribers.
3. `App` subscribed via `scene.onUpdate(this.triggerRender)` in `componentDidMount`. **`triggerRender()`** calls `setState({})` — an empty state change that forces a React re-render.
4. `App.render()` calls `renderer.getRenderableElements()` (`scene/Renderer.ts`), which builds a **`canvasNonce`** from the `sceneNonce` and returns fresh element arrays.
5. The canvas components are `React.memo`'d with custom `areEqual()` checks. They re-render when `canvasNonce`, `scale`, `elementsMap`, or `visibleElements` change (interactive also watches `selectionNonce`). On re-render, their `useEffect` calls the render function, which pulls shapes from `ShapeCache` — regenerating any that were invalidated.

## Why versioning matters

`version` / `versionNonce` serve two purposes: they let the renderer cheaply detect "did this element actually change?", and they're the basis for **collaboration reconciliation** — `version` orders changes, and the random `versionNonce` is a deterministic tie-breaker when two clients edit concurrently.

**In one sentence:** a mutation bumps an element's version and invalidates its cached shape → the Scene regenerates its nonce and notifies React → `App` re-renders → memoized canvas components detect the nonce change and repaint, with the static canvas only redrawing changed shapes while the interactive canvas animates the UI overlay every frame.
