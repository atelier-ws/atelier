[assistant] OkHttp processes every request by passing it down an ordered stack of `Interceptor`s, each one wrapping the next. Here's how it works in this codebase.

## 1. The chain is assembled per-call

When you call `execute()` or `enqueue()`, it lands in `RealCall.getResponseWithInterceptorChain()` (`RealCall.kt:208`). It builds a single flat list of interceptors in a fixed order:

```kotlin
val interceptors = mutableListOf<Interceptor>()
interceptors += client.interceptors            // your application interceptors
interceptors += RetryAndFollowUpInterceptor()  // retries + redirects/auth follow-ups
interceptors += BridgeInterceptor()            // app request <-> network request
interceptors += CacheInterceptor()             // serve/store cached responses
interceptors += ConnectInterceptor            // open a connection + exchange
if (!forWebSocket) {
  interceptors += client.networkInterceptors   // your network interceptors
}
interceptors += CallServerInterceptor          // write request / read response on the wire
```

The order matters: **application interceptors** run first (outermost) and see the logical request/response, while **network interceptors** run later (inner, just before the wire) and see redirects, retries, and the actual bytes. Note network interceptors are skipped for WebSockets.

## 2. Execution: recursion via `proceed()`

It then creates a `RealInterceptorChain` at `index = 0` with `exchange = null` and calls `chain.proceed(originalRequest)` (`RealCall.kt:221-232`).

The whole mechanism lives in `RealInterceptorChain.proceed()` (`RealInterceptorChain.kt:312`):

```kotlin
override fun proceed(request: Request): Response {
  check(index < interceptors.size)
  calls++
  ...
  val next = copy(index = index + 1, request = request)  // chain pointing at the NEXT interceptor
  val interceptor = interceptors[index]                   // the current one
  val response = interceptor.intercept(next) ?: throw NullPointerException(...)
  ...
  return response
}
```

Each interceptor's `intercept(chain)` does pre-processing, calls `chain.proceed(request)` to hand off to the next link, then post-processes the `Response` on the way back. Because each `proceed()` builds a fresh copy advanced by one index, the chain unwinds like a recursive stack:

- **Going down** (request side): each interceptor transforms the request and delegates inward.
- **Going up** (response side): the response bubbles back out through each interceptor in reverse.

The terminal interceptor, `CallServerInterceptor`, does *not* call `proceed()` — it actually talks to the server and returns a `Response`, ending the descent.

## 3. Guarantees enforced along the way

`proceed()` enforces the contract:
- `check(index < interceptors.size)` — you can't proceed past the end.
- For **network interceptors** (where `exchange != null`), it verifies the request keeps the same host/port (`RealInterceptorChain.kt:318`) and that `proceed()` is called **exactly once** (`calls == 1`, and the `next.calls == 1` check at line 337). Application interceptors have no such restriction — they may call `proceed()` zero, one, or many times (which is how retry logic works).

The `exchange` is null for the application-interceptor portion and becomes non-null once `ConnectInterceptor` has acquired a connection — that's the boundary between the two interceptor types, as the class doc notes (`RealInterceptorChain.kt:46-51`).

## Summary

A request flows outermost → innermost: app interceptors → retry/follow-up → bridge → cache → connect → network interceptors → call-server, each calling `proceed()` to descend; the response then propagates back out in reverse. The recursion is implemented by `RealInterceptorChain` copying itself with an incremented `index` at each hop.
