[assistant] I now have a thorough, source-grounded picture. Here's how tokio actually schedules and runs async tasks.

## The big picture

Tokio has **two schedulers**: the multi-threaded one (`scheduler/multi_thread/`, the default for `#[tokio::main]`) and a single-threaded `current_thread` one. The interesting machinery is the multi-threaded, work-stealing scheduler, so that's what I'll focus on.

A scheduler starts a **fixed number of workers** (default = number of CPUs), each driven by one OS thread. Each worker owns a **`Core`** (`worker.rs:111`) holding its run queue and local state. The `Core` is stored in an `AtomicCell` so it can be *handed off* to another thread when the current one blocks (`block_in_place`).

## The queues

There are three places a runnable task can live:

1. **The LIFO slot** (`worker.rs:115`) — a single-task slot per core. When a worker schedules a task, it's stashed here *before* the run queue, so the most-recently-woken task runs next. This optimizes message-passing/ping-pong locality (e.g. task A wakes task B, B runs immediately on the same core with hot cache).
2. **The local run queue** (`queue.rs`) — a fixed-size ring buffer, capacity **256** (`queue.rs:63`). It's single-producer (the owning worker pushes) / multi-consumer (other workers can *steal* from it). The head is an atomic that also encodes an in-progress stealer to give ABA resilience (`queue.rs:36-50`).
3. **The global injection queue** (`inject/`) — a mutex-guarded queue used when a task is scheduled from *outside* a worker thread, or when a local queue overflows.

## How a task gets scheduled (the wake path)

When a future's waker is invoked, it ultimately calls `Handle::schedule_task` (`worker.rs:1327`). The routing depends on *where* the wake happens:

- **From within the current worker** (`ptr_eq` check passes and the thread still holds a core) → `schedule_local` (`worker.rs:1353`). A normal wake goes into the **LIFO slot**; the task previously in the slot (if any) is bumped to the back of the run queue. A *yield* (or when the LIFO optimization is disabled) always goes to the **back of the run queue** to be fair. If the local queue is full, `push_back_or_overflow` spills half the queue into the global inject queue.
- **From anywhere else** (another runtime, a non-runtime thread, or a resource driver) → `push_remote_task` puts it on the **global inject queue** and calls `notify_parked_remote` to wake a sleeping worker (`worker.rs:1340-1342`).

## The worker run loop

Each worker spins in `Context::run` (`worker.rs:561`). Per iteration (`worker.rs:570-607`):

1. **Tick + maintenance** — increment a tick counter; periodically check for shutdown and metrics (`worker.rs:1262`).
2. **`next_task`** (`worker.rs:1062`) picks the next task to run:
   - Every `global_queue_interval` ticks, it pulls from the **global queue first**, then local. This guarantees the inject queue can't be starved by a busy local queue.
   - Otherwise it pulls **local first** (`next_local_task` = LIFO slot, then run queue). If local is empty, it pulls a *batch* from the inject queue — roughly `inject.len() / num_workers + 1` tasks — taking a fair share and pushing the rest onto its own local queue.
3. **Run the task** via `run_task` (`worker.rs:630`): the core is parked in thread-local context, then `task.run()` polls the future under a **cooperative budget** (`coop::budget`, `worker.rs:675`). After the poll, it drains the **LIFO slot** in a loop — running freshly-woken tasks immediately — but only up to `MAX_LIFO_POLLS_PER_TICK` (`worker.rs:743`) to avoid two ping-ponging tasks starving everything else, and only while coop budget remains (`worker.rs:716`).
4. **Steal** (`worker.rs:1141`): if local and inject are empty, the worker transitions to *searching* and tries to steal. It starts at a **random** worker and walks the list, calling `steal_into`, which moves **half** the victim's queue (`queue.rs:265`) into its own. As a fallback it checks the global queue again.
5. **Park**: if there's nothing to steal, the worker parks (sleeps) on the I/O/timer driver (`worker.rs:810`, `park_internal` at `844`). One worker at a time owns the driver and blocks on epoll/etc.; when I/O events arrive it wakes the corresponding tasks onto its local queue and relies on work-stealing to spread them.

## Concurrency control that keeps it efficient

- **Searching state throttling**: a worker only attempts to steal if fewer than half the workers are already searching (`transition_to_searching`, `worker.rs:1171`), so you don't get a thundering herd of stealers.
- **Selective notification**: when a worker has surplus tasks it calls `should_notify_others` (`worker.rs:1192`) — only waking another worker if there's genuinely stealable work (`run_queue.len() + lifo > 1`) and this worker isn't itself searching.
- **Cooperative scheduling**: the coop budget (~128 polls) forces long-running tasks to yield back to the scheduler periodically, preventing a single hot task from monopolizing a worker.

## Shutdown

Cleanly torn down in stages (`worker.rs:7-32`): close the inject queue + `OwnedTasks`, each worker drains its tasks in parallel (`pre_shutdown`), then the last worker to hand back its core empties all the queues. The dual close-bits on both the inject queue and `OwnedTasks` prevent a task from leaking after shutdown.

---

**In one sentence:** tokio runs a pool of worker threads, each polling tasks from a fast per-core LIFO slot + local ring buffer, falling back to a shared global queue and randomized work-stealing from peers, with cooperative budgeting and careful "searching"/notification accounting to balance load while keeping woken tasks on cache-hot cores.
