[assistant] Here's how gin routes a request through its middleware chain, end to end.

## The core idea: one flat handler slice + an index cursor

Gin doesn't wrap middleware in nested closures (like the classic `next(w, r)` onion). Instead, every route stores a **single flat slice** of handlers — middleware and the final route handler all together — and the `Context` walks it with an integer cursor. This is what makes gin's chaining so cheap.

```go
type HandlerFunc func(*Context)          // gin.go:51 — middleware and handlers are the same type
type HandlersChain []HandlerFunc         // gin.go:57 — a flat slice, no nesting
```

## 1. Building the chain (registration time)

When you call `router.Use(...)` or register a route, handlers get concatenated into one slice.

- `RouterGroup.Use` (`routergroup.go:65`) appends middleware to the group's own `Handlers`.
- Registering a route (`RouterGroup.handle` → `combineHandlers`) merges the group's middleware with the route's handlers:

```go
func (group *RouterGroup) combineHandlers(handlers HandlersChain) HandlersChain {   // routergroup.go:241
	finalSize := len(group.Handlers) + len(handlers)
	assert1(finalSize < int(abortIndex), "too many handlers")   // hard cap (~63)
	mergedHandlers := make(HandlersChain, finalSize)
	copy(mergedHandlers, group.Handlers)                        // group middleware first
	copy(mergedHandlers[len(group.Handlers):], handlers)        // route handler(s) last
	return mergedHandlers
}
```

So order is: **group/global middleware first, the actual route handler last** (`HandlersChain.Last()` at gin.go:60 is literally "the main one"). The fully-combined slice is stored on the radix-tree node via `engine.addRoute` (`tree.go:135`, node field `handlers HandlersChain`).

## 2. Dispatch (request time)

`ServeHTTP` (`gin.go:662`) is the `http.Handler` entry point. It grabs a pooled `Context`, resets it, and calls `handleHTTPRequest`:

```go
c := engine.pool.Get().(*Context)   // sync.Pool — contexts are reused
c.writermem.reset(w)
c.Request = req
c.reset()
engine.handleHTTPRequest(c)
engine.pool.Put(c)                  // returned for reuse
```

`handleHTTPRequest` (`gin.go:690`) does the actual routing: it picks the tree for the HTTP method, walks the radix tree with `root.getValue(rPath, ...)` to match the path, and on a hit copies the matched node's handler slice onto the context and kicks off the chain:

```go
if value.handlers != nil {
	c.handlers = value.handlers   // the combined middleware+handler slice
	c.fullPath = value.fullPath
	c.Next()                      // start walking the chain
	c.writermem.WriteHeaderNow()
	return
}
```

If nothing matches, `c.handlers` is instead set to `engine.allNoRoute` (404) or `engine.allNoMethod` (405), which still run through the same `Next()` mechanism via `serveError`.

## 3. Walking the chain: `Context.Next`

This is the heart of it (`context.go:188`):

```go
func (c *Context) Next() {
	c.index++
	for c.index < safeInt8(len(c.handlers)) {
		if c.handlers[c.index] != nil {
			c.handlers[c.index](c)   // call handler at cursor, passing the context
		}
		c.index++
	}
}
```

`c.index` starts at `-1` (set by `reset()`), so the first `Next()` advances to `0` and loops through the slice.

The clever part is **re-entrancy**. When a middleware itself calls `c.Next()`, it advances the cursor and runs the *rest* of the chain to completion before returning — so any code the middleware writes *after* its `c.Next()` call runs on the way back out. That's how a logger or timer wraps the downstream handlers:

```go
func Logger(c *gin.Context) {
	start := time.Now()
	c.Next()                    // run everything downstream
	log(time.Since(start))      // runs after the handler returns
}
```

A middleware that does **not** call `Next()` still works — the `for` loop in the *outer* `Next()` frame just continues to the next handler. So `Next()` is only needed when you want to run code after downstream handlers.

## 4. Short-circuiting: `Abort`

To stop the chain (e.g. failed auth), middleware calls `c.Abort()` (`context.go:207`):

```go
func (c *Context) Abort() { c.index = abortIndex }   // abortIndex is a large sentinel
```

Setting `index` to `abortIndex` makes the `for c.index < len(c.handlers)` condition false, so no further handlers run. Note it does **not** stop the *current* handler — you typically `return` after aborting. Helpers like `AbortWithStatusJSON` combine `Abort()` with writing a response. `IsAborted()` just checks `index >= abortIndex`, and that same sentinel is why `combineHandlers` caps the chain length below `abortIndex`.

## The full path

```
http.Server
  └─ Engine.ServeHTTP                    (pool.Get Context, reset)
       └─ handleHTTPRequest              (radix-tree match → c.handlers = node.handlers)
            └─ c.Next()                  (index++, loop the flat slice)
                 ├─ handlers[0]  middleware  → may call c.Next() (recurse) or c.Abort()
                 ├─ handlers[1]  middleware
                 └─ handlers[n]  route handler (the "Last" one)
```

**In short:** middleware and route handlers are the same `func(*Context)` type, flattened into one slice at registration and stored per-route in the radix tree. At request time the matched slice is dropped onto a pooled `Context`, and `Context.Next()` drives it with an index cursor — recursion through `Next()` gives you before/after wrapping, and `Abort()` (jumping the index to a sentinel) gives you short-circuiting.
