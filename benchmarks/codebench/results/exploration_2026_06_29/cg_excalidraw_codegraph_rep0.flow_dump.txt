## codegraph
# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few
calls; a grep/read exploration is dozens.

## Overview: a two-layer canvas + memoized render pipeline

Excalidraw does **not** use React to draw shapes. Shapes are painted imperatively onto `<canvas>` elements. React only manages the DOM/UI chrome and decides *when* to hand fresh element data to the canvas painters. There are two stacked canvases:

- **Static canvas** — committed elements (`renderStaticScene`)
- **Interactive canvas** — selection boxes, handles, remote cursors (`renderInteractiveScene`)
- Plus a **NewElementCanvas** used as an optimization while you're actively drawing a shape.

## The data model: `Scene`

`Scene` (`packages/element/src/Scene.ts`) is the source of truth. It holds `elements`, an `elementsMap`, derived non-deleted collections, and a `sceneNonce`. The key mechanism is a callback/nonce pattern:

- `replaceAllElements()` rebuilds the maps and calls `triggerUpdate()` (Scene.ts:271).
- `triggerUpdate()` (Scene.ts:303) generates a fresh random `sceneNonce` and fires every registered callback. `App` registers a callback that calls `setState`, so a scene change → React re-render.

## Updating an element: `mutateElement`

There are three layers, and which one you call determines whether the UI repaints:

1. **`mutateElement(element, elementsMap, updates)`** (`mutateElement.ts:37`) — the low-level primitive. It mutates the element **in place**, diffs each key to skip no-op writes, invalidates the `ShapeCache` when geometry changes (width/height/points/fileId, line 136), and bumps `version`, `versionNonce`, and `updated`. Its docstring explicitly warns: *this won't trigger a re-render.*
2. **`Scene.mutateElement(...)`** (Scene.ts:411) — wraps the above, then, only if the element is actually in the scene, its version changed, and `informMutation` is set, calls `triggerUpdate()` to schedule a repaint.
3. **`App.mutateElement` / `ExcalidrawImperativeAPI.mutateElement`** (App.tsx:4644) — the public entry point, delegating to `scene.mutateElement`.

There's also `newElementWith` (immutable copy, used where new object identity is needed) and `bumpVersion` (version bump only).

`version`/`versionNonce` exist for collaboration conflict resolution; `sceneNonce` exists to cheaply detect "did anything change" for render memoization.

## From state change to pixels

1. A scene update fires the Scene callback → `App.setState` → React re-renders `App.render()` (App.tsx:2091).
2. `render()` calls `this.renderer.getRenderableElements({...})` (App.tsx:2104) passing zoom, scroll, viewport size, and the editing/new element.
3. **`Renderer.getRenderableElements`** (`Renderer.ts:208`) builds a `canvasNonce` from the scene nonce and calls the **memoized** `_getRenderableElements` (Renderer.ts:159). Because it's `memoize`d on `canvasNonce` + viewport params, an unrelated re-render returns the cached result and does zero recomputation. When it does run, it:
   - filters out the element being text-edited and the new-element-being-drawn (`getRenderableElementsMap`),
   - computes `visibleElements` via viewport culling — `getVisibleCanvasElements` keeps only elements passing `isElementInViewport` (Renderer.ts:47).
4. `render()` passes `elementsMap` / `visibleElements` / `canvasNonce` as **props** to the `<StaticCanvas>` and `<InteractiveCanvas>` React components.
5. `StaticCanvas` (`components/canvases/StaticCanvas.tsx`) is a memoized component; on relevant prop change it calls **`renderStaticScene(renderConfig, throttle)`**.

## The actual paint: `renderStaticScene`

`renderStaticScene` (`renderer/staticScene.ts:491`) either paints immediately or, when `throttle` is true, defers through `renderStaticSceneThrottled` — a `throttleRAF` wrapper that coalesces paints to the animation frame rate (staticScene.ts:482). `_renderStaticScene` then:

- normalizes canvas dimensions for DPR, bootstraps the context, applies `zoom.value` scale, draws the grid,
- iterates `visibleElements` and calls **`renderElement(...)`** for each (staticScene.ts:336), rendering bound text with its container, embeddables on top, link icons, and pending flowchart nodes.

## `ShapeCache` — the rendering cache

`renderElement` (`packages/element/src/renderElement.ts`) uses **`ShapeCache`** (`shape.ts:81`) to memoize the expensive Rough.js shape geometry per element. `mutateElement` deletes an element's `ShapeCache` entry when its geometry changes, so the next paint regenerates only that shape while everything else reuses cached geometry.

## Summary of the chain

```
mutateElement (in-place, bump version, invalidate ShapeCache)
  └─ Scene.triggerUpdate() → new sceneNonce + fire callbacks
       └─ App.setState → App.render()
            └─ Renderer.getRenderableElements() [memoized on nonce, viewport-culled]
                 └─ props → <StaticCanvas> / <InteractiveCanvas>
                      └─ renderStaticScene() [throttled to RAF]
                           └─ renderElement() per visible element [ShapeCache]
```

The recurring theme is **layered caching to avoid redundant work**: `sceneNonce`/`canvasNonce` short-circuit React-driven recomputation, viewport culling limits what's considered, `throttleRAF` coalesces paints, and `ShapeCache` avoids regenerating unchanged shape geometry.
