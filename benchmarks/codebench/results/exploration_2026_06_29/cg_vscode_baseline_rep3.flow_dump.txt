[assistant] The extension host runs in a **separate process** (or a Web Worker in the browser), so all communication happens over an asynchronous, binary **RPC protocol**. Here's how it works, from the high-level API down to the wire.

## The layered architecture

```
proxy.$methodName(args)          ← typed proxy call
   │
RPCProtocol                      ← request/reply correlation, serialization
   │
IMessagePassingProtocol          ← send(VSBuffer) / onMessage
   │
PersistentProtocol               ← framing, acks, reconnection
   │
Socket / NamedPipe / MessagePort ← actual bytes between processes
```

## 1. Proxies — the typed API surface
Code never sends raw messages. Each side calls methods on **lazy proxies** identified by a `ProxyIdentifier` (`src/vs/workbench/services/extensions/common/proxyIdentifier.ts`). The two registries of services are defined in `src/vs/workbench/api/common/extHost.protocol.ts`:
- **`MainContext`** — services living in the main/renderer side (e.g. `MainThreadCommands`, `MainThreadFileSystem`)
- **`ExtHostContext`** — services living in the extension host (e.g. `ExtHostDocuments`, `ExtHostCommands`)

By convention every remotable method is prefixed with **`$`** — the proxy's `Proxy` handler intercepts any `$method` access and turns it into an RPC call.

## 2. RPCProtocol — the engine
`RPCProtocol` (`src/vs/workbench/services/extensions/common/rpcProtocol.ts`) is the core. `getProxy()` hands out the lazy proxies; `set()` registers the local objects that handle incoming calls. On a call it:
1. Allocates an incrementing message id, stores a `LazyPromise` in `_pendingRPCReplies`.
2. Serializes the args and `send()`s them.
3. On the other side, deserializes, invokes the real `$method`, and serializes the reply.
4. The reply id resolves the pending promise.

It also handles cancellation tokens, `RequestInitiator` direction tracking, and **responsiveness monitoring** (detecting a hung/unresponsive extension host).

## 3. Serialization
Arguments are encoded as `VSBuffer`. JSON args that contain binary blobs use `stringifyJsonWithBufferRefs` / `parseJsonAndRestoreBufferRefs`, which lift `VSBuffer` instances out into a side array referenced by `$$ref$$` placeholders (`SerializableObjectWithBuffers`). URIs are rewritten through an `IURITransformer` so paths are valid on the remote side. Message types distinguish plain-JSON requests, mixed (JSON+buffer) requests, acks, cancels, and the various OK/Err replies.

## 4. Transport
The wire is abstracted by `IMessagePassingProtocol` (`src/vs/base/parts/ipc/common/ipc.ts`): just `send(buffer)` + `onMessage`. The robust implementation is **`PersistentProtocol`** (`src/vs/base/parts/ipc/common/ipc.net.ts`), which adds message framing (`ProtocolReader`/`ProtocolWriter`), acknowledgments, and an unacknowledged-message queue enabling **reconnection** (important for remote/SSH scenarios).

## 5. Bootstrapping the connection
- **Desktop (Node):** `src/vs/workbench/api/node/extensionHostProcess.ts` → `_createExtHostProtocol()` picks the transport (a **socket/named pipe** wrapped in `PersistentProtocol`, or a `MessagePort`), receives the init data from the main process, then constructs `ExtensionHostMain` (`extensionHostMain.ts`), which wraps the protocol in an `RPCProtocol`.
- **Web (Worker):** `src/vs/workbench/api/worker/extensionHostWorker.ts` uses a `MessageChannel`/`MessagePort` instead of a socket; everything above it is identical.
- **Main side:** `extensionHostManager.ts` (`_createExtensionHostCustomers()`) builds the matching `RPCProtocol` and registers the `MainThread*` services.

## Summary
The main process and extension host talk via an **asynchronous, promise-based RPC layer over a binary message channel**. You call `$`-prefixed methods on typed proxies; `RPCProtocol` serializes them (`VSBuffer`, with buffer-ref extraction for binary data) and correlates replies; `PersistentProtocol` frames and reliably delivers the bytes over a socket/named pipe (desktop) or `MessagePort` (web/worker). The `MainContext`/`ExtHostContext` identifiers in `extHost.protocol.ts` are the canonical catalog of what each side can call.
