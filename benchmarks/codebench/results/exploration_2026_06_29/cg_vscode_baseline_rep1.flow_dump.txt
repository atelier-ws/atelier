[assistant] The extension host runs in a **separate process** from the renderer/main process, and they communicate over a layered RPC stack. Here's how it works, from the wire up to the API:

## The layered architecture

```
Extension code  →  getProxy(MainContext.X).$method()
        │
   RPCProtocol            (bidirectional RPC, proxies + serialization)
        │
  IMessagePassingProtocol  (send(VSBuffer) / onMessage)
        │
  PersistentProtocol       (framing, ACKs, reconnection, replay)
        │
  Socket / MessagePort     (TCP socket, named pipe, or worker MessagePort)
```

## 1. The transport layer

The lowest level is `IMessagePassingProtocol` (`src/vs/base/parts/ipc/common/ipc.ts:99`) — just `send(buffer: VSBuffer)` and `onMessage: Event<VSBuffer>`.

The concrete implementation is `PersistentProtocol` (`src/vs/base/parts/ipc/common/ipc.net.ts:816`), which adds the resilient bits:
- **Message framing** — a 13-byte header (`ProtocolConstants.HeaderLength`) with message type, id, and ack id.
- **Acknowledgement & replay** — outgoing messages sit in `_outgoingUnackMsg` until ACKed, so they can be replayed after a dropped connection.
- **Reconnection** — `beginAcceptReconnection()` re-attaches a new socket; there's up to a 3-hour `ReconnectionGraceTime`, plus keep-alive pings and timeout detection.

The actual channel underneath is chosen at startup (`extensionHostProcess.ts:_createExtHostProtocol`): a **Node worker MessagePort**, a **TCP socket** (`NodeSocket`/`WebSocketNodeSocket`, e.g. for remote dev), or a **named pipe**.

## 2. The RPC layer

`RPCProtocol` (`src/vs/workbench/services/extensions/common/rpcProtocol.ts:117`) sits on top of the transport and provides actual method calls in both directions:

- `getProxy(identifier)` (line 243) returns a lazy JS `Proxy`. Calling `proxy.$someMethod(args)` triggers `_remoteCall` (line 461), which serializes the call and sends a request message.
- `set(identifier, impl)` (line 268) registers a local implementation; incoming requests hit `_receiveRequest` (line 360), which invokes the real method and sends back a reply via `_receiveReply` (line 403).

There's a fixed set of message types (request/reply/ack/cancel/error, lines 940-953), including cancellation-token-aware variants.

## 3. Proxy identifiers — the service catalog

Both sides agree on a registry of named services via `ProxyIdentifier` (`proxyIdentifier.ts:33`, created with `createProxyIdentifier`). Two big tables in `src/vs/workbench/api/common/extHost.protocol.ts`:
- **`MainContext`** (line ~3975) — ~80 services living in the main/renderer process: `MainThreadCommands`, `MainThreadDebugService`, `MainThreadEditors`, etc.
- **`ExtHostContext`** (line ~4063) — services living in the extension host: `ExtHostCommands`, `ExtHostConfiguration`, etc.

By convention, all RPC methods are prefixed with `$`.

## 4. Serialization

Arguments are marshalled by `stringifyJsonWithBufferRefs` (`rpcProtocol.ts:42`). It's JSON-based but with special handling for **binary buffers** — `VSBuffer` and `SerializableObjectWithBuffers<T>` are passed by reference alongside the JSON rather than base64'd into it, which keeps large binary payloads efficient. A `IURITransformer` also rewrites `URI` objects (important when the host is on a remote machine with a different filesystem).

## 5. Startup & wiring

- `ExtensionHostManager` (`extensionHostManager.ts:58`) calls `extensionHost.start()`, gets back an `IMessagePassingProtocol`, then in `_createExtensionHostCustomers` (line 249) constructs the `RPCProtocol` and registers every "main thread" service from the `ExtHostCustomersRegistry`.
- On the extension-host side, `extensionHostProcess.ts` does the handshake (`connectToRenderer`, line 332): it receives the `IExtensionHostInitData` payload (workspace, extensions, env), sends `Ready`, then `Initialized`.

## In short

When an extension calls something like `vscode.window.showInformationMessage(...)`, the API implementation in the extension host calls a `$`-prefixed method on a `MainContext` proxy. `RPCProtocol` serializes it (JSON + buffer refs) into a `VSBuffer`, `PersistentProtocol` frames and reliably delivers it over a socket/MessagePort to the renderer, where the corresponding `MainThread*` service executes it and sends the reply back along the same path. It's fully **bidirectional**, **asynchronous** (everything returns a `Promise`), **cancellable**, and **reconnection-resilient**.
