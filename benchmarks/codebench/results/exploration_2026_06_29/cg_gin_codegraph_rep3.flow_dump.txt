# Codegraph — code intelligence over an indexed knowledge graph

Codegraph is a SQLite knowledge graph of every symbol, edge, and file in
the workspace — pre-computed structure you would otherwise re-derive by
reading files (cached intelligence: thousands of parse/trace decisions you
don't pay to re-reason each run). Reads are sub-millisecond; the index lags
writes by ~1s through the file watcher. Reach for it BEFORE *and* while
writing or editing code — not just for questions: one call returns the
verbatim source PLUS who calls it and what it affects, so you edit with the
blast radius in view. More accurate context, in far fewer tokens and
round-trips than reading files yourself.

## One tool: codegraph_explore — use it instead of reading files

There is a single tool, `codegraph_explore`, and it is Read-equivalent. It
takes either a natural-language question or a bag of symbol/file names and
returns the **verbatim, line-numbered source** of the relevant symbols
grouped by file — the same `<n>\t<line>` shape `Read` gives you, safe to
`Edit` from — PLUS the call path among them (including dynamic-dispatch hops
like callbacks, React re-render, and JSX children that grep can't follow) and
a blast-radius summary of what depends on them.

Whether you're answering "how does X work" or implementing a change (fixing a
bug, adding a feature), call `codegraph_explore` before you Read. ONE call
usually answers the whole question. Codegraph IS the pre-built search index —
so running your own grep + read loop, or delegating the lookup to a separate
file-reading sub-task/agent, repeats work codegraph already did and costs more
for the same answer. A direct codegraph answer is typically one to a few

[assistant] Gin's middleware chain is a single flat slice of functions that gets built at route-registration time and walked with an index cursor at request time. Here's the full path.

## 1. Building the chain (registration time)

Middleware and handlers are all the same type — `HandlerFunc func(*Context)` — and a route's chain is just `HandlersChain []HandlerFunc` (`gin.go:51,57`). There's no separate "middleware" concept at runtime.

- **`Use()`** appends middleware to a router group's own `Handlers` slice (`routergroup.go:65`).
- When you register a route (`GET`, `POST`, …), it flows into `handle()` (`routergroup.go:86`), which calls **`combineHandlers()`** (`routergroup.go:241`). That allocates one merged slice = *group's middleware first, then the route's own handlers*:
  ```go
  mergedHandlers := make(HandlersChain, len(group.Handlers)+len(handlers))
  copy(mergedHandlers, group.Handlers)          // middleware
  copy(mergedHandlers[len(group.Handlers):], handlers)  // route handler(s)
  ```
- The combined chain is stored in the **radix tree** via `engine.addRoute` → `node.addRoute` (`tree.go:135`), one tree per HTTP method, keyed by path. The chain lives on `node.handlers` (`tree.go:106`).

So by the time a request arrives, each route already has a pre-computed, flat array like `[Logger, Recovery, AuthMiddleware, ..., YourHandler]`.

## 2. Dispatching a request (request time)

`Engine.ServeHTTP` (`gin.go:662`) is the `http.Handler` entry point:

1. Grabs a `Context` from a `sync.Pool`, resets it with the new request/writer.
2. Calls **`handleHTTPRequest(c)`** (`gin.go:690`).

`handleHTTPRequest` does the routing:
- Picks the tree matching `c.Request.Method` (`gin.go:708-713`).
- Looks up the path with `root.getValue(...)` (`gin.go:715`), which returns the matched node's `handlers` and any path params.
- On a hit, it copies the chain onto the context and starts walking it (`gin.go:719-724`):
  ```go
  c.handlers = value.handlers
  c.fullPath = value.fullPath
  c.Next()
  ```
- On a miss it assigns `engine.allNoRoute` / `allNoMethod` chains and calls `serveError` (which also calls `c.Next()`), so even 404/405 run through a handler chain.

## 3. Walking the chain — `Context.Next()`

The chain is driven by an index cursor on the context (`context.go:188`):

```go
func (c *Context) Next() {
    c.index++
    for c.index < safeInt8(len(c.handlers)) {
        if c.handlers[c.index] != nil {
            c.handlers[c.index](c)
        }
        c.index++
    }
}
```

Key mechanics:

- **`index` starts at -1** (set in `reset()`); the first `Next()` bumps it to 0 and runs handler 0.
- **Middleware controls the flow by calling `c.Next()` itself.** When a middleware calls `Next()`, it recurses *deeper* into the chain — everything after it runs, then control returns to the line after the middleware's `Next()` call. This is what enables "before/after" logic (e.g. Logger records start time, calls `Next()`, then logs latency after downstream handlers finish). A middleware that never calls `Next()` still works, because the `for` loop in the *outer* `Next()` frame keeps advancing the index — the chain runs linearly.
- **`Abort()`** (`context.go:207`) sets `index = abortIndex` (a large sentinel value, 63), so the `for c.index < len(c.handlers)` guard fails and no further handlers run. The *current* handler keeps executing. `combineHandlers` even asserts the chain size stays below `abortIndex` (`routergroup.go:243`) so this sentinel can never collide with a real index.

After `Next()` returns, `handleHTTPRequest` calls `c.writermem.WriteHeaderNow()` to flush and returns the context to the pool.

## Summary

| Phase | What happens | Where |
|---|---|---|
| Register | Group middleware + route handlers merged into one flat `HandlersChain`, stored in per-method radix tree | `combineHandlers` `routergroup.go:241`, `addRoute` `tree.go:135` |
| Match | Method tree lookup by path returns the node's chain + params | `handleHTTPRequest` `gin.go:690` |
| Execute | Chain copied to context, walked via an index cursor; middleware call `Next()` to nest, `Abort()` to stop | `Context.Next` `context.go:188` |

The elegant part: there's no separate middleware stack or wrapping — middleware and final handler are indistinguishable functions in one slice, and cooperative `Next()`/index bookkeeping is the entire execution engine.
