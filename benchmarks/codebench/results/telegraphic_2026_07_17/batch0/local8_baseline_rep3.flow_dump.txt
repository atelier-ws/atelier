## How it minimizes final image size

The final `runtime` stage ships **only** three things: production `node_modules`, compiled `dist/`, and `package.json`. Everything used to build — TypeScript, devDependencies, and source `.ts` files — stays behind in earlier stages and never lands in the shipped image.

Key decisions:

- **Separate `prod-deps` stage** — instead of copying the build's `node_modules` (which contains `typescript`, `@types/*`, etc.), a dedicated stage runs `npm ci --omit=dev` so the runtime gets a lean, production-only tree.
- **`node:22-alpine`** base — Alpine keeps the base ~50MB vs. several hundred for the default Debian image.
- **`npm ci`** — reproducible, lockfile-driven installs; the `--mount=type=cache` speeds rebuilds without bloating layers.
- **Manifests copied before source** — the dependency layers stay cached unless `package.json`/`package-lock.json` change.
- **`USER node` + `NODE_ENV=production`** — smaller attack surface and production runtime behavior.

## Two things to adjust for your app

1. **Build output path** — the runtime copies `/app/dist` and runs `dist/index.js`. If your `tsconfig.json` `outDir` or entrypoint differ, update the `COPY --from=build` and `CMD` lines.
2. **Build script** — assumes `npm run build` invokes `tsc`. Confirm that script exists in your `package.json`.

If you share your actual `package.json`/`tsconfig.json`, I'll tune the paths, port, and any runtime assets (e.g. `prisma`, static files) precisely.
