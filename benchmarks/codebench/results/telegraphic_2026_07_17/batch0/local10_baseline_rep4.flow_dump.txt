## What it does

- **Catches render errors** via `getDerivedStateFromError` — the required lifecycle for React error boundaries (must be a class component; there's no Hooks equivalent).
- **Shows a fallback UI with a retry button** — the built-in `DefaultFallback` renders an accessible `role="alert"` panel and a "Try again" button that clears the error state so the children re-render.
- **Logs error details** via `componentDidCatch`, which logs to the console and forwards `(error, errorInfo)` — including the component stack — to an optional `onError` handler for Sentry/Datadog/etc.

## Beyond the minimum

- **Custom fallback** — pass a node or a render function `({ error, retry }) => ...`.
- **`onError` hook** for external monitoring.
- **`resetKeys`** — the boundary auto-resets when a supplied key (e.g. route or user id) changes, so a stale error doesn't stick around after navigation.
- The dev-only error message uses `import.meta.env?.DEV` (Vite). If you're on CRA/webpack, swap that for `process.env.NODE_ENV !== "production"`.

## Usage

```tsx
import { ErrorBoundary } from "./ErrorBoundary";

<ErrorBoundary
  onError={(error, info) => reportToSentry(error, info)}
  resetKeys={[location.pathname]}
>
  <Dashboard />
</ErrorBoundary>
```

## Two caveats worth knowing

1. Error boundaries **only catch errors during render/lifecycle/constructors** of descendants — not in event handlers, async callbacks, `setTimeout`, or SSR. Handle those with regular try/catch.
2. It won't catch errors thrown by itself, only by its children.
