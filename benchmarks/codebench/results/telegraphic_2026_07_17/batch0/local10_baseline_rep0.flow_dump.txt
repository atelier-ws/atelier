It covers all three requirements and a couple of practical extras:

- **Catches render errors** — implements `getDerivedStateFromError` (switches to fallback) and `componentDidCatch` (side effects). Class component, since React error boundaries can only be classes.
- **Fallback UI with retry** — a built-in `role="alert"` card showing the error message plus a **Try again** button that clears the error state and re-renders `children`.
- **Logs error details** — `componentDidCatch` logs the error and `componentStack` to `console.error`, with a note on where to swap in Sentry/telemetry.

Extras worth knowing:
- `fallback` prop accepts either a static node or a render function `({ error, reset }) => ...` for a custom UI.
- `onError` prop forwards the error to your own logger.
- `resetKeys` auto-resets the boundary when a listed value changes (e.g. the current route), so a fixed input recovers without a manual click.

Usage:

```tsx
<ErrorBoundary onError={reportToSentry} resetKeys={[location.pathname]}>
  <App />
</ErrorBoundary>
```

Two notes on scope: this is an empty workspace with no `package.json`, so nothing is wired up — you'll need React installed and a bundler/TS config to use it. And by design this catches *render*-phase errors only; errors thrown in event handlers, async callbacks, or SSR aren't caught by any React error boundary — wrap those in try/catch.
