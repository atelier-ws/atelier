[assistant] Database connection pooling is a technique for reusing a set of open database connections rather than opening and closing one for every request.

## The problem it solves

Opening a database connection is expensive. Each new connection typically requires:

- A TCP handshake (and often a TLS handshake)
- Authentication and authorization
- Session/memory allocation on the database server

That can take tens to hundreds of milliseconds. If your app opens a fresh connection for every query, you pay that cost constantly, and under load you can exhaust the database's connection limit and its memory.

## How a pool works

A connection pool is a cache of pre-established, ready-to-use connections managed by your app (or a proxy):

1. **On startup**, the pool opens a number of connections (`min` size).
2. **When code needs a connection**, it *borrows* one from the pool instead of opening a new one.
3. **When done**, the code *returns* the connection to the pool — it's reset, not closed.
4. **If all connections are busy**, the pool either opens more (up to a `max` limit) or makes the caller wait.
5. **Idle connections** beyond the minimum are eventually closed to free resources.

```
App threads ──▶ [ borrow ]
                 ┌──────────────────────┐
                 │  Connection Pool     │
                 │  [conn1][conn2][conn3]│──▶ Database
                 │  [conn4]  ...        │
                 └──────────────────────┘
App threads ◀── [ return ]
```

## Key configuration knobs

| Setting | What it controls |
|---|---|
| **Min pool size** | Connections kept open even when idle (warm capacity) |
| **Max pool size** | Ceiling on total connections (protects the DB) |
| **Connection timeout** | How long a caller waits for a free connection before erroring |
| **Idle timeout** | When to close surplus idle connections |
| **Max lifetime** | Retire connections after a set age to avoid stale/leaked state |
| **Validation/test query** | Check a connection is still alive before handing it out |

## Benefits

- **Lower latency** — no per-request connection setup.
- **Higher throughput** — connections are reused across many requests.
- **Resource protection** — the `max` size bounds load on the database.
- **Stability** — prevents runaway connection counts under traffic spikes.

## Trade-offs and pitfalls

- **Sizing is critical.** Too small → requests queue and time out. Too large → you overwhelm the DB (each connection consumes server memory/CPU). A common formula: pool size ≈ `((core_count × 2) + effective_spindle_count)` — often smaller than people expect.
- **Connection leaks.** If code borrows a connection and never returns it (missing `close()`/`finally`), the pool drains and everything stalls. Use try-with-resources / context managers.
- **Stale connections.** A connection idle for hours may have been dropped by the DB or a firewall; validation and max-lifetime settings handle this.
- **Serverless/short-lived environments** (e.g. Lambda) don't pool well per-instance — that's why external poolers exist.

## Common implementations

- **In-app libraries:** HikariCP (Java), `pgbouncer`-style logic in ORMs, `database/sql`'s built-in pool (Go), SQLAlchemy's `QueuePool` (Python), `node-postgres` Pool (Node).
- **External proxies:** PgBouncer / Pgpool-II (PostgreSQL), ProxySQL (MySQL) — a middleware layer that pools on behalf of many app instances, useful when you have many clients each with their own small pool.

**In short:** connection pooling trades a bit of memory (keeping connections warm) for large gains in latency, throughput, and stability — as long as you size it correctly and always return connections to the pool.
