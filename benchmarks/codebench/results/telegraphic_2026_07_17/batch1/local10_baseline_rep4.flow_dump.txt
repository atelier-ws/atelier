[assistant] Queues and topics are both messaging abstractions, but they model fundamentally different delivery semantics: **one consumer per message** vs. **many consumers per message**.

## Queue — point-to-point (competing consumers)

A message goes to **exactly one** consumer. Multiple consumers can read from the same queue, but each message is delivered to only one of them.

**Use a queue when:**
- **Work distribution / load balancing** — you have tasks that need to be done once, and you want to spread them across a pool of workers (e.g. image resizing, sending emails, processing orders).
- **You care that each unit of work happens exactly once** (or at-least-once with idempotency), not that everyone hears about it.
- **You want to scale throughput** by adding more consumers — they naturally share the load.
- **Buffering / smoothing load** — decouple a fast producer from slower workers so spikes don't overwhelm downstream.

Mental model: *"Someone, do this job."*

## Topic — publish/subscribe (fan-out)

A message is delivered to **every** subscriber. Each subscriber gets its own copy.

**Use a topic when:**
- **Broadcasting events** — one thing happened, and multiple independent systems need to react (e.g. `OrderPlaced` → billing, inventory, email, analytics all need to know).
- **Decoupling producers from an unknown/growing set of consumers** — the publisher doesn't know or care who's listening; you can add new subscribers without touching the producer.
- **Event-driven architectures** — different services maintain their own view or trigger their own workflows from the same event stream.

Mental model: *"This happened — whoever cares, react."*

## Quick decision guide

| Question | Answer → use |
|---|---|
| Should each message be processed by exactly one worker? | **Queue** |
| Should multiple independent systems each react to the message? | **Topic** |
| Is this a *command/task* ("do X")? | **Queue** |
| Is this an *event/notification* ("X happened")? | **Topic** |
| Do you scale by adding competing workers? | **Queue** |
| Do you scale by adding new independent subscribers? | **Topic** |

## Combining them

The two aren't mutually exclusive — a very common pattern is **topic + a queue per subscriber**. The topic fans out the event, and each subscriber has its own durable queue where a pool of that subscriber's workers competes for the copies. This gives you both broadcast (across services) and load-balancing (within a service).

Some systems blur the line:
- **Kafka** — a partitioned log where *consumer groups* give you queue-like competing-consumer behavior within a group, and topic-like fan-out across groups. One primitive, both semantics.
- **AWS SNS (topic) + SQS (queue)** — the canonical fan-out-then-load-balance combo.
- **RabbitMQ** — exchanges (routing) fan out to queues; the "topic" is a routing concept, delivery still lands in queues.

**Rule of thumb:** commands/jobs → queue; events/notifications → topic. If you're unsure whether more consumers might need the data later, lean toward a topic (or log) — it's easier to add subscribers than to retrofit fan-out onto a queue.
