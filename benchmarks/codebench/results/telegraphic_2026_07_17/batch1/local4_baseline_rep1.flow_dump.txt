[assistant] Here's a practical playbook for tracking down and fixing a memory leak in a long-running Node process.

## 1. Confirm it's actually a leak

Watch RSS/heap over time under steady load. A leak grows without bound and never recovers after GC; normal churn plateaus.

```bash
node --expose-gc app.js
```

```js
setInterval(() => {
  global.gc(); // force GC so you measure retained, not garbage
  const { heapUsed, rss } = process.memoryUsage();
  console.log(`heap=${(heapUsed/1e6).toFixed(1)}MB rss=${(rss/1e6).toFixed(1)}MB`);
}, 10_000);
```

If `heapUsed` climbs monotonically across GCs, it's a heap leak. If `heapUsed` is flat but `rss` keeps growing, suspect native/Buffer/C++ addon memory or fragmentation instead.

## 2. Capture and compare heap snapshots

This is the highest-signal step. Take a snapshot, run load, take another, and diff.

```js
const v8 = require('v8');
const fs = require('fs');
// call this on a signal or timer
function snapshot(tag) {
  const file = `/tmp/heap-${tag}-${Date.now()}.heapsnapshot`;
  fs.writeFileSync(file, ''); // v8.writeHeapSnapshot returns the path
  v8.writeHeapSnapshot(file);
}
```

Load both `.heapsnapshot` files into Chrome DevTools (`chrome://inspect` → Memory), use the **Comparison** view between two snapshots, and sort by "# Delta". The constructors that keep growing are your culprits. The **retainers** pane shows *what* is holding them alive — follow it to the root.

For live inspection: `node --inspect app.js`, open `chrome://inspect`, use Allocation instrumentation on timeline.

## 3. The usual suspects

Most Node leaks are one of these:

- **Unbounded caches / Maps / arrays** — a module-level `Map` or array you `push` to but never evict. Use an LRU (e.g. `lru-cache`) with a max size, or a `WeakMap`/`WeakRef` when keys are objects whose lifetime you don't control.
- **Event listeners not removed** — `emitter.on(...)` in a per-request or per-connection path without a matching `removeListener`/`off`. The `MaxListenersExceededWarning` is the tell. Use `once` where appropriate.
- **Timers / intervals** — `setInterval` that's never `clearInterval`'d, or timers holding closures over big objects.
- **Closures capturing large scope** — a long-lived callback that closes over a big buffer/request object keeps it all alive.
- **Global/module state** — accumulating in singletons, module-scoped variables, or `process`-level registries.
- **Promises that never settle** — pending promises and their `.then` chains retain everything captured.
- **Streams not consumed/destroyed** — backpressure or undrained streams buffering data.

## 4. Fix patterns

- Bound every cache with a size/TTL limit and eviction.
- Pair every `on`/`addListener`/`setInterval`/subscribe with its teardown — often in a `finally`, `close`, or cleanup hook.
- Prefer `WeakMap`/`WeakRef`/`FinalizationRegistry` for associating data with objects you don't own.
- Scope large data narrowly so it goes out of scope and becomes collectable.
- For request-scoped work, make sure nothing request-scoped gets stored in module/global state.

## 5. Verify the fix

Re-run the same load test and confirm heap plateaus across GCs. Keep the two-snapshot diff as your regression check.
