[assistant] Use a **queue** for work distribution, a **topic** for event broadcast. The deciding question is: does each message need to be handled *once*, or observed by *many*?

## Queue — point-to-point, competing consumers

One message is delivered to exactly one consumer. Multiple workers pull from the same queue and *compete* for messages, so adding workers scales throughput.

Reach for a queue when:
- A message represents **work that must happen once** — process a payment, resize an image, send an email.
- You want to **load-balance** across a pool of interchangeable workers.
- You need **backpressure** — the queue absorbs bursts and workers drain at their own pace.
- Order and idempotency matter within a single logical stream of work.

## Topic — publish/subscribe, fan-out

One message is delivered to *every* subscriber. Publishers don't know who's listening; each subscriber gets its own copy.

Reach for a topic when:
- A message represents a **fact/event that already happened** — `OrderPlaced`, `UserSignedUp`.
- **Multiple independent consumers** each need to react (billing, analytics, notifications, search-indexing all care about the same order).
- You want **loose coupling** — add a new subscriber later without touching the publisher.

## The rule of thumb

| | Queue | Topic |
|---|---|---|
| Delivery | 1 message → 1 consumer | 1 message → N subscribers |
| Intent | "do this task" (command) | "this happened" (event) |
| Adding consumers | more throughput | more independent reactions |
| Coupling | producer knows there's a job | producer doesn't know who cares |

## Practical notes

- **They compose.** A very common pattern is **topic → queues** (fan-out then work-distribution): publish an event to a topic, each interested service has its own queue subscription, and *within* that service a pool of workers competes on its queue. This is exactly what AWS SNS→SQS, or Kafka consumer groups, give you.
- **Broker terminology varies.** Kafka has only "topics," but a *consumer group* makes a topic behave like a queue (each partition goes to one consumer in the group), while multiple groups give pub/sub fan-out. RabbitMQ has queues plus exchanges that route to them. So the queue-vs-topic distinction is often about *subscription semantics*, not two separate products.
- **Rough default:** if you're issuing a **command** to be executed, think queue. If you're **announcing an event** others may care about, think topic.
