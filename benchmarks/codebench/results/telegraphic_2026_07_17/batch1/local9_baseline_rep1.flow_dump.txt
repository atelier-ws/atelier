[assistant] Git rebase and git merge both integrate changes from one branch into another, but they do it differently and produce different history.

## git merge

Takes the changes from one branch and combines them into another via a **merge commit** that ties the two histories together.

```
      A---B---C  feature
     /         \
D---E---F---G---M  main
```

- **Non-destructive** — existing commits are never changed.
- History is preserved exactly as it happened, including the branch topology.
- Creates an extra merge commit each time (can get noisy with frequent merges).

## git rebase

Moves your branch's commits so they replay **on top of** another branch, rewriting them as new commits with new hashes.

```
Before:                 After rebase onto main:
      A---B---C feature        A'--B'--C' feature
     /                        /
D---E---F---G main    D---E---F---G main
```

- **Rewrites history** — A, B, C become brand-new commits A', B', C'.
- Produces a **linear, clean** history with no merge commits.
- You resolve conflicts commit-by-commit as they replay.

## Key differences

| | Merge | Rebase |
|---|---|---|
| History | Preserved, non-linear | Rewritten, linear |
| Commit hashes | Unchanged | New hashes |
| Merge commit | Yes | No |
| Traceability | Shows when/how branches joined | Looks like sequential work |

## The golden rule

**Never rebase commits that have been pushed to a shared branch that others are working on.** Rebasing rewrites history, so it changes commit hashes — collaborators who based work on the old commits will get diverged histories. Rebase private/local branches; merge (or only rebase before pushing) for shared ones.

## Practical guidance

- **Rebase to update a feature branch** with the latest `main` before opening a PR — keeps a tidy, linear history: `git rebase main`
- **Merge to integrate a finished feature** into `main`, preserving the context that it was a distinct unit of work.
- **`git pull --rebase`** avoids the little "Merge branch 'main'..." commits that pollute history when syncing.
- **Interactive rebase** (`git rebase -i`) additionally lets you squash, reorder, edit, or drop commits before sharing.

A common workflow combines both: rebase your feature branch onto `main` to clean it up, then merge it in.
